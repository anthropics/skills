# 職業分析フレームワーク

任意の職業・ペインポイントからAgent Skillsを動的に提案するための抽象的分析フレームワーク。

---

## 分析原則

**決め打ちのカテゴリリストは使用しない。** 以下のフレームワークに従って、任意の職業から動的にスキルを推論する。

---

## Step 1: 職業特性の抽出

職業名から以下の4軸で特性を推論する：

### 1.1 業務の性質軸

| 軸 | 質問 | 例 |
|---|------|-----|
| **創造 ↔ 処理** | 新しいものを作るか、既存のものを処理するか | デザイナー(創造) / 経理(処理) |
| **対人 ↔ 対物** | 人と関わるか、モノ・データと関わるか | 営業(対人) / エンジニア(対物) |
| **定型 ↔ 非定型** | ルーティンワークか、都度判断が必要か | 事務(定型) / コンサル(非定型) |
| **社内 ↔ 社外** | 主に社内向けか、社外向けか | 人事(社内) / PR(社外) |

### 1.2 特性からの推論ルール

```
創造寄り → generator, creator, designer 系スキル
処理寄り → processor, formatter, validator 系スキル
対人寄り → writer, communicator, summarizer 系スキル
対物寄り → analyzer, extractor, converter 系スキル
定型寄り → template, automation, batch 系スキル
非定型寄り → assistant, advisor, helper 系スキル
```

---

## Step 2: ペインポイントのパターン認識

ペインポイントの記述から、以下のキーワードパターンを検出し、スキルタイプを推論する。

### 2.1 時間・効率系

**検出キーワード**: 時間がかかる、毎週、毎月、繰り返し、手作業、手動、コピペ、転記

```
パターン: 「{作業}に時間がかかる」「毎{週/月}{作業}している」
  ↓
推論: 自動化・生成系スキル
命名: {作業対象}-generator, {作業対象}-automator
```

### 2.2 情報処理系

**検出キーワード**: 集計、比較、分析、まとめ、抽出、整理、探す、調べる

```
パターン: 「{対象}を集計/分析している」「{複数}を比較する」
  ↓
推論: データ処理・分析系スキル
命名: {対象}-analyzer, {対象}-aggregator, {対象}-comparer
```

### 2.3 ドキュメント系

**検出キーワード**: 作成、更新、ドキュメント、資料、書類、フォーマット、テンプレート

```
パターン: 「{文書}の作成/更新が追いつかない」「{資料}を作る」
  ↓
推論: 文書生成・フォーマット系スキル
命名: {文書}-creator, {文書}-formatter, {文書}-updater
```

### 2.4 コミュニケーション系

**検出キーワード**: メール、連絡、報告、依頼、返信、文面、案内、通知

```
パターン: 「{種類}メールの作成」「{相手}への報告」
  ↓
推論: 文章作成・テンプレート系スキル
命名: {目的}-email-writer, {目的}-report-generator
```

### 2.5 検証・チェック系

**検出キーワード**: チェック、確認、レビュー、漏れ、ミス、検証、照合、監査

```
パターン: 「{対象}のチェックに時間がかかる」「{確認作業}が煩雑」
  ↓
推論: 検証・バリデーション系スキル
命名: {対象}-checker, {対象}-validator, {対象}-reviewer
```

### 2.6 変換・統合系

**検出キーワード**: 変換、統合、マージ、移行、連携、インポート、エクスポート

```
パターン: 「{形式A}から{形式B}に変換」「複数の{対象}を統合」
  ↓
推論: 変換・統合系スキル
命名: {対象}-converter, {対象}-merger, {形式A}-to-{形式B}
```

---

## Step 3: スキル設計の推論

### 3.1 入力の推論

ペインポイントと職業から入力データを推論：

| 手がかり | 推論される入力 |
|---------|--------------|
| 「〜のデータから」「〜を元に」 | 明示されたデータ形式 |
| 「レポート作成」「資料作成」 | 元データ（CSV, Excel, DB等） |
| 「メール作成」 | 宛先情報、要件、コンテキスト |
| 「チェック」「レビュー」 | チェック対象の文書・データ |
| 職業が技術系 | コード、ログ、設計書等 |
| 職業が事務系 | Excel、PDF、フォーム等 |

### 3.2 出力の推論

「出力形式の希望」が空欄の場合、以下で推論：

| スキルタイプ | デフォルト出力 |
|------------|--------------|
| レポート・報告系 | Excel or PDF |
| 技術ドキュメント系 | Markdown |
| メール・コミュニケーション系 | テキスト |
| データ処理系 | CSV or JSON |
| プレゼン系 | PowerPoint |

### 3.3 命名の推論アルゴリズム

```
1. ペインポイントから主要な「対象」を抽出
   例: 「週次売上レポートの作成に時間がかかる」→ 対象 = weekly-sales-report

2. パターン認識でスキルタイプを決定
   例: 「作成に時間がかかる」→ タイプ = generator

3. 組み合わせて命名
   例: weekly-sales-report-generator

4. 希望するスキル名があればそれを優先
```

---

## Step 4: 提案生成テンプレート

### 4.1 分析結果の構造

```yaml
職業分析:
  職業名: "{入力された職業}"
  推論された特性:
    業務性質: "{創造/処理} 寄り"
    関係対象: "{対人/対物} 寄り"
    作業パターン: "{定型/非定型} 寄り"

ペインポイント分析:
  原文: "{入力されたペインポイント}"
  検出パターン: "{時間系/情報処理系/etc.}"
  キーワード: ["{検出されたキーワード}"]

スキル設計:
  名前: "{推論されたスキル名}"
  タイプ: "{generator/analyzer/etc.}"
  入力: "{推論された入力}"
  出力: "{推論された出力}"
```

### 4.2 提案出力フォーマット

```markdown
## 提案 #{番号}: {記入者名}さん

**分析**
- 職業「{職業名}」→ {推論された特性の要約}
- ペインポイント「{要約}」→ {検出パターン}系

**提案スキル**
| 項目 | 内容 |
|------|------|
| スキル名 | `{スキル名}` |
| 目的 | {ペインポイントを解決する1文} |
| 入力 | {推論された入力} |
| 出力 | {推論された出力} |

**このスキルでできること**
- {機能1: ペインポイントの直接解決}
- {機能2: 副次的なメリット}
- {機能3: 時間削減・品質向上}
```

## Step 5: エッジケース対応

### 5.1 職業名が曖昧な場合

```
入力: 「担当者」「スタッフ」「メンバー」
  ↓
対応: 部門/チーム列とペインポイントから職業特性を推論
```

### 5.2 ペインポイントが抽象的な場合

```
入力: 「業務効率化したい」「もっと楽にしたい」
  ↓
対応: 解決したいタスク列から具体的な作業を特定し、それを分析
```

### 5.3 複数のパターンが検出された場合

```
入力: 「データを集計してレポートを作成する作業に時間がかかる」
  ↓
検出: [情報処理系, ドキュメント系, 時間系]
  ↓
対応: 最も具体的なアクション（この場合「レポート作成」）を主目的とする
      → report-generator（集計はその手段として組み込む）
```

### 5.4 既存スキルとの重複可能性

生成前に以下をチェック：
- 類似の目的を持つスキルが既に存在しないか
- 汎用スキル（pdf, xlsx等）で対応可能ではないか
- 複数リクエストで共通化できるスキルはないか

---

## 推論の品質基準

生成された提案は以下を満たすこと：

1. **具体性**: 「何をするスキルか」が1文で明確
2. **トリガー可能性**: どんな依頼文で起動するか想像できる
3. **1 Skill = 1 Purpose**: 複数目的が混在していない
4. **実現可能性**: Claude + 既存ツールで実現可能な範囲
5. **価値**: 繰り返し使う価値がある（5回以上の使用見込み）
