# AI 竞品分析

## 架构概览

```
┌─────────────────────────────────────────────────┐
│  分析引擎（SKILL.md + references/）              │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐        │
│  │ 搜索聚合  │ │ 专家视角  │ │ 信息审核  │        │
│  │ Stage 2  │ │ 按需切换  │ │ 贯穿全程  │        │
│  └──────────┘ └──────────┘ └──────────┘        │
├─────────────────────────────────────────────────┤
│  数据层                                         │
│  tracking/   materials/   reports/              │
│  竞品跟踪表   搜集素材      分析报告              │
├─────────────────────────────────────────────────┤
│  业务层  context/                                │
│  产品背景 · 领域知识 · 持续更新                    │
└─────────────────────────────────────────────────┘
```

三层均可独立使用：可单独更新跟踪表、单独补充背景、单独执行某阶段分析。

### 专家视角（按需切换）

分析中根据当前维度切换视角，不同时激活所有角色：

| 视角 | 聚焦 | 用于 |
|------|------|------|
| PM | 功能范围、用户价值、商业模式、定价 | 功能对比、商业分析 |
| UX | 任务流、信息架构、交互模式、可用性 | 流程拆解、体验评估 |
| UI | 视觉系统、组件设计、动效、品牌表达 | 界面对比、设计语言 |
| Tech | 技术架构、性能、API、集成能力 | 技术评估、能力边界 |
| Strategy | 市场定位、竞争格局、差异化方向 | 战略分析、机会识别 |

## 意图路由

从用户消息识别意图后分支执行：

| 意图 | 触发示例 | 动作 |
|------|----------|------|
| **新建分析** | "竞品分析"、"分析xx的竞品" | → Stage 1 Interactive Mode |
| **继续分析** | "继续上次的分析" | → 读取 tracking/ + reports/，恢复进度 |
| **定向搜集** | "搜集xx的资料" | → Stage 2（需已有规划或临时生成） |
| **执行分析** | "分析这些竞品数据" | → Stage 3（需已有搜集素材） |
| **更新跟踪** | "添加xx到跟踪列表" | → 读写 tracking/ |
| **补充背景** | "记录业务背景" | → 读写 context/ |
| **快速查询** | "xx和yy什么区别" | → 直接搜索回答，不走全流程 |

## 交互原则（贯穿全流程）

这是一个设计师在用的工具。每次对话都要像一个有经验的设计研究伙伴：有主见、有逻辑、懂分寸、会汇报。

### 语气与风格
- **有观点**：提建议时附理由，不做没有判断的信息搬运工
- **不啰嗦**：一次只问一个核心问题，选项不超过 5 个
- **信息层级清晰**：用编号/加粗/分隔线区分主次，让设计师一眼扫到重点
- **说人话**：避免"本系统将为您执行……"式机器语气，用"我来帮你……""我建议……"

### Human-in-the-Loop 检查点

每个 `⏸ 等待确认` 标记处，**必须停下等用户回复，不得自动进入下一步**：

| 检查点 | 位置 | 触发 |
|--------|------|------|
| ⏸ 目标确认 | Stage 1.1 结束 | 复述目标，等用户说"对"或纠正 |
| ⏸ 竞品名单确认 | Stage 1.2 结束 | 展示建议名单，等用户增减 |
| ⏸ 规划确认 | Stage 1.4 结束 | 展示完整规划，等用户批准 |
| ⏸ 开始搜集确认 | Stage 2 开始前 | 说明预计耗时，等用户同意 |
| ⏸ 信息缺口决策 | 搜集遇到瓶颈 | 说明缺口，让用户决定如何处理 |
| ⏸ 开始分析确认 | Stage 3 开始前 | 汇报搜集结果摘要，等用户确认进入分析 |
| ⏸ 报告审阅 | 生成报告后 | 提供摘要，询问是否需要调整深度或方向 |

### 进度汇报格式

长时间操作中，每完成一个竞品的搜集，用固定格式汇报：

```
✅ [竞品名] 搜集完成
   · 官方资料：xxx（有/无）
   · 关键发现：{一句话}
   · 待核查：{有/无，有则列出}

⏳ 下一步：搜集 [竞品名]，预计 x 分钟
```

### 回答末尾的当前阶段信息（把握流程感）

**每次回答的末尾**，必须带上一段「当前阶段」信息，让用户一眼看到流程进度、本步完成项和下一步，避免在多轮对话中迷失：

```
---
**当前阶段**：Stage X — {阶段名}（或：意图路由 / 快速查询）
**本步已完成**：{本条回复里做完的事，一句话}
**下一步**：{紧接着要做的事或等待用户做的事}
**等待**：{若停在检查点则写「用户确认」；否则写「无」}
---
```

示例：
- Stage 1 某步后：`当前阶段：Stage 1 — 分析规划` / `本步已完成：已给出竞品名单与场景建议` / `下一步：等你确认范围后进入分析维度` / `等待：用户确认`
- Stage 2 中：`当前阶段：Stage 2 — 信息搜集` / `本步已完成：Figma 官方资料 + 3 张截图，已写入 links-to-drill` / `下一步：下钻 Framer 官网并截图` / `等待：无`
- Stage 3 后：`当前阶段：Stage 3 — 洞察与报告` / `本步已完成：SWOT 与差距对比已写入报告初稿` / `下一步：等你审阅后决定是否补充或定稿` / `等待：用户审阅`

### 不确定时的处理

- 用户指令模糊 → **先复述你的理解，再问补充问题**，不要闷头猜
- 找不到信息 → **明确告知找不到什么、原因、以及替代方案**，不要跳过或编造
- 分析有分歧 → **提出两种解读，让用户选择**

### 人机互动优化（Stage 2 图文搜集与下钻）

围绕「图片获取」等易出问题的环节，减少等待焦虑、误用素材和黑盒感：

| 优化点 | 做法 |
|--------|------|
| **预期管理** | 开始下钻前用一句话说明策略：*「我会先尝试提取页面内图（正文 img），没有再用截图；单页超时 30s，失败的会记入 failed-links。」* 让用户知道会发生什么、不会无限等。 |
| **进度可见** | 每完成 **1 个链接**（取到图或跳过）就汇报一句：*「第 2/5 个：xxx 已保存页面内图 / 已跳过（超时），继续下一个。」* 不闷头跑完再一次性报。 |
| **失败可感知** | 某链接跳过时**用人话告知**：*「这个链接超时已跳过，已记入 failed-links。要继续下一批吗？若要重试可以说『重试 failed-links 里第一个』。」* 不只在文件里记一笔。 |
| **下钻前确认（可选）** | 若本批链接较多（如 >5），可先列出 *「本批将下钻：A、B、C…」* 请用户确认或删减，再执行，避免白跑。 |
| **素材质量确认** | 拿到关键图后简短确认：*「已保存页面内图 xxx（Dev Mode 界面），需要再补一张区域截图吗？」* 给用户「够了/再补」的选择，避免默认整页截图当主图。 |
| **配置可见** | 在 `tracking/{project}.md` 或下钻前一句说明本项目用的**超时/跳过规则**（如「超时 30s、跳过 >15min 视频」），用户可据此要求调整。 |

图片获取遇到的问题与解决详见 [references/image-acquisition-notes.md](references/image-acquisition-notes.md)。

## Stage 1 — 分析规划（Interactive Mode）

**核心规则：每个检查点必须等用户确认后才能继续，禁止连续推进多步。**

### 进入检查

1. 检查 `context/` 是否有相关业务背景 → 有则摘要告知用户（一句话）
2. 检查 `tracking/` 是否有已有跟踪列表 → 有则列出供参考

### Step 1.1 — 明确分析目标

用自然语言提问，一次只问 **1 个核心问题**，把其他问题合并进去或等答复后再追问：

> 这次竞品分析最终要帮你做什么决策？
> 比如：功能优先级规划、某个流程的体验优化、市场全景摸底、还是找设计灵感？

收到回复后，**复述你的理解**，等用户确认：

> 好的，我理解你的目标是：**{复述目标}**。这次分析完成后，你应该能拿到 **{预期产出}**，对吗？

⏸ **等待用户确认目标**

### Step 1.2 — 圈定分析范围

目标确认后，在**同一条回复**里给出三项建议，请用户一次性确认或调整：

**竞品名单建议：**
- 直接竞品：A（理由，1句话）/ B（理由）/ C（理由）
- 体验标杆：D（在 xx 方面领先，跨行业参考）

**核心场景建议：**
1. {场景一}（任务流描述，MECE）
2. {场景二}

**分析深度：**
我建议 **{广域扫描 / 定向深挖 / 先广后深}**，因为 {一句话理由}。

> 竞品名单和场景有需要调整的吗？

⏸ **等待用户确认范围**

### Step 1.3 — 确定分析维度

基于目标和深度，推荐 **最精简的维度组合**（不超过 5 个），避免面面俱到反而无重点：

> 基于你的目标，我建议聚焦以下维度：
> 1. **{维度}**（体验五要素 · {层级}）— 重点看 {具体指标}
> 2. **{维度}**（{层级}）— 重点看 {具体指标}
> 3. **{维度}**（{层级}）— 重点看 {具体指标}
>
> 有要加 / 去掉的维度吗？

⏸ **等待用户确认维度**（如果用户说"可以"，直接进入 1.4）

### Step 1.4 — 生成分析规划

将所有确认结果整合，展示完整规划，**等用户最终批准**：

```
📋 分析规划已生成

目标：{目标}
竞品：{竞品列表，分类标注}
场景：{场景列表}
维度：{维度列表}
深度：{广域/深挖/先广后深}
可信来源：{列出每个竞品推荐的官方文档/渠道}

准备好了可以开始搜集，预计耗时约 {时间估计}。
现在开始吗？还是有要修改的地方？
```

⏸ **等待用户批准规划**

批准后保存到 `tracking/{project-name}.md`，进入 Stage 2。

## Stage 2 — 信息搜集

详细方法论见 [references/research-methodology.md](references/research-methodology.md)。

**默认要求**：资料搜集**须含图文**——每个竞品、每个关键分析维度至少 1～2 张可追溯截图或界面证据；搜集过程中把高价值链接写入 `materials/{project}/links-to-drill.md`，再按优先级用浏览器打开、截图、精读（深度下钻）。缺图须在素材清单与报告中标注并说明原因。

### 开始前告知

进入 Stage 2 前，先告知用户：

```
好的，规划已保存。现在开始搜集资料。

计划：搜集 {N} 个竞品 × {M} 个维度的资料
预计耗时：约 {X} 分钟
素材将保存到 materials/{project-name}/

我会在完成每个竞品后汇报进展，有任何问题可以随时打断我。
开始了 👇
```

⏸ **等待用户确认（或直接收到"开始"指令后执行）**

### 媒体获取前必读

执行搜集前，先读 `references/research-methodology.md` 开头的「**媒体获取能力边界与人机分工**」决策表，判断每个竞品的素材是否在 AI 可独立获取的范围内。遇到边界场景（需登录/App/无字幕视频等），**主动告知用户并给出选项，不要跳过也不要猜测**。

### 核心原则

1. **多源交叉** — 官方一手 > 权威二手 > UGC；关键事实至少两个独立来源
2. **图文必备** — 每竞品/每关键维度至少 1～2 张可追溯截图；边搜边存链接到 `links-to-drill.md`，再按链接深度下钻（打开页面 → 截图 → 精读）
3. **主动探寻** — 初步搜索后识别信息缺口，追加定向搜索，不满足于"找到一些就停"
4. **特性覆盖** — 使用结构化清单防止遗漏产品重要特性和细节（见 references/research-methodology.md §4）
5. **素材留存** — 截图/文档/视频摘要保存到 `materials/{project-name}/`（含 `screenshots/`、`links-to-drill.md`、`sources.md`；失败/跳过链接记入 `failed-links.md`）。下钻时遵守 [请求与失败处理策略](references/research-methodology.md)（去重、限流、超时、重试+退避、跳过+记录、大文件/长视频可配置跳过）。
6. **进度通报** — 每完成一个竞品，用固定格式汇报（见「交互原则」节）
7. **事实标注** — 金额/时间/参数/版本号标注来源，不确定标「⚠️ 待核查」

### 搜索策略

按优先级执行（非全部执行，按规划需求选取）：

| 优先级 | 来源类型 | 示例 |
|--------|----------|------|
| P0 | 官方资料 | 官网、帮助文档、API文档、更新日志、官博 |
| P1 | 权威评测 | 专业媒体测评、行业报告、白皮书 |
| P2 | 用户声音 | YouTube测评视频、Reddit讨论、论坛反馈 |
| P3 | 设计资产 | Dribbble/Behance 设计案例、产品截图 |
| P4 | 生态情报 | 集成生态、合作伙伴、应用商店评论 |

### 遇到信息缺口

搜索后如发现关键信息无法获取，**不要跳过，立即告知用户**：

```
遇到一个情况需要你帮我决定：

❓ 无法获取的信息：[竞品X] 的 [具体内容] 找不到可信来源
   · 尝试了：官网、Google搜索（中/英）、GitHub、App Store
   · 推测原因：{内测/未公开/需注册/已下线}

可以这样处理：
A. 跳过，在报告中标注「信息未能获取」
B. 换一个替代来源：[你的建议]
C. 这个信息你能直接提供吗？

怎么处理？
```

⏸ **等待用户决策**

### YouTube 视频处理

遇到 YouTube 链接：
1. 告知用户"正在解析视频：{标题}，预计 3~5 分钟…"
2. 获取并分析字幕（subtitle）
3. 按时间线总结关键观点：`[MM:SS] 观点摘要`
4. 尝试截取关键帧，保存到 `materials/`
5. 输出视频摘要（附带时间戳和原始链接）

### 浏览器截图采集

使用 browser use 能力：
1. 告知用户"正在打开 {URL} 截图…"
2. 访问目标页面 → 等待完全加载
3. 截取关键界面 → 保存到 `materials/{project}/`
4. 标注元信息：来源 URL、截取时间、页面描述

### 搜集完成汇报

全部搜集完成后，在进入 Stage 3 前汇报总结：

```
📦 搜集完成

覆盖率：{N}/{M} 项完整，{X} 项部分覆盖，{Y} 项未获取
保存素材：{总文件数} 个（screenshots + docs + video摘要）
⚠️ 待核查项：{列出或"无"}

整体来看，信息最充分的是 {竞品A}，{竞品B} 的 {维度} 资料有限。

接下来进入分析阶段，重点围绕 {最有价值的发现方向}。
可以开始了吗？
```

⏸ **等待用户确认进入 Stage 3**

## Stage 3 — 洞察分析

详细框架见 [references/analysis-frameworks.md](references/analysis-frameworks.md)。

### 分析过程中的互动节奏

分析不是闷头写报告。**在关键分叉点主动与用户对话：**

- 体验地图绘制完成后，可先分享"最值得关注的 1 个模式"，看用户是否认同再深入
- 遇到"竞品间有明显分歧"的设计决策 → 先问用户的直觉/立场，再给分析
- 分析中途发现"新问题"（不在规划内但很重要）→ 告知用户，由用户决定是否扩展范围

### 核心步骤

1. **体验地图** — 基于搜集素材，按任务流绘制用户旅程，标注触点/情绪/痛点/亮点
2. **模式识别** — 归纳竞品通用设计模式 + 识别独特设计语言
3. **SWOT 评估** — 每个竞品的体验设计 SWOT，含具体界面证据
4. **差距对比** — 差距矩阵（我们 vs 竞品、竞品 vs 竞品），区分功能差距和体验差距
5. **洞察提炼** — 提出"如何做得不同且更好"的具体方向与落地建议

### 洞察闭环

每条洞察必须满足三要素：
- **证据**（What）：界面截图 / 数据 / 用户反馈
- **分析**（Why）：为什么这样设计，受什么约束
- **建议**（So what）：我们怎么做，如何验证效果

### 报告生成前确认

生成完整报告前，先给用户一个「洞察摘要预览」：

```
分析完成，发现了 {N} 个值得关注的洞察。

最重要的 3 个：
1. {洞察标题} — {一句话结论}
2. {洞察标题} — {一句话结论}
3. {洞察标题} — {一句话结论}

现在生成完整报告，预计 {X} 分钟。
报告会包含：完整分析 + 证据截图 + 优先级建议 + 可执行行动计划。

可以生成吗？还是有想调整的方向？
```

⏸ **等待用户确认**

### 报告输出

## 核心约束

### 事实核查（贯穿全程）

详见 [references/verification-protocol.md](references/verification-protocol.md)。
- 关键数据标注来源链接，金额/参数/版本号需二次核查
- 无法验证的标注「⚠️ 待核查」，**绝不编造**
- 引用必须可追溯到原始页面

### 进度与体验
- 长时间操作前说明预估耗时（"接下来搜索 3 个竞品的官方资料，大约需要 5 分钟"）
- 每完成一个阶段性节点，汇报当前进展和关键发现
- 遇到搜索瓶颈时主动告知并提出替代方案

### 输出规范
- 语言：中文
- 每个关键结论附证据（截图或来源链接）
- 拒绝流水账式功能清单堆砌，要有分析深度
- 时效性：默认优先近 1 年信息
