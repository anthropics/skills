# Stage 2 — 信息搜集方法论

## 默认原则：图文必备 + 链接下钻

- **图文为默认要求**：每个竞品、每个关键分析维度，至少应有 **1～2 张可追溯的截图或界面证据**（产品界面、工作流、文档关键页、数据图等）。若某维度无法取得截图，须在素材清单与报告中标注「缺图」并说明原因（如需登录、区域限制、已下线）。
- **链接池与深度下钻**：初步搜索时**边搜边存**——将高价值链接（官网功能页、帮助文档、评测文、视频）写入 `materials/{project-name}/links-to-drill.md`，再按优先级用浏览器逐个打开、截图、精读；避免「只记结论不落链接、事后无法回溯」。

**参考**：  
- 行业动态 [asset-collection.md](../../industry-dynamics-trends/checklists/asset-collection.md)：视觉证据标准、根据链接调用浏览器截图、不足时重试策略。  
- [Youtube-clipper-skill](https://github.com/op7418/Youtube-clipper-skill)：先列资源/链接再按需深度处理、结构化输出目录；本流程用 `links-to-drill.md` + `materials/` 实现「先积链接、再下钻截图与精读」。

## 媒体获取能力边界与人机分工

**先读这里再执行**：不同媒体来源的获取方式差异很大，误判能力边界会导致 AI 无效等待或产出质量虚高。

### 分工决策表

| 场景 | AI 能做 | 能做的质量 | 推荐分工 |
|------|---------|-----------|----------|
| **公开静态产品页**（官网、帮助文档） | 提取页面内图 + 区域截图 + 文本精读 | 高 | **AI 全做** |
| **YouTube 视频（有字幕）** | 字幕解析 + 时间线摘要 + 关键帧截图 | 高 | **AI 全做** |
| **YouTube 视频（无字幕）** | 定时截图序列（近似，非真实视频理解） | 低 | **人机配合**（见下） |
| **页面内嵌视频播放**（非 YouTube） | 定时截图序列（每 N 秒截一次） | 低，非连续理解 | **人机配合**（见下） |
| **本地视频文件**（mp4/webm，用户提供） | 通过视频模态子任务做完整分析 | 高（需视频模型支持） | **AI 全做（需文件）** |
| **需要登录/购买的产品** | 无法访问 | — | **人机配合**（见下） |
| **移动端 App / 小程序** | 无法操作 | — | **人机配合**（见下） |
| **区域限制/内测产品** | 无法访问 | — | **人机配合**（见下） |
| **Figma/设计工具内部操作** | 无法操作 | — | **人机配合**（见下） |

**关键约束**：当前浏览器 MCP 不提供视频流或逐帧接口。AI 对「正在播放的视频」的理解本质是「离散时间点截图的画面分析」，不是真正的视频理解。**不要在报告中用「视频分析显示……」这样的表述夸大来源质量**，应标注「基于截图序列」。

---

### 人机配合：AI 遇到边界时的标准话术

遇到 AI 无法直接获取的内容，**不要跳过，不要猜测**，用以下方式告知用户并请求协作：

**场景 1：需要登录 / 移动 App / 区域限制**
```
这个竞品的 [功能/页面] 我没办法直接访问（原因：{需要登录/移动端/区域限制}）。

如果你有访问条件，帮我提供以下任意一种素材就能继续：
A. 直接截图：[具体需要截哪些页面/状态]
B. 录一段操作视频（mp4）：[说明录制什么操作，30s~3min即可]
C. 你来描述：[说明我需要了解哪些关键信息]

哪种方式方便？
```

**场景 2：页面内视频 / 无字幕 YouTube，需要高质量分析**
```
这个视频没有字幕，我只能通过定时截图做近似分析，
不能保证捕捉到所有关键画面，精度有限。

更好的方式是：
A. 你把视频下载成 mp4 发给我 → 我用视频模型做完整逐帧分析（更准）
B. 我用截图序列做近似分析（每 {N} 秒截一次），你知道精度有限
C. 跳过这个视频，换其他来源

选哪个？
```

**场景 3：用户提供了视频文件（mp4/webm）**
```
收到视频文件：{文件名}（{时长}）
我来用视频模型做完整分析，会生成：
  · 按时间线的操作流程拆解
  · 关键界面截帧（保存到 materials/）
  · 用户旅程注释（触点 / 情绪 / 痛点）

预计耗时约 {X} 分钟，开始吗？
```

### 本地视频文件分析工作流

当用户提供了视频文件（最高质量路径）：

```
1. 确认文件可访问（文件路径或 materials/ 下）
2. 告知用户预计时间："视频 {时长}，分析约需 {X} 分钟，我来处理 👇"
3. 通过视频模态子任务（Task tool，支持视频附件的模型）处理：
   · 生成时间线操作拆解：[MM:SS] 用户行为 + 界面状态
   · 标注关键时刻：首次操作 / 困惑节点 / 亮点时刻
4. 从视频中截取关键帧 → 保存到 materials/{project}/screenshots/
5. 整理为「用户旅程注释稿」，可直接用于 Stage 3 体验地图绘制
6. 在 sources.md 中记录：来源「用户提供视频」+文件名+分析日期
```

**注意**：视频模态分析依赖支持视频的模型（如 Gemini）。当前 Cursor 中通过 Task tool 的 `attachments` 参数传入视频文件路径触发。如当前子任务不支持视频附件，回退到「逐帧截图序列」方案并告知用户。

---

### 页面内嵌视频定时截图工作流（近似方案）

当必须从浏览器播放视频中获取信息，且用户确认接受精度有限：

```
1. 告知用户："将通过每 {N} 秒截一次来分析视频，非真实视频理解，可能遗漏细节"
2. browser_navigate 打开页面 → 找到视频播放器 → 点击播放
3. 设定间隔（默认 10 秒，可根据视频长度调整）：
   · 视频 ≤ 3 分钟 → 每 10 秒截一次（约 18 张）
   · 视频 3~10 分钟 → 每 20 秒截一次
   · 视频 > 10 分钟 → 建议用户提供录制文件，定时截图质量不够
4. 每次截图后 browser_snapshot 确认当前时间戳和字幕（如有）
5. 完成后整理截图序列，标注「定时截图/近似分析」
6. 在 sources.md 备注：「页面内视频截图序列，间隔 Xs，非逐帧视频分析」
```

## 搜索执行流程

```
分析规划（来自 Stage 1）
  │
  ├─ 1. 生成搜索关键词矩阵
  ├─ 2. 按优先级逐源搜索（同时维护「待下钻链接」清单）
  ├─ 3. 初步结果评估与信息缺口识别
  ├─ 4. 深度下钻：按链接清单打开页面 → 截图 + 精读
  ├─ 5. 追加定向搜索（填补缺口）
  ├─ 6. 高价值资料精读
  ├─ 7. 截图与素材归档（确保每竞品/每维度有图文）
  └─ 8. 汇总搜集清单，标注覆盖率与待核查项
```

## 1. 搜索关键词矩阵

针对每个竞品 × 每个分析维度，生成组合关键词：

| 维度 | 关键词模板 | 示例 |
|------|-----------|------|
| 产品概览 | `{品牌} {产品} 官网 / about / overview` | "Figma overview" |
| 功能详情 | `{产品} {功能} tutorial / how to / documentation` | "Figma auto layout tutorial" |
| 更新动态 | `{产品} {年份} update / changelog / what's new` | "Figma 2026 updates" |
| 定价 | `{产品} pricing / plans / comparison` | "Figma pricing 2026" |
| 用户评价 | `{产品} review / vs / Reddit / complaints` | "Figma vs Sketch Reddit 2026" |
| 技术架构 | `{产品} architecture / API / SDK / tech stack` | "Figma plugin API docs" |

**搜索语言**：中英文各搜一轮，英文资料通常更全面。

### 待下钻链接清单（links-to-drill.md）

在步骤 2～3 中，把值得深挖的 URL 写入 `materials/{project-name}/links-to-drill.md`，格式示例：

```markdown
| 优先级 | URL | 说明 | 下钻后 |
|--------|-----|------|--------|
| P0 | https://... | 官网 Dev Mode 页 | 截图 + 摘录 |
| P1 | https://... | 帮助文档「代码片段」 | 截图 + 摘录 |
```

下钻时：用浏览器打开 → 整页或区域截图 → 保存到 `screenshots/` 并写入 `sources.md`；必要时滚动、展开折叠后再截。**截图须可追溯至该 URL**。执行时须遵守下方「链接下钻：请求与失败处理策略」。

### 链接下钻：请求与失败处理策略

按以下流水线执行，减少卡死、重复劳动和无效请求：

| 阶段 | 策略 | 落地方式 |
|------|------|----------|
| **加入任务** | 去重 | URL 已在 `sources.md` 或 `links-to-drill.md` 的「下钻后」列标注为已处理（如 ✅ 已截图）则**不再加入**本批下钻队列。 |
| **进入队列** | 限流并发 | 每批下钻 **2～3 个链接**，完成后再取下一批；避免同时打开过多页面导致超时或卡顿。 |
| **发起请求** | 超时控制 | 单页导航 + 截图整体超时（建议 **30 秒**）：超时则**跳过**该链接，记入 `failed-links.md`，不阻塞后续链接。 |
| **成功** | 校验 → 保存 | 导航成功且拿到有效内容（有正文或截图）再写入 `sources.md` 并更新 `links-to-drill.md`「下钻后」列。 |
| **可重试错误** | 自动重试 + 退避 | 网络错误、5xx、元素未找到/ref 失效：**最多重试 2 次**，每次间隔 **5 秒**（退避）；仍失败则按「多次失败」处理。 |
| **不可重试 / 多次失败** | 跳过 + 记录 | 4xx、登录墙、区域限制、或重试 2 次后仍失败：**跳过**该链接，在 `materials/{project}/failed-links.md` 记一条（URL、原因、日期），并可在 `links-to-drill.md` 备注「已跳过：原因」。 |
| **大文件 / 超长视频** | 可配置跳过 | 若项目配置了「跳过时长 >X 分钟的视频」或「跳过超大文件」，则**不加入下钻队列**或加入时标记为「仅摘录不下载」；配置见下表。 |

**可配置项**（执行时按项目取默认或约定）：

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| 单页超时（秒） | 30 | 导航+截图总时间，超时则跳过并记录。 |
| 最大重试次数 | 2 | 可重试错误的重试次数。 |
| 重试退避（秒） | 5 | 两次重试之间的等待时间。 |
| 每批并发链接数 | 2～3 | 同时下钻的链接数，避免过载。 |
| 跳过视频时长（分钟） | 可选，如 15 | 超过此时长的视频不拉取全文/不下载，仅记链接与摘要（若有）。 |
| 跳过大文件（MB） | 可选，如 50 | 超过该大小的资源不下载，仅记 URL（若有）。 |

**failed-links.md 格式**（新建于 `materials/{project}/` 下，仅在有跳过/失败时创建）：

```markdown
| URL | 原因 | 日期 |
|-----|------|------|
| https://... | 超时 30s | 2026-03-04 |
| https://... | 403 需登录 | 2026-03-04 |
```

## 2. 多源搜索策略

### 一手来源（优先级最高）

| 来源 | 操作 | 留存 |
|------|------|------|
| 官网 | 浏览产品页、功能页，截图关键界面 | 截图 + URL |
| 帮助文档 | 搜索目标功能的使用说明 | 摘要 + 链接 |
| API/SDK 文档 | 评估技术开放度与集成能力 | 要点摘录 |
| 更新日志 | 近 6 个月的功能迭代方向 | 时间线摘要 |
| 应用商店 | 评分、评论、更新频率 | 数据记录 |
| 官方博客/公众号 | 产品理念、设计决策说明 | 引用 + 链接 |

### 二手来源

| 来源 | 操作 | 注意 |
|------|------|------|
| 专业媒体测评 | 搜索近 1 年的权威评测文章 | 注意发布日期 |
| 行业报告 | Gartner / Forrester / IDC 等 | 标注报告日期和版本 |
| YouTube 测评 | 视频解析（见下方工作流） | 注意视频时效性 |
| Reddit / 论坛 | 真实用户痛点和吐槽 | 区分个例与共性 |
| 竞品对比文章 | `A vs B` 类第三方对比 | 警惕软文/affiliates |

### 来源可信度分级

| 等级 | 来源类型 | 处理方式 |
|------|----------|----------|
| A 高可信 | 官方文档、学术论文、权威媒体 | 直接引用 |
| B 中可信 | 专业博客、KOL 评测、行业报告 | 交叉验证后引用 |
| C 低可信 | 论坛评论、社交媒体、匿名帖 | 仅作参考线索，标注来源等级 |
| D 存疑 | AI 生成内容、过期信息、利益相关方 | 必须独立验证，否则不引用 |

## 3. 信息缺口识别（主动探寻）

初步搜索后，对照分析规划中的维度做覆盖率检查：

```
对每个竞品 × 每个维度：
  ✅ 已覆盖（有可信来源）
  ⚠️ 部分覆盖（信息不完整或来源单一）
  ❌ 未覆盖（缺失）

对 ⚠️ 和 ❌ 项：
  → 调整关键词，换搜索引擎/渠道重新搜索
  → 尝试间接途径（如从竞品的竞品资料中找线索）
  → 实在找不到 → 在报告中明确标注"该信息未能获取"
```

**向用户汇报**：完成缺口分析后告知用户覆盖情况，对关键缺口征求意见（是否需要通过其他方式补充，如用户自行体验产品）。

## 5. YouTube 视频分析工作流

**超长视频**：若项目约定「跳过时长 >X 分钟的视频」（见上方可配置项），则仅记链接与标题、不拉取全文或关键帧，避免长时间占用。

### 有字幕（主路径，质量高）

```
1. 告知用户 → "正在解析视频：{标题}，预计 3~5 分钟"
2. 获取字幕 → 优先英文/中文字幕
3. 分段摘要 → 按主题切分：
   [MM:SS - MM:SS] 主题标题
   - 关键观点 1（附时间戳）
   - 关键观点 2
4. 关键帧截图 → browser_navigate 到视频页 → 拖动进度条到目标时间点 → 截图保存
5. 输出格式：
   ## 视频摘要：{标题}
   - 来源：{URL} | 发布日期：{date} | 时长：{duration}
   - 分析来源：字幕文本 ✓
   ### 关键观点（带时间戳）
   - [02:15] xxx 功能的交互设计采用了...
   - [08:30] 与竞品 Y 的主要差异在于...
   ### 截帧证据
   - ![关键帧描述](materials/{project}/{filename})
```

### 无字幕（需用户决策）

先告知用户并给出选择，**不要默默执行低质量截图序列**：

```
这个视频没有字幕，无法做文本级别的内容分析。

我能做的：
A. 定时截图序列（每 {N} 秒一张）→ 分析界面变化，但可能漏掉关键细节
B. 你把视频下载为 mp4 → 我用视频模型做完整分析（质量更高）
C. 跳过，换文字资料替代

选哪个？（A 的分析报告会标注「截图序列近似，非字幕分析」）
```

若用户选 A，执行定时截图工作流（见上方「页面内嵌视频」节），完成后输出：

```
## 视频截图分析：{标题}
- 来源：{URL} | 时长：{duration}
- ⚠️ 分析基于定时截图序列（间隔 Xs），非字幕/视频理解，可能有遗漏
### 界面变化时间线
- [约 00:30] 截图1：{界面描述}
- [约 01:00] 截图2：{界面描述}
```

## 6. 图文获取优先级：页面内图 > 截图

**原则**：**网页原有图片优先，截图作为补充或最后手段**。正文/产品区内的 `<img>` 通常就是官方产品图，直接提取 `src` 并下载保存即可，更快且更准；整页或区域截图慢、且容易只截到首屏导航而漏掉正文关键图。

| 优先级 | 方式 | 做法 |
|--------|------|------|
| 1 | **页面内图** | 下钻时解析正文区域（如 `article-body`、`.article-content`），提取 `img[src]` 直链，用 curl/fetch 下载到 `screenshots/` 或 `images/`，在 sources 注明「页面内图」及来源 URL。 |
| 2 | **区域截图** | 若页面无合适内嵌图，再用 browser 对**具体 DOM 节点**（如某 section、某 figure）做区域截图，避免整页截到无关首屏。 |
| 3 | **整页/首屏截图** | 仅作备用（如无内嵌图、且区域难以定位时）；在 sources 备注「首屏/整页，非正文关键图」以免误用。 |

**执行顺序**：打开页面 → 先查正文区是否有产品相关 `img` → 有则下载并登记 → 缺图或需补充时再对目标区域截图。

### 浏览器下钻与截图步骤（在「页面内图优先」前提下）

```
1. 从 links-to-drill.md 取 URL（去重、限流每批 2～3 个），browser_navigate，超时 30s
2. 短等待 2～3s 后 browser_snapshot；在正文/产品区域查找 img 元素，若有则取 src 直链并下载保存（优先）
3. 若无合适页面内图，再对目标区域做 browser_screenshot（或整页作备用），保存到 screenshots/ 并登记
4. sources.md 中区分类型：「页面内图」vs「截图」；整页截图须备注「首屏/整页，非正文关键图」以免误用
5. 可重试错误重试最多 2 次、间隔 5s；不可重试或多次失败记入 failed-links.md
```

**截图/取图优先级**：核心功能界面（优先用页面内图）> 关键交互状态 > 定价页 > 错误/空状态。  
**避免长时间无响应**：页面加载或点击后 DOM 会变，旧元素 ref 可能失效。建议：用短等待（2～3 秒）后 `browser_snapshot` 再操作或截图；避免单次很长等待；整页截图（fullPage）耗时会略长，可先截首屏。  
**图文不足时**：换关键词（加 UI、screenshot、workflow、documentation、demo）、换渠道（官网 → 帮助中心 → 评测/视频）、换语言；仍不足则在报告中标注「证据不足」并降低结论力度。

## 7. 素材归档

所有搜集的素材统一存放在 `materials/{project-name}/` 下：

```
materials/{project-name}/
├── screenshots/          # 界面截图（图文证据，默认必备）
├── docs/                 # 下载的文档/白皮书摘录
├── videos/               # 视频摘要笔记
├── links-to-drill.md     # 待下钻链接清单（搜索阶段积累，下钻后标注）
├── failed-links.md       # 本批下钻中跳过/失败的链接及原因（可选，有失败时创建）
└── sources.md            # 素材索引（来源、日期、用途）
```

**图文自检**：归档前检查——每个竞品 × 每个分析维度是否至少有 1 张可追溯截图；缺的标「缺图」并记入 sources.md 备注。

`sources.md` 格式：
```markdown
| 序号 | 类型 | 文件名 | 来源 URL | 获取日期 | 备注 |
|------|------|--------|----------|----------|------|
| 1 | 截图 | figma-autolayout.png | https://... | 2026-03-03 | 核心功能界面 |
```
