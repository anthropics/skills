# Stage 3 — 分析框架与洞察提炼

## 分析执行流程

```
搜集素材（来自 Stage 2）
  │
  ├─ 1. 体验地图绘制（按场景/任务流）
  ├─ 2. 模式识别（归纳通用模式 + 独特设计）
  ├─ 3. SWOT 评估（每竞品，体验视角）
  ├─ 4. 差距对比（结构化矩阵）
  ├─ 5. 洞察提炼（机会点 + 可执行建议）
  └─ 6. 生成报告
```

## 1. 体验地图绘制

基于 Stage 2 搜集的素材，为每个核心场景绘制竞品的用户旅程：

```markdown
## 场景：{场景名}

| 阶段 | 用户行为 | 竞品A | 竞品B | 情绪 |
|------|----------|-------|-------|------|
| 发现 | 了解产品 | 落地页引导... | SEO + 社区... | 😐→😊 |
| 首次使用 | 注册/开始 | 3步注册... | 社交登录... | 😊→😐 |
| 核心任务 | 完成xx | 5步完成... | 3步完成... | 😐→😊 |
| 异常处理 | 出错/卡住 | 有引导... | 无提示... | 😟 |

### 触点与情绪标注
- 🟢 亮点（Delight）：xxx
- 🔴 痛点（Friction）：xxx
- 💡 机会（Opportunity）：xxx
```

## 2. 模式识别

### 通用模式归纳

从多个竞品中提取共性设计模式：

```markdown
### 模式：{模式名称}
- **表现**：多个竞品都采用了xxx方式
- **适用场景**：当用户需要xxx时
- **竞品实例**：A 的做法 / B 的做法（附截图引用）
- **评价**：该模式有效的原因 / 局限性
```

### 独特设计识别

标注某竞品独有的设计语言或策略：

```markdown
### 竞品 A 独特之处
- **设计决策**：xxx
- **推测原因**：基于其目标用户/商业模式/技术能力
- **可借鉴度**：高/中/低（为什么）
```

## 3. SWOT 评估

从体验设计角度对每个竞品做 SWOT：

```markdown
## SWOT：竞品 A

| 维度 | 内容 | 证据 |
|------|------|------|
| **S 优势** | 交互流畅，xx操作仅需2步 | [截图1] |
| **W 劣势** | 信息层级混乱，新手找不到xx | [截图2] |
| **O 机会** | 该领域普遍缺少xx能力 | [来源链接] |
| **T 威胁** | 其xx交互已成用户习惯，难以挑战 | [用户反馈] |
```

SWOT 规则：
- 每个判断必须附具体证据（截图、数据、用户反馈）
- S/W 聚焦体验层面，不泛泛而谈"用户体验好/不好"
- O/T 要关联到我方产品可执行的动作

## 4. 差距对比

### 功能差距矩阵

```markdown
| 功能/能力 | 我们 | 竞品A | 竞品B | 差距说明 |
|-----------|------|-------|-------|----------|
| xx功能 | ✅ 基础 | ✅ 完善 | ❌ 无 | A领先，有xx细节 |
| yy能力 | ❌ 无 | ✅ 有 | ✅ 有 | 行业标配，我们缺失 |
| zz特性 | ✅ 有 | ❌ 无 | ❌ 无 | 我们的差异化优势 |
```

### 体验差距矩阵

```markdown
| 体验维度 | 我们 | 竞品A | 竞品B | 关键差异 |
|----------|------|-------|-------|----------|
| 首次上手耗时 | ~10min | ~3min | ~5min | A的引导最优 |
| 核心任务步骤 | 7步 | 4步 | 5步 | A精简了xx步骤 |
| 错误恢复 | 无引导 | 有回退 | 有撤销 | 我们缺失容错设计 |
```

### 竞品之间的差距

不仅看"我们 vs 竞品"，也分析竞品之间的差异，发现行业分歧点：
- 竞品 A 和 B 在 xx 方面采取了完全不同的策略，说明...
- 行业尚未形成统一范式的领域 = 创新空间

## 5. 洞察提炼

### 洞察闭环（每条必须完整）

```markdown
### 洞察 {序号}：{一句话标题}

**证据（What）**
在 {竞品} 的 {场景} 中观察到 {具体现象}。
[附截图/来源]

**分析（Why）**
这一设计选择背后的原因是 {用户需求/技术约束/商业考量}。
{与其他竞品对比，强化分析深度}

**建议（So what）**
- 可执行动作：{具体的设计/功能改进方向}
- 预期效果：{对哪个指标/体验维度有影响}
- 验证方式：{如何验证这个改进是否有效}
- 优先级建议：P0/P1/P2（理由）
```

### 机会点分类

将所有洞察归类：

| 类型 | 描述 | 行动建议 |
|------|------|----------|
| **快速跟进** | 行业标配但我们缺失 | 尽快补齐 |
| **体验优化** | 竞品做了但做得不好 | 做得更好 |
| **差异化创新** | 竞品都没做或做法分歧 | 探索新方案 |
| **防御巩固** | 我们的优势可能被追赶 | 加深护城河 |

## 6. 报告生成

按 `assets/report-template.md` 输出完整报告。

生成前检查：
- [ ] 每条洞察都有证据支撑
- [ ] 关键数据已标注来源
- [ ] 截图/引用可追溯
- [ ] 建议可执行且绑定了验证方式
- [ ] 无待核查项遗留（或已明确标注）

保存到 `reports/{project-name}-{date}.md`，并更新 `tracking/` 中的项目状态。
