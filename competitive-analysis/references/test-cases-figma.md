# 竞品分析 Skill 测试用例 — 以 Figma 为例

> **用途**：验证 competitive-analysis 技能在「云端/协作设计工具」场景下的意图路由、交互节奏与产出规范。  
> **示例领域**：Figma 及竞品（Sketch、即时设计、Penpot、Framer 等）。

---

## 执行测试口令

在新对话中 @ competitive-analysis 或打开本技能后，任选下面一句发送即可启动测试：

| 用途 | 口令（复制即用） |
|------|------------------|
| **启动完整流程测试** | `按 competitive-analysis/references/test-cases-figma.md 执行竞品分析技能测试，以 Figma 为例，从 Stage 1 开始，等我逐步确认。` |
| **只测意图路由** | `按 test-cases-figma 做意图路由测试：依次用 R1～R7 的示例输入验证，告诉我每个意图是否走对。` |
| **直接走 Figma 分析** | `我们做类似 Figma 的云端协作设计工具，想对 Figma、即时设计、Penpot 做竞品分析，重点看「评论与批注」和「实时协作」的体验，用来做功能优先级和交互优化。帮我规划一下并执行。` |

首次测试建议用「启动完整流程测试」，便于逐轮确认检查点；回归或抽查时可用「只测意图路由」或「直接走 Figma 分析」。

---

## 测试前准备

- 确保技能已被 Cursor 加载（@competitive-analysis 或描述匹配时触发）。
- 测试环境建议在 `competitive-analysis/` 目录或包含该 skill 的 workspace 下对话。
- 可先清空或备份 `tracking/`、`materials/`、`reports/` 以便观察生成物。

## 一、意图路由测试

| 用例 ID | 意图 | 用户输入（示例） | 预期动作 | 通过标准 |
|--------|------|------------------|----------|-----------|
| R1 | 新建分析 | 「我想做竞品分析，我们在做类似 Figma 的协作设计工具，想看看 Figma、Sketch、即时设计这些竞品的协作和评论体验」 | 进入 Stage 1，先问「这次分析最终要帮你做什么决策？」 | 不直接开始搜集，先明确目标 |
| R2 | 继续分析 | 「继续上次的 Figma 竞品分析」 | 读取 `tracking/*.md` 与 `reports/*.md`，摘要进度并问下一步 | 能列出已有规划/报告并延续 |
| R3 | 定向搜集 | 「搜集一下 Penpot 的协作和评论功能资料」 | 进入 Stage 2（若有规划则按规划维度搜集；若无则临时生成范围） | 产出进入 `materials/{project}/`，并做进度汇报 |
| R4 | 执行分析 | 「素材都齐了，分析这些竞品数据并给结论」 | 进入 Stage 3，做体验地图/SWOT/差距/洞察 | 产出含证据与建议，且先给洞察摘要再问是否生成报告 |
| R5 | 更新跟踪 | 「把 Framer 加到我们设计工具竞品跟踪列表里」 | 读写 `tracking/*.md`，更新竞品清单 | tracking 文件中出现 Framer 及分类/状态 |
| R6 | 补充背景 | 「记录一下我们产品背景：我们做国内团队的协作设计工具，目前强在组件库和设计规范，弱在实时协作」 | 读写 `context/*.md` | context 下出现结构化业务背景 |
| R7 | 快速查询 | 「Figma 和 Sketch 在协作这块主要区别是什么？」 | 直接搜索回答，不走 Stage 1/2/3 | 有结论、有依据，不创建 tracking/reports |

## 二、完整流程测试（Figma 竞品分析）

**场景**：产品/设计团队希望系统分析「协作设计工具」竞品，支撑功能优先级与体验优化。

### TC-F1：Stage 1 — 分析规划（Interactive Mode）

| 步骤 | 角色 | 操作 | 预期结果 | 检查点 |
|------|------|------|----------|--------|
| 1.1 | 用户 | 发送：「帮我对 Figma 做竞品分析，我们在做协作设计产品，想重点看评论与批注、实时协作这两块」 | AI 复述目标并问「这次分析最终要帮你做什么决策？」或等价一问 | ⏸ 一次只问一个核心问题 |
| 1.2 | 用户 | 回复：「主要是功能优先级和体验优化，想找最佳实践」 | AI 复述目标，等用户确认 | ⏸ 目标确认前不进入 1.2 |
| 1.3 | 用户 | 回复：「对」 | AI 给出竞品名单建议（含直接竞品 + 可选体验标杆）、核心场景、分析深度 | 名单中宜含 Figma、Sketch、即时设计等；有理由 |
| 1.4 | 用户 | 回复：「竞品就 Figma、即时设计、Penpot，场景可以」 | AI 给出分析维度建议（≤5 个），如结构层/框架层/表现层等 | ⏸ 维度确认 |
| 1.5 | 用户 | 回复：「可以」 | AI 输出完整规划（目标/竞品/场景/维度/深度/可信来源/预计耗时），问「现在开始吗？」 | ⏸ 规划批准 |
| 1.6 | 用户 | 回复：「开始」 | 规划写入 `tracking/{project-name}.md`，进入 Stage 2 前告知计划与耗时 | 存在 tracking 文件 |

### TC-F2：Stage 2 — 信息搜集

| 步骤 | 角色 | 操作 | 预期结果 | 检查点 |
|------|------|------|----------|--------|
| 2.1 | - | Stage 1 批准后自动/用户说「开始搜集」 | AI 说明计划（N 个竞品 × M 维度）、预计耗时、素材保存位置，并等待确认后开始 | ⏸ 开始搜集确认 |
| 2.2 | - | 每完成一个竞品 | 汇报格式含：✅ [竞品名] 搜集完成；官方资料/关键发现/待核查；⏳ 下一步 | 固定进度汇报格式 |
| 2.3 | 用户 | 若 AI 发现信息缺口 | AI 说明无法获取的内容、已尝试途径、可选处理（A/B/C），等用户选择 | ⏸ 信息缺口决策 |
| 2.4 | - | 全部搜集完成 | 汇报覆盖率、素材数量、待核查项，并问「可以开始分析了吗？」 | ⏸ 确认进入 Stage 3；素材在 materials/ |

### TC-F3：Stage 3 — 洞察分析

| 步骤 | 角色 | 操作 | 预期结果 | 检查点 |
|------|------|------|----------|--------|
| 3.1 | - | 用户确认进入分析 | AI 进行体验地图、模式识别、SWOT、差距对比、洞察提炼（可分步与用户互动） | 洞察含证据/分析/建议 |
| 3.2 | - | 分析完成、生成报告前 | AI 给出洞察摘要（至少 Top 3）、报告将包含内容、预计时间，问「可以生成吗？」 | ⏸ 报告生成前确认 |
| 3.3 | 用户 | 回复：「生成」 | 按 `assets/report-template.md` 生成报告，保存到 `reports/{project-name}-{date}.md` | 报告结构符合模板；有来源索引与待核查汇总 |
| 3.4 | - | 报告生成后 | AI 给出执行摘要（约 3 句话）并问是否需要深挖或调整方向 | ⏸ 报告审阅（开放式） |

## 三、Figma 示例 — 可直接粘贴的用户话术

便于快速触发技能与各阶段，复制即用。

### 新建分析（走完整流程）

```
我们做类似 Figma 的云端协作设计工具，想对 Figma、即时设计、Penpot 做竞品分析，
重点看「评论与批注」和「实时协作」的体验，用来做功能优先级和交互优化。帮我规划一下并执行。
```

### 目标确认后（模拟逐轮回复）

1. **目标**：「主要是功能优先级和体验优化，找最佳实践和差距。」
2. **范围**：「竞品就 Figma、即时设计、Penpot，场景按你建议的来。」
3. **维度**：「就按你提的 3 个维度，可以。」
4. **规划**：「开始吧。」
5. **搜集**：「开始搜集。」
6. **分析**：「可以开始分析了。」
7. **报告**：「生成报告吧。」

### 快速查询（不走全流程）

```
Figma 和即时设计在「评论与批注」上的主要区别是什么？
```

### 继续分析

```
继续上次的 Figma 竞品分析，把 Framer 也加进去做对比。
```

### 补充背景

```
记录业务背景：我们产品是面向国内团队的协作设计工具，目前强在组件库和设计规范，弱在实时协作和评论体验。
```

## 四、检查点清单（Human-in-the-Loop）

执行测试时确认以下节点 **必须停顿等待用户**，不得自动跳步：

- [ ] **目标确认** — Stage 1.1 结束，复述目标后等待
- [ ] **竞品名单确认** — Stage 1.2 结束，展示名单后等待
- [ ] **规划确认** — Stage 1.4 结束，展示完整规划后等待
- [ ] **开始搜集确认** — Stage 2 开始前，说明耗时后等待
- [ ] **信息缺口决策** — 搜集中遇到缺口时，给出选项后等待
- [ ] **开始分析确认** — Stage 3 开始前，汇报搜集结果后等待
- [ ] **报告生成确认** — 给出洞察摘要后等待
- [ ] **报告审阅** — 报告生成后询问是否深挖/调整（可开放结束）

## 五、产出物检查

| 产出 | 路径 | 检查项 |
|------|------|--------|
| 跟踪表 | `tracking/{project-name}.md` | 含目标、竞品清单、搜集/分析进度、历史报告链接 |
| 素材 | `materials/{project-name}/` | 截图/docs/videos 等，可有 `sources.md` 索引 |
| 报告 | `reports/{project-name}-{date}.md` | 符合 `report-template.md`；有执行摘要、竞品概览、场景分析、SWOT、差距矩阵、洞察与建议、行动计划、来源索引、待核查汇总 |
| 背景 | `context/*.md` | 若执行过「补充背景」，应有结构化产品/竞争/关注重点 |

---

## 六、回归测试建议

1. **先跑 R1 + TC-F1**：验证新建分析是否严格按 Stage 1 逐步确认，不跳步。
2. **再跑 R7**：验证快速查询不创建 tracking/reports。
3. **有素材后跑 R4 + TC-F3**：验证 Stage 3 洞察闭环与报告模板。
4. **最后跑 R2**：在已有 tracking/reports 下验证「继续分析」能正确恢复上下文。

完成以上测试并勾选检查点后，可认为以 Figma 为例的技能测试通过。
