# 图片获取：遇到的问题与解决

> 基于 Figma D2C 项目实际执行整理，用于避免重复踩坑并优化人机协作。

## 遇到的问题与解决

### 1. 长时间无响应、像卡死

**现象**：点击「Dismiss」或截屏时，长时间没有返回，用户不知道是卡死还是在等。

**原因**：
- 页面加载或点击后 DOM 重绘，之前的元素 ref（如 `e6`）失效，自动化在查找元素时重试或挂起；
- 单次 `browser_wait_for` 设得过长（如 10s），期间没有任何中间反馈。

**解决**（已写入 [research-methodology.md](research-methodology.md)）：
- **短等待 + 再取 ref**：操作前先 `browser_wait_for` 2～3 秒，再 `browser_snapshot` 取**当前**页面元素 ref，再点击或截图，不依赖旧 ref；
- **避免单次过长等待**：用多次 2～3s 的短等待 + 快照检查，替代一次 10s 等待；
- **超时与跳过**：单页整体超时 30s，超时则跳过该链接并记入 `failed-links.md`，不阻塞后续。

---

### 2. 截到首屏而非正文关键图

**现象**：整页截图只截到导航/首屏，真正需要的产品图在正文里（如 `article-body > p[3] > img`），没有入镜。

**原因**：整页截图从视口顶部开始，长页面时正文内嵌图在下方，未进入截图范围或占比小；且截图慢、对滚动/分辨率敏感，不一定准。

**解决**（已写入 research-methodology §6）：
- **优先级**：**网页原有图片 > 区域截图 > 整页截图**；
- **先取页面内图**：下钻时解析正文区域（如 `article-body`、`.article-content`），提取 `img[src]` 直链，用 curl/fetch 下载保存，在 sources 注明「页面内图」；
- **整页截图仅作备用**：无合适内嵌图或需补全时再用；sources 中整页截图须备注「首屏/整页，非正文关键图」以免被误当主图使用。

---

### 3. 缺少系统的请求与失败处理

**现象**：下钻时没有去重、超时、重试、失败记录，容易重复抓、卡死、或丢失败信息。

**解决**（已写入 research-methodology「链接下钻：请求与失败处理策略」）：
- **加入任务 → 去重**：URL 已在 sources 或 links-to-drill「下钻后」标注则不再加入；
- **进入队列 → 限流**：每批 2～3 个链接；
- **发起请求 → 超时**：单页 30s，超时跳过并记录；
- **可重试错误 → 重试 + 退避**：最多 2 次，间隔 5s；
- **不可重试/多次失败 → 跳过 + 记录**：记入 `failed-links.md`（URL、原因、日期）；
- **大文件/超长视频**：可配置直接跳过（见可配置项表）。

---

## 小结表

| 问题 | 原因 | 解决 |
|------|------|------|
| 长时间无响应 | ref 失效、单次长等待 | 短等待 + snapshot 再操作；超时 30s 跳过 |
| 截到首屏漏正文图 | 整页截图从顶部开始、慢且不准 | 页面内图优先，提取 img src 下载；整页截图仅备用 |
| 重复/卡死/失败无记录 | 无去重、无限流、无超时与重试策略 | 去重、限流、超时、重试退避、failed-links 记录 |
