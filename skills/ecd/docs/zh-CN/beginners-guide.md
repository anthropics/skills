## 5 分钟：ECD 到底是什么？

### 一句话版本

> **ECD 是一个帮你"先想清楚再动手"的 AI 编程助手方法论。它在你写代码之前，强迫你把需求想清楚，然后严格按照计划执行。**

### 类比版本

```
盖房子：
  ❌ 没有 ECD：你说"盖个房子"→ 工人直接砌墙 → 你说"不对我要两层"→ 拆了重来
  ✅ 有 ECD：  你说"盖个房子"→ 建筑师追问"几层？几个房间？预算？"→ 画图纸 →
                你确认 → 工人按图纸盖 → 验收

写代码：
  ❌ 没有 ECD：你说"加个认证"→ AI直接写代码 → 你说"不对我要JWT不是Session"→ 重写
  ✅ 有 ECD：  你说"加个认证"→ ECD 追问"JWT还是Session？哪些路由保护？"→ 冻结方案 →
                你确认 → AI按方案写代码 → 验收
```

### ECD 的四个步骤

```
你的需求 →  pre(追问澄清) →  plan(制定方案) →  code(写代码) →  achieve(验收)
           "你到底要什么？"   "我们这样做"     "按计划执行"    "看看对不对"
```

### 什么时候应该用 ECD？

| 你的需求是这样的 | 用 ECD 吗？ |
|-----------------|:----------:|
| "加个暗色模式"（简单明确） | 🟡 可选（L1 轻量模式） |
| "做个注册API"（中等功能） | 🟢 推荐（L2 标准模式） |
| "加上认证系统"（安全敏感） | 🔴 **强烈推荐**（L3 完整模式） |
| "让应用变快"（模糊需求） | 🔴 **必须用**（避免白费功夫） |
| "修个拼写错误"（极小改动） | ❌ 用原生 Claude 就行 |
| "重构一段代码"（行为不变） | ❌ 用 Superpowers 更合适 |

## 决策树：我该用 ECD 吗？

```
你有一个编程任务
    │
    ├── 任务是什么类型？
    │   │
    │   ├── 修拼写/改字号/加tooltip（<10行代码）
    │   │   └── ❌ 不用 ECD。直接在 Claude Code 里说要做啥。
    │   │
    │   ├── UI调整/单组件/简单功能（明确知道要什么）
    │   │   └── 🟡 可选 ECD L1。用 "/ecd" 触发，自动走轻量模式。
    │   │
    │   ├── API开发/中等功能/多文件改动
    │   │   └── 🟢 推荐 ECD L2。用 "/ecd" 触发，自动走标准模式。
    │   │
    │   ├── 认证/支付/数据安全/架构变更
    │   │   └── 🔴 强烈推荐 ECD L3。用 "/ecd" 触发，自动走完整模式。
    │   │
    │   └── 需求连你自己都说不清楚（"让应用更好"、"优化一下"）
    │       └── 🔴 必须用 ECD。它会帮你把需求搞清楚再动手。
    │
    └── 但如果你：
        ├── 在做快速原型（先跑起来再说）→ ❌ 不用 ECD
        ├── 在重构（行为不变，只改内部实现）→ ❌ 用 Superpowers
        └── 已有 ECD 项目做小修改 → 🟢 用 ECD 增量模式
```

## 第一次使用：跟着做一遍

> **场景**：你有一个 React 项目，想加暗色模式切换。
>
> 我们从头到尾走一遍，让你看看 ECD 实际是怎么用的。

### 第 0 步：确认 ECD 已安装

在 Claude Code 中输入 `/ecd`，如果看到 ECD 技能的提示信息，说明已安装。

如果没安装，推荐使用插件方式（自动更新）：

打开 `~/.claude/settings.json`（Windows: `%USERPROFILE%\.claude\settings.json`），添加：

```json
{
  "extraKnownMarketplaces": {
    "ecd-marketplace": {
      "source": { "source": "github", "repo": "Zyc-Bryce/ECD" }
    }
  },
  "enabledPlugins": {
    "ecd@ecd-marketplace": true
  }
}
```

> 如果文件已有其他内容，把 `extraKnownMarketplaces` 和 `enabledPlugins` **合并进去**，不要覆盖。

重启 Claude Code 即可。或者用命令行方式：

```powershell
# 克隆到 skills 目录（Windows PowerShell）
git clone https://github.com/Zyc-Bryce/ECD.git $env:USERPROFILE\.claude\skills\evolutionary-constraint-development
```

### 第 1 步：提出需求

在 Claude Code 中输入：

```
/ecd 给这个 React 项目加个暗色模式切换
```

### 第 2 步：ECD 自动评估复杂度

你会看到 ECD 在后台静默分析（不需要你参与）：

```
[ECD 复杂度评估]
Q1 代码影响面：≤3 文件 → L1
Q2 安全风险：UI样式 → L1
Q3 需求清晰度：明确 → L1
最终等级：L1 ECD-Lite
路由：A-Lite → J-Lite → code → achieve-Lite
```

> **你不用做任何事**，ECD 自动判定这是简单任务，走轻量流程。

### 第 3 步：A-Lite 阶段（ECD 向你提问）

ECD 会问最多 3 个关键问题：

```
🤖 ECD [A-Lite]：在开始之前，需要确认几个关键点：

Q1: 项目目前用的 CSS 方案是什么？
    A) CSS 变量（:root 中定义）  B) Tailwind CSS  C) styled-components  D) 其他

Q2: 暗色模式需要跟随系统偏好吗？
    A) 是，首次打开跟随 prefers-color-scheme
    B) 不，始终默认亮色，用户手动切换

Q3: 用户的主题选择需要持久化吗（刷新后保留）？
    A) 是，存 localStorage  B) 不，刷新后恢复默认
```

你回答：

```
A, A, A
```

### 第 4 步：审批确认

ECD 呈现一个简洁的审批包：

```
🤖 ECD [审批包 — L1 Lite]

🎯 目标：添加暗色模式切换（CSS变量方案，跟随系统偏好，localStorage持久化）

✅ 保留范围：
  • CSS变量定义（亮色+暗色两套）
  • ThemeContext + useTheme hook
  • 切换按钮组件
  • App根组件集成
  • localStorage持久化
  • 系统偏好检测（prefers-color-scheme）

❌ 排除范围：
  • 第三方组件库的暗色适配
  • 过渡动画

✅ 验收标准：
  1. 切换按钮可见且可点击
  2. 刷新页面后保持用户选择
  3. 首次访问跟随系统偏好
  4. 无首次加载闪烁（FOUC）

🔒 已冻结决策：CSS变量方案、手动选择覆盖系统偏好

确认？(回复"确认"继续，或提出修改)
```

你回复：

```
确认
```

### 第 5 步：J-Lite + code（ECD 自动执行）

审批后，以下步骤**自动在后台完成**，不需要你参与：

```
🤖 ECD [J-Lite]：编译交接包... ✓
🤖 ECD [code]：
  实现单元 1/5：CSS变量定义 ✓
  实现单元 2/5：ThemeContext ✓
  实现单元 3/5：Toggle组件 ✓
  实现单元 4/5：App集成 ✓
  实现单元 5/5：localStorage持久化 ✓
```

### 第 6 步：achieve（验收关闭）

```
🤖 ECD [achieve-Lite]：
  验证 1/4：切换按钮可见可点击 ✓
  验证 2/4：刷新后保持选择 ✓
  验证 3/4：首次跟随系统偏好 ✓
  验证 4/4：无FOUC闪烁 ✓
  
  裁决：✅ archived（归档关闭）
```

### 完成！

从需求到代码，**全程你只回答了 3 个问题 + 点了 1 次确认**。这是 L1 轻量级流程。

## 四个核心命令

ECD 有四个命令，对应四个阶段。但你通常只需要记住：**输入 `/ecd` + 你的需求，剩下的事情 ECD 会自动处理。**

| 命令 | 全称 | 谁在做 | 干什么 | 你需要做什么 |
|------|------|--------|--------|-------------|
| `pre` | preprocess | ECD | 追问澄清，冻结"要做什么" | 回答 ECD 的问题 |
| `plan` | planning | ECD + 子Agent | 制定方案，冻结"怎么做" | 确认审批包 |
| `code` | coding | ECD | 严格按照方案写代码 | 看着就行 |
| `achieve` | achieve | ECD | 验证结果，决定是否通过 | 看看结果 |

### pre — "你到底要什么？"

```
类比：建筑师在画图纸之前，反复确认你的需求。

ECD 会：
✅ 反问你不清楚的地方（"你说的'快'是指首屏加载还是点击响应？"）
✅ 指出矛盾（"你要暗色模式但项目用的是内联样式，这有冲突"）
✅ 挖掘你没说出来的需求（"认证系统需要忘记密码流程吗？"）
✅ 冻结所有决策（"确认用JWT方案，不回退"）

你不会看到：
❌ ECD 直接开始写代码
❌ ECD 猜测你的意图
❌ ECD 跳过它认为"不重要"的问题
```

### plan — "我们这样做"

```
类比：建筑师画出详细图纸，包括材料清单、施工顺序、验收标准。

ECD 会：
✅ 生成技术方案（"方案A: JWT + bcrypt；方案B: Session + Redis"）
✅ 拆解为实现单元（"第1步建数据模型，第2步写验证中间件..."）
✅ 启动独立审查（用另一个AI检查方案有没有漏洞）
✅ 编译为"代码交接包"（下一阶段严格按这个包执行）

你不会看到：
❌ plan阶段写任何业务代码
❌ 跳过审查直接编译交接包
❌ 交接包里有模糊描述（"实现认证功能"——这不叫交接包）
```

### code — "按图纸施工"

```
类比：建筑工人拿着图纸施工，不自行修改设计。

ECD 会：
✅ 严格按交接包执行（不发明新功能）
✅ 每个实现单元完成后自动验证
✅ 遇到模糊之处标记为"计划缺陷"，回流到pre阶段

ECD 不会：
❌ "我觉得用户可能还需要XX功能" → 禁止！功能由pre/plan决定
❌ 跳过验证直接进入下一个单元
❌ 自己决定验收标准
```

### achieve — "验收通过了吗？"

```
类比：监理验收房子，检查每个房间是否符合图纸。

ECD 会：
✅ 运行所有验证命令
✅ 检查首次使用体验（"打开页面有没有明显的布局崩坏？"）
✅ 做出明确裁决：

裁决有三种：
  • archived —— 通过了，关闭
  • achieved_with_followups —— 通过了但有遗留问题需要后续处理
  • not_achieved —— 不通过，需要返工

不会：
❌ "基本可以了"（含糊其辞）
❌ 跳过失败的验证
❌ 把未通过的标记为通过
```

## 三级路由：L1/L2/L3 怎么选

ECD 会自动判断你的任务属于哪个等级。但你也可以手动指定。

### 等级速查

| | L1 轻量级 | L2 标准级 | L3 完整级 |
|---|----------|----------|----------|
| **俗称** | "快速模式" | "标准模式" | "完整模式" |
| **适合** | 改样式、修Bug、加组件 | 做API、中等功能 | 认证、支付、架构变更 |
| **流程长短** | 很短（2-3阶段） | 中等（5-7阶段） | 很长（10阶段） |
| **会问你几个问题** | 最多3个 | 3-5个 | 可能10+个 |
| **AI自己审自己？** | 不审查 | 可选1-2次审查 | 强制5次独立审查 |
| **耗时** | 快（类似直接写代码） | 中等 | 慢（但安全） |
| **Token消耗** | 15k-30k | 35k-55k | 65k-105k |

### 自动分类的原理

ECD 在后台回答 3 个问题来决定你的等级：

```
问题1：要改多少文件？
  1-3个 → L1 | 4-10个 → L2 | 10+个 → L3

问题2：出错了有多严重？
  只是UI不好看 → L1 | 功能会出错 → L2 | 可能丢数据/被攻击 → L3

问题3：需求说得清楚吗？
  非常清楚 → 不升级 | 有些模糊 → 可能升级 | 很模糊 → 强制L3

最终等级 = 三个问题中最高的一级
```

### 你如何手动控制

```
"用 /ecd L1 加个暗色模式"       → 强制轻量级
"用 ECD Full 做认证系统"         → 强制完整级
"用 ECD standard 加个API"        → 强制标准级
"--tier lite 修个bug"            → 强制轻量级
```

### 等级会升级吗？

会。如果你选了 L1，但 A-Lite 阶段发现了隐藏的复杂度，ECD 会告诉你并建议升级：

```
🤖 ECD：[L1→L2 升级] 原本判定为 L1，但在分析中发现：
  • 这个"简单API"实际上涉及6个文件（超过L1阈值）
  • 涉及用户数据处理（安全风险高于预期）
  
  建议升级到 L2 Standard。是否同意？
```

## 场景速查表

### 场景 1："加个按钮/改个颜色/修个拼写"

```
不用 ECD。
直接在 Claude Code 里说：
  "把登录按钮改成绿色"
  "导航栏有个拼写错误'Loign'改成'Login'"
```

### 场景 2："加个暗色模式/多语言/分页"

```
用 /ecd → 自动走 L1
流程：3个问题 → 确认 → 自动写代码 → 验收
耗时：2-3轮交互
```

### 场景 3："做一个用户注册API"

```
用 /ecd → 自动走 L2
流程：5-7个问题 → 2-3个方案供选择 → 审查 → 确认 → 写代码 → 验收
耗时：3-5轮交互
额外收益：会发现邮箱唯一性竞态、用户枚举漏洞等安全问题
```

### 场景 4："加上完整的认证系统"

```
用 /ecd → 自动走 L3（安全敏感）
流程：10+个问题 → 多方案 → 独立AI攻击审查 → 红蓝对抗 → 确认 → 写代码 → 验收
耗时：4-8轮交互，但交付的认证方案覆盖了原生Claude会遗漏的安全漏洞
```

### 场景 5："让应用变快"

```
必须用 /ecd → Q3模糊 → 强制L3
ECD 会阻止你立即写代码，先追问：
  "哪里慢？首屏？交互？API？"
  "有Lighthouse数据吗？"
  "目标指标是什么？LCP<2.5s？FID<100ms？"
在需求明确之前，ECD不会让你浪费Token做猜测性优化。
```

### 场景 6："这个应用需要 GDPR 合规"

```
用 /ecd → L3（PII/法律合规）
红蓝对抗会发现：数据删除的软删除vs硬删除、第三方数据共享声明、cookie同意更新等
```

### 场景 7："我想做一个 todo 应用"

```
用 /ecd → 通常 L2（如果只说"做todo"没有细节→可能L3）
ECD 会问：数据存哪？需要分类吗？需要截止日期吗？多用户？拖拽排序？
避免你收到一个过于简单或过于复杂的todo应用。
```

### 场景 8："上次用 ECD 做的功能，现在要加个小东西"

```
用 ECD 增量模式：
  "在上次 ECD handoff 里给登录页加个 loading spinner"
  → ECD 检测到已有bundle → 增量模式 → 只更新交接包 → Token仅~10k
```

### 场景 9："重构代码但不改功能"

```
不用 ECD。用 Superpowers：
  /superpowers:test-driven-development → 先写测试锁定行为 → 重构 → 测试通过
```

### 场景 10："bug修复（功能已经定义好了）"

```
不用 ECD 的完整流程。直接在 Claude Code 中描述bug。
或者用 Superpowers:systematic-debugging。
除非这个bug的修复涉及产品行为的重新定义。
```

## 常见错误与如何纠正

### ❌ 错误 1：所有任务都用 `/ecd`

```
症状：修个拼写错误也打 /ecd
后果：2分钟的活变成10分钟的流程
纠正：<10行代码的改动直接用原生Claude
```

### ❌ 错误 2：用 `/ecd` 但跳过审批直接说"确认"

```
症状：ECD 给了审批包，你没仔细看就确认了
后果：冻结了错误的需求，后面全白做
纠正：审批包是 ECD 最重要的环节——花30秒仔细看
```

### ❌ 错误 3：需求说不清楚就期望 ECD 帮你猜

```
症状：说"做个好东西"然后期望 ECD 自动理解
后果：ECD Stage A 会反复追问，你感到烦躁
纠正：如果真说不清楚需求，这正是 ECD 的价值——
      它会帮你理清。耐心回答追问，或者先说"我自己也还没想清楚，帮我一起梳理"
```

### ❌ 错误 4：在 code 阶段要求改产品行为

```
症状：code 执行到一半说"不对，应该再加个XX功能"
后果：ECD 必须回流到 Stage A 重新规划
纠正：产品行为在 pre/plan 阶段冻结。code 阶段只提"按照已确认的方案，
      这里技术上有个问题"——不提案产品层面的改动。
      如果确实要改产品行为，说"需要回流到 Stage A"。
```

### ❌ 错误 5：安全敏感任务强制用 L1

```
症状：做支付功能但说"用 ECD L1 快点"
后果：跳过了红蓝对抗，安全漏洞被忽略
纠正：支付/认证/隐私/合规 → 强制 L3。这不是"慢"，这是"必要的投资"。
```

### ❌ 错误 6：把 ECD 当成写代码的工具

```
症状：以为 ECD 是一种"更好的代码生成器"
后果：不理解为什么 ECD 花这么多时间在"不写代码"的事情上
纠正：ECD 的核心价值在 pre/plan（"想清楚"），不是 code（"写代码"）。
      ECD 是一个需求工程工具+交付方法论，不只是代码生成器。
```

### ❌ 错误 7：不利用增量模式

```
症状：已有 ECD bundle，改一个小地方又跑完整 /ecd
后果：不必要的 Token 浪费
纠正：对已有bundle的修改，用增量关键词触发：
      "修改"、"增加"、"调整" → ECD 自动识别为增量
```

## FAQ：你最可能问的 20 个问题

### Q1: ECD 会让我写代码变慢吗？

**L1（简单任务）：几乎不快不慢。** ECD 的 L1-Lite 流程 Token 消耗（18.5k）和原生 Claude（15.2k）接近，多了 3 个追问。

**L2（中等任务）：会多花一点时间，但返工大幅减少。** 原生 Claude 在中等任务上有 37% 的返工率——"快速写完"之后还要花更多时间改。

**L3（复杂任务）：前期花时间，但避免了大返工。** 原生 Claude 在复杂任务上的返工率高达 65%。

### Q2: 我什么时候绝对不该用 ECD？

- 修拼写错误、改颜色、加 tooltip（<10行代码）
- 重构（行为不变，只改内部实现）
- 快速原型/探索性编码（"我也不知道要什么，先写出来看看"）
- 已经有明确技术规格书的任务（需求已经冻结了）

### Q3: ECD 和 Superpowers 应该选哪个？

| 场景 | 选哪个 |
|------|--------|
| 需求模糊，需要先想清楚 | ECD |
| 需求明确，需要工程质量（TDD/review/worktree） | Superpowers |
| 两者都需要 | ECD pre/plan + Superpowers execute |
| 安全敏感 | ECD（红蓝对抗不可替代） |

### Q4: ECD 会一直问我问题吗？我烦了怎么办？

L1 最多 3 个问题。L2 通常 3-5 个。L3 可能 10+ 个。

如果你觉得问题太多，可能是：
- 你的任务确实是 L3 复杂度（比如"加认证"隐藏在"简单需求"背后）
- 你的需求表述太模糊，ECD 在帮你理清

如果确实不想回答：说"用 L1 快速模式"——但接受快速模式可能遗漏问题的风险。

### Q5: 我不懂技术，能用 ECD 吗？

**可以，但你至少需要知道你的项目是什么技术栈。** ECD 会问技术问题（"项目用的什么框架？"），如果你不确定，ECD 可以帮你探查项目结构来回答。

### Q6: ECD 能帮我做设计决策吗？

**能。这正是 Stage B（方案发散）的用途。** ECD 会给出 2-3 个技术方案并说明各自的优劣，你可以选择。

### Q7: 如果 ECD 的方案我不满意怎么办？

在审批阶段提出修改意见。ECD 会重新调整方案。**审批之前，一切都可以改。**

### Q8: ECD 执行到一半我改变主意了怎么办？

告诉 ECD。它会判断变更的影响范围：
- 改产品语义 → 回流 Stage A
- 改实现方案 → 回流 Stage C
- 纯增量 → 回流 Stage J
- Bug 修复 → 直接 code

### Q9: 我能看到 ECD 的"后台工作"吗？

能。ECD 的所有中间产物都保存在 bundle 目录中（markdown 文件）。你可以随时打开查看。

### Q10: ECD 的 bundle 是什么？我需要手动管理吗？

Bundle 是一组 markdown 文件，记录 ECD 从需求到代码的完整过程。**你不需要手动管理**——ECD 自动创建和更新。

### Q11: Token 消耗到底有多大？会不会很贵？

| 任务等级 | Token 范围 | 类比（如果用 GPT-4） |
|---------|-----------|-------------------|
| L1 Lite | 15k-30k | ~$0.03-0.06 |
| L2 Standard | 35k-55k | ~$0.07-0.11 |
| L3 Full | 65k-105k | ~$0.13-0.21 |

> 注意：Token 价格取决于你使用的模型。以上按 GPT-4 级别估算。

### Q12: ECD 的子Agent是什么？我需要管吗？

子Agent 是 ECD 调用的**独立 AI 实例**，用来审查主 AI 的方案。**你不需要管**——ECD 自动调用它们，然后把结果展示给你。

可以理解为：ECD 请了另一个 AI 来"挑刺"，确保主 AI 的方案没有漏洞。

### Q13: 为什么 L3 需要 5 次独立审查？

因为 L3 任务（认证/支付/架构）的犯错成本太高。一个独立 AI 容易"当局者迷"——它自己写的方案自己看不出问题。所以 ECD 用另外的 AI 来攻击和审查。

### Q14: 增量模式是什么？怎么触发？

增量模式用于**已有 ECD bundle 的项目做后续小修改**。比如你用 ECD 做了一个功能，后来想加个小东西——增量模式跳过完整流程，只更新必要的部分。

触发方式：说"修改"、"增加"、"在上次ECD基础上..."等关键词。

### Q15: ECD 和普通 ChatGPT/Claude 对话有什么本质区别？

| | 普通对话 | ECD |
|---|---------|-----|
| 你说"做X" | AI 直接写代码 | ECD 先追问"X具体是什么？" |
| 中途改需求 | AI 尽量适应 | ECD 回流到规划阶段重新来 |
| 完成后 | AI 说"好了" | ECD 运行验证，给出通过/不通过裁决 |
| 方案质量 | 依赖主AI一个人的判断 | 有其他AI独立审查 |

### Q16: 我能在团队中使用 ECD 吗？

**能。** ECD 的 bundle（markdown 文件）可以作为团队沟通的"真相源"：
- 产品经理看审批包确认需求
- 开发者看交接包写代码
- 评审者看约束账本审查

### Q17: ECD 支持哪些编程语言/框架？

**所有。** ECD 是方法论，不绑定任何技术栈。它在 Claude Code 中运行，能处理任何 Claude 能处理的语言和框架。

不过，ECD 的 Web 应用质量门槛（路由冻结、组件职责、加载/空态/错误状态）主要针对前端项目优化。

### Q18: ECD 有 CLI 工具，我必须用吗？

**不必须。** CLI 工具（`ecd.py`）是可选的辅助工具，用于渲染和校验 bundle。ECD 的核心行为由 SKILL.md 驱动，不依赖 CLI。

### Q19: 我可以只学一部分 ECD 吗？

**可以。** 建议学习路径：
1. **新手**：只学 L1-Lite 流程（pre → code → achieve，跳过复杂阶段）
2. **进阶**：学习 L2-Standard（加上方案发散和可选审查）
3. **高级**：学习 L3-Full（全部10阶段+5次独立审查）

### Q20: ECD 会取代我作为开发者的工作吗？

**不会。** ECD 是一个方法论工具，它帮助你：
- 想清楚要做什么（而不是盲目开始）
- 避免遗漏安全问题（而不是上线后发现）
- 有据可查地交付（而不是"我记得当时是这么想的"）

决策权始终在你手里——ECD 提供方案，你审批。

## 术语通俗词典

ECD 有一些术语听起来很学术。这里是它们的通俗解释：

| 术语 | 通俗解释 | 类比 |
|------|---------|------|
| **演进约束开发 (ECD)** | 整套方法论的名字 | — |
| **演进约束语言 (ECL)** | ECD 用来记录过程的格式 | 施工日志的模板 |
| **语义冻结** | "这个功能到底做什么"已经定死了，不能再改 | 建筑图纸盖章确认后不能再改结构 |
| **约束求解** | 把需求看作"限制条件"，代码就是满足所有条件的方案 | 给你一堆要求（3室2厅、朝南、<100万），找到符合所有条件的房子 |
| **发散→收敛** | 先列出所有可能方案，再筛选出最好的一个 | 逛商场先全看一遍，最后买一件 |
| **审批包** | ECD 呈现给你的确认清单（目标+范围+假设+决策） | 合同签字前的最后一页摘要 |
| **交接包 (handoff)** | 给"写代码阶段"的详细说明书 | 给施工队的施工图纸 |
| **约束账本 (constraint ledger)** | 记录所有已确认的决策和限制 | 项目备忘录："确认用JWT"、"确认不用微服务" |
| **子Agent (subagent)** | ECD 调用的独立 AI 来审查主 AI 的工作 | 请第三方监理来检查施工方的方案 |
| **红蓝对抗 (Red/Blue)** | 一个 AI 攻击方案找漏洞，另一个 AI 防守修补漏洞 | 红队模拟黑客攻击，蓝队加固防线 |
| **bundle** | 存放所有 ECD 过程文件的文件夹 | 项目档案袋 |
| **code_ready** | 交接包的"就绪"标志——等于 True 才能开始写代码 | 图纸盖了"批准施工"章 |
| **achieve** | 验收裁决：通过(archived)/有遗留(achieved_with_followups)/不通过(not_achieved) | 监理验收：合格/有条件合格/返工 |
| **回流 (reflow)** | 发现后面阶段的问题，回到前面的阶段重做 | 施工发现图纸有误，回到设计阶段修改 |
| **分类器 (classifier)** | ECD 自动判断你的任务属于 L1/L2/L3 的机制 | 医院分诊台：感冒去内科、骨折去骨科 |
| **增量模式** | 对已有 ECD 项目做小修改，不需要重走全流程 | 房子已经盖好了，只换一个水龙头——不用重新画图纸 |
