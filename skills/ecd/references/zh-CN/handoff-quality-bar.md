# 交接质量门槛

在设置 `ecl.code_handoff.code_ready=true` 之前使用此参考。

## 核心规则

交接包仅在下一个编码模型可以在不发明高影响含义的情况下实现时，才是代码就绪的。

如果编码者仍需询问产品含义、数据应如何行为或如何判断成功，则交接包未就绪。

## 必须冻结的范围

交接包必须显式冻结：

- `repo_grounding`：计划所依赖的仓库事实
- `frozen_product_decisions`：不可漂移的高影响产品语义
- `domain_model`：关键实体、字段、状态和不变量
- `data_contract`：持久化或 API 行为，即使答案是"仅浏览器本地状态"
- `ui_contract`：路由、面板、表单、视图以及空/错误/加载状态
- `function_contracts`：编码者必须创建或修改的具体函数或模块
- `file_plan`：逐文件变更计划
- `implementation_units`：有序的执行单元
- `verification_commands`：命令级检查
- `browser_checks`：用户可见的走查检查
- `acceptance_checks`：必须满足什么才能称为工作完成

## 实现单元门槛

每个实现单元应回答：

- 改了什么
- 为什么存在
- 拥有哪些文件
- 创建或编辑哪些函数或模块
- 依赖哪些之前的单元
- 编码者在继续之前如何验证它
- 该单元的"完成"意味着什么

## Web 应用质量门槛

对于 React、Next.js 或 Vite 工作，额外冻结：

- 状态所有权
- 乐观或悲观的持久化行为
- 可见失败状态的文案
- 首次打开时必须工作的浏览器交互
- 单独测试是否足够，还是浏览器检查是强制性的
- 确切可见的 UI 模式，而非"分组或标签"这类析取描述

## 失败规则

如果以上任何一项留给"实现者决定"，`code_ready` 必须保持 `false`。
