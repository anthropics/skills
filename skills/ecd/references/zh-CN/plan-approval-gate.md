# 计划审批门

在 `ecd pre` 期间使用此参考。

## 目的

将原始用户想法转化为已批准的、冻结的计划目标，之后 B-H 再收敛完整 bundle。

## 规则

- 在信任请求之前先审计它。
- 在要求用户重述已经可发现的信息之前，先搜索本地仓库实际情况。
- 将用户的描述视为低可靠性证据：他们可能不确定、自相矛盾、偏向解决方案，或未意识到自己的隐藏需求。
- 不要以"少问问题"为优化目标。在 A 阶段趁用户交互仍被允许时，积极收集语义。
- 提出广泛但具体的澄清问题，涵盖目标、非目标、示例、反例、工作流、优先级、权衡、失败处理、数据语义、UI 状态和验收期望。
- 优先采用批量澄清方式，以揭示矛盾点和缺失语义，而非单个最小跟进。
- 仅当剩余未知项真正是低影响的实现细节而非潜在产品语义时，才停止提问。

## 审批包

在进入 B-H 之前，呈现一个简短的审批包，包含：

- `reframed_goal`：你现在认为用户实际想要的产品或变更
- `retained_scope`：本次将交付的内容
- `excluded_scope`：本次不会交付的内容
- `critical_assumptions`：实质性影响语义的假设
- `frozen_for_code`：`code` 将视为固定真相的决策

## 分级审批包 `[v1.2]`

审批包因复杂度等级而异。呈现与复杂度分类器确定的等级匹配的审批包。

**每个审批包必须以分类理由行开头** `[v1.2]`：

```
[L2] 判定依据: 6文件 + 功能逻辑风险 + 需求明确
```

格式：`[等级] 判定依据: Q1结果 + Q2结果 + Q3结果`，其中各 Q 结果为简短中文描述：
- Q1："≤3文件" / "4-10文件" / ">10文件"
- Q2："UI样式风险" / "功能逻辑风险" / "数据/安全风险"
- Q3："需求明确" / "部分待定" / "需求模糊→强制L3"

### Lite 审批包 (L1)

对于 L1 任务，在进入 J-Lite 之前呈现紧凑审批包：

- **分类理由行**（必需，见上文）
- `reframed_goal`：你现在认为用户实际想要的变更
- `retained_scope`：将交付的内容
- `excluded_scope`：不会交付的内容
- `acceptance_checks`：3-5 个具体、可验证的检查项
- `frozen_for_code`：code 将视为固定真相的决策

**规则：** 呈现此审批包之前最多 3 个澄清问题。如果需要更多，任务可能是 L2 或 L3。

### Standard 审批包 (L2)

对于 L2 任务，在 Lite 包基础上增加：

- **分类理由行**（必需，见上文）
- `critical_assumptions`：实质性影响语义的假设
- `key_non_goals`：明确排除在范围外的内容
- `risk_register`：前 3 个风险及缓解说明

### Full 审批包 (L3)

对于 L3 任务：

- **分类理由行**（必需，见上文）
- 与 v1.0 不变 — 完整的 5 字段审批包（reframed_goal、retained_scope、excluded_scope、critical_assumptions、frozen_for_code），具有完整的语义覆盖。对于 L3，还包括 Stage A 完整输出中的 scenario_fragments 和 blocking_unknowns。

## 退出规则

在用户明确批准审批包之前，不得继续进入 `ecd plan` 和 B-H。

如果用户回复有变更，更新审批包并再次请求批准。

用户批准后，不要将 B-H 或 J 变成逐阶段的访谈。后续阶段主要在后台收敛，仅当出现新的高影响歧义、矛盾或阻碍时才返回给用户。

## 个人项目进度规划器启发式

对于个人项目进度工具，确保审批包至少冻结：

- 什么算作项目、想法、计划项或捕获项
- 应用是否为单用户
- 持久化是仅本地还是服务器端
- 哪些时间范围是重要的，如长期、短期或灵感火花
- 仪表盘首次打开时必须展示什么
