# 阶段执行手册 v2.1

## 等级模型 (v1.1)

ECD v1.1 使用 3 问题复杂度分类器将任务路由到三个等级。本手册涵盖全部三个等级。**L3 (Full) 行为与 ECD v1.0 完全相同。**

### 等级选择摘要

| 等级 | 适用场景 | 阶段 | 子Agent | Token 目标 |
|------|---------|------|--------|-----------|
| **L1 Lite** | ≤3 文件，仅 UI 风险，需求清晰 | A-Lite → J-Lite → code → achieve-Lite | 0 | 15k-30k |
| **L2 Standard** | 4-10 文件，中等风险，基本清晰 | A → B → C → D(可选) → E(精简) → H(可选) → J → code → achieve | 0-2 | 35k-55k |
| **L3 Full** | >10 文件 或 高风险 或 需求模糊 | A → B → C → D → E → F → G → H → J → code → achieve | 5（全部强制） | 65k-105k |

### Lite 模板映射

当等级为 L1 时，使用存在的 `_lite` 模板变体：
- `stage-a-lite.md` 代替 `stage-a.md`
- `stage-j-lite.md` 代替 `stage-j.md`
- `code-handoff-lite.md` 代替 `code-handoff.md`
- `constraint-ledger-lite.md` 代替 `constraint-ledger.md`

### 微型任务快速通道 `[v1.2]`

当**全部**以下条件满足时，跳过 pre/plan 直接进入 code：

- 预估代码改动：**<10 行**
- 安全/正确性风险：**L1**（仅 UI 样式/文案/排版）
- 需求清晰度：**明确无歧义**

微型任务行为：
- 不生成 bundle、不启动子Agent、不渲染审批包
- 直接执行代码改动 → 快速验证 → 完成
- 预估 Token：**3k-8k**
- 如果对分类有任何疑问，回退到正常的 L1-Lite 流程

典型微型任务：修复拼写错误(T02)、调整字号(T06)、添加占位符(T15)、修改悬停效果(T17)

## 共享账本

`05-constraint-ledger.md` 是以下内容的单一真相源：

- 保留目标
- 已确认事实
- 被质疑的主张
- 冻结约束
- 已丢弃选项
- 风险
- 依赖链
- 验证语义
- 阶段引用

审批关闭后，用户交互通常应大幅减少。B-H 和 J 主要在后台收敛，除非出现新的高影响歧义或矛盾。

## A-Lite / 预处理 (Lite) `[L1]`

仅适用于 L1 任务。简化版 Stage A，冻结范围而不做深度语义挖掘。

- 将原始请求视为不可靠输入（与 Full A 相同）
- 先探查仓库和工作区实际情况再提问
- 提出**最多 3 个**澄清问题（vs Full A 无限制），最大化语义覆盖率
- 仅冻结 code 需要知道的内容：目标、范围边界、验收标准
- 跳过：发散(B)、深度拆解(C)、对抗审查(D/G/H)、伴侣文档(91-99)

必需输出：
- `user_stated_request`
- `reframed_request`
- `retained_scope`
- `excluded_scope`
- `acceptance_checks`（3-5 个具体检查）
- `key_assumptions`
- `frozen_for_code`

退出门：
- 重述后的目标和范围边界清晰到足以执行
- 用户已明确批准紧凑审批包
- 剩余未知项真正是低影响的实现细节

## A / 预处理 `[L2, L3]`

- 将原始请求视为不可靠输入
- 先探查仓库和工作区实际情况再向用户提问
- 提取可能的真实目标
- 假设用户可能尚未理解自己的真实目标、约束或期望结果
- 暴露歧义、错误假设、缺失事实、隐藏偏好、反目标以及未声明的约束
- 生成在审批前最大化语义覆盖率的澄清问题
- 跨越示例、反例、工作流、优先级、权衡、失败处理、边缘情况和验收含义进行审讯
- 在生成语义覆盖率饱和的审批就绪重述包后停止

必需输出：

- `user_stated_request`
- `ambiguity_points`
- `dubious_claims`
- `factual_gaps`
- `hidden_assumptions`
- `suspected_real_goals`
- `scenario_fragments`
- `success_signals`
- `non_goals`
- `follow_up_questions`
- `blocking_unknowns`
- `reframed_request`

退出门：

- 请求已围绕可能的真实目标重述
- 用户已被充分提问，隐藏的产品语义不太可能在后续阶段造成意外
- 审批包可以展示给用户

## B / 发散 `[L2, L3]`

- 生成材质不同的选项，而非风格变体
- 让每个选项说明覆盖了哪些盲区
- 仅保留一条路径

## C / 需求 `[L2, L3]`

- 将保留路径拆解为需求单元
- 冻结实现相关语义
- 塑造需求单元以便后续可干净映射到实现单元或任务批次
- 识别接口、验证目标和非目标
- 捕获需求到任务的可追溯性和可能的切割线，而不提前发明文件顺序
- 使包足够具体，让编码者不会发明产品含义

## D / 挑刺 `[L2, L3]`

**L2：** 可选。仅当 requirement_units > 3 或存在横切关注点时运行。
**L3：** 强制。完整独立挑刺，包含启动的子Agent。

- 启动一个独立挑刺Agent（L3强制，L2可选）
- 攻击模糊、矛盾或浪费的需求
- 移除伪需求和错误拆解

## E / 补全 `[L2, L3]`

**L2（精简）：** 仅识别和解决高影响依赖缺口。跳过执行链编译。
**L3（完整）：** 带执行顺序的完整依赖链。

- 完成端到端依赖链
- 将已解决的依赖转化为依赖感知的执行链，供后续阶段、批次和任务使用
- 移除隐藏前置条件
- 移除计划单元、批次和验证步骤之间的隐藏前置条件
- 不给 `code` 留下任何高影响依赖缺口

## F / 探测 `[L3]`

- 尽可能运行真实的可执行验证
- 优先选择仓库检查、脚本、测试、实验和环境检查
- 记录假设、方法、预期信号、终止标准和结果

## G / 红蓝 `[L3]`

- 启动独立的红方和蓝方 Agent
- 红方攻击边缘情况、滥用路径、依赖断裂和非法状态
- 蓝方缓解、约束或显式接受残余风险

## H / 评审 `[L2, L3]`

**L2：** 可选。仅当 handoff 有 >3 个实现单元时运行。
**L3：** 强制。完整独立评审，包含启动的子Agent。

- 启动一个独立评审Agent（L3强制，L2可选）
- 如果下一个编码者仍需发明以下内容，则拒绝该包：
  - 产品含义
  - 验证含义
  - 状态行为
  - 依赖行为
- 裁决必须为以下之一：
  - `approved`
  - `approved_with_conditions`
  - `rejected`

## J-Lite / 编译交接 (Lite) `[L1]`

仅适用于 L1 任务。将 A-Lite 输出直接编译为最小交接包。

- 直接吸收 A-Lite 输出（不存在 B-H 阶段）
- 编译最小 `90-code-handoff.md`，仅包含：
  - `repo_grounding`
  - `frozen_product_decisions`
  - `file_plan`
  - `implementation_units`（简化：每单元 goal + files + verification）
  - `verification_commands`
  - `acceptance_checks`
  - `code_ready`
- 跳过所有伴侣文档（91/92/95/96/98/99）
- 仅生成 `90-code-handoff.md` 和 `97-code-preflight.md`
- 仅当最小交接包对编码者足够时设置 `code_ready=true`

## J / 编译交接 `[L1, L2, L3]`

- 启动一个独立编译子Agent
- 将保留的 A-H 结果吸收到伴侣文档和最终代码就绪包中
- 按依赖顺序编译执行阶段、代码批次和实现单元
- 编译 `97-code-preflight.md` 作为用户和编码者在 `/code` 之前和期间维护的共享执行工作台
- 将评审条件和残余阻碍项编译为显式的 code-ready 或 scaffold 裁决
- 使包可以导出为 OpenSpec 风格的 `proposal/design/tasks` 而无需重新解释
- 保持 `90-code-handoff.md` 作为唯一真实的 `/code` 入口
- 生成伴侣文档 `91/92/95/96/97/98/99` 作为检查面，而非备用真相源

## 代码交接 `[L1, L2, L3]`

交接包必须冻结：

- 审批基础
- 仓库落地
- 产品语义
- 领域/数据/UI 契约
- 函数级契约
- 文件计划
- 实现单元
- 验证命令
- 浏览器检查
- 验收检查
- 重入触发器

仅当以上内容明确且未解决的缺口为空时，才允许 `code_ready=true`。

每个实现单元还必须冻结：

- 确切变更
- 任务就绪的完成信号
- 要添加或更新的测试

## `/code`

- 仅阅读 handoff 及其显式引用的文件
- 按单元顺序执行
- 边做边验证
- 如果语义仍然缺失，停止并重新进入 plan

## achieve-Lite / 关闭 (Lite) `[L1]`

仅适用于 L1 任务。基于最小交接包的轻量关闭。

- 运行交接包中的验证命令
- 检查首次打开体验没有明显损坏
- 裁决：`achieved` 或 `not_achieved`
- 归档状态：`archived` 或 `left_open`
- 写入 `Runs/<run-id>/03-achieve.md`
- 不得事后弱化验收标准
- 如果首次加载体验损坏，不得声称 achieved
