# RAG and Retrieval Systems (2025-2026)

## Vector Database Selection Guide

**Decision matrix by use case:**

| Situation | Recommendation | Why |
|---|---|---|
| Already on Postgres, <10M vectors | **pgvector** | HNSW indexes match dedicated DBs at this scale; vectors colocate with app data |
| Maximum filtered search speed | **Qdrant** | 1840 QPS on 1M-vector benchmarks; best-in-class filtered search; self-host or managed |
| Built-in hybrid search (keyword + vector) | **Weaviate** | Native BlockMax WAND + RSF; managed cloud available; good developer experience |
| Zero-ops managed, fastest time to production | **Pinecone** | Serverless tier, no infrastructure; pay for convenience; not the fastest |
| Multi-tenant SaaS product | **Weaviate** | Strong tenant isolation, per-tenant indexes |
| Chroma | Local prototyping only | No native hybrid search; don't use in production |

**Native hybrid search support (2025-2026)**: Weaviate, Milvus 2.5+, Qdrant v1.9+, LanceDB, Pinecone (proprietary sparse). ChromaDB and plain pgvector do NOT have native hybrid.

**Performance baseline**: With HNSW indexes, pgvector on equivalent compute matches or beats Qdrant on 1M vectors (per Supabase benchmarks). Pinecone Serverless trades latency for convenience.

## Embedding Models

**Current leaders (MTEB benchmark, 2025-2026)**:

| Model | Dimensions | Pricing | Best For |
|---|---|---|---|
| Cohere embed-v4 | 1024 | ~$0.10/1M tokens | Best MTEB, strong multilingual |
| Voyage AI voyage-3-large | 1024 | Competitive | Best RAG retrieval (14% over OpenAI on RTEB) |
| OpenAI text-embedding-3-large | 3072 | $0.13/1M tokens | Strong all-around, mature ecosystem |
| OpenAI text-embedding-3-small | 1536 | $0.02/1M tokens | Excellent value for most use cases |
| Jina embeddings-v3 | 768 | $0.02/1M tokens | Beats OpenAI large at OpenAI small pricing |
| BGE-M3 / E5-large | 1024 | Free (self-hosted) | Best open-source option |

**Matryoshka embeddings** (OpenAI, Cohere): Truncate to fewer dimensions with minor quality loss. Start with 768-1024 dimensions for most applications — higher dimensions give marginal improvements but increase storage costs proportionally.

**Multilingual**: Cohere Embed v3 trained on balanced multilingual data; 15-20% better than OpenAI on non-Latin scripts (Chinese, Arabic, Hindi).

**Practical recommendation**: Start with `text-embedding-3-small` (great value). Upgrade to Voyage voyage-3 or Cohere embed-v4 for production RAG where retrieval quality is critical.

## Chunking Strategies

**The chunking decision determines retrieval quality more than the vector DB choice.**

### Fixed-size chunking (simplest, often suboptimal)
Split by token count (e.g., 512 tokens with 10% overlap). Works for uniform documents. Poor at preserving semantic units.

### Semantic chunking (recommended for mixed content)
Split by semantic similarity breaks rather than fixed size: embed sentences, concatenate adjacent sentences with high similarity, split at similarity drops. Variable-length chunks that preserve meaning. Available in LlamaIndex and LangChain.

### Document-structure-aware chunking
Split by headings, paragraphs, code blocks. Preserve document hierarchy. Best when source documents have clear structure (PDFs with headers, Markdown, HTML).

### Parent-child chunking (best of both worlds)
Index small chunks (128 tokens) for precise retrieval. When retrieved, return the parent chunk (512+ tokens) for fuller context. LlamaIndex's `ParentDocumentRetriever` implements this natively.

**Two-stage approach for production**: Variable-length semantic chunking → parent-child retrieval. Semantic chunking for split points, parent retrieval for context. This pattern achieves ~99% retrieval accuracy on SQuAD when combined with reranking.

### Contextual Retrieval (Anthropic's approach)
Before embedding each chunk, prepend chunk-specific context generated by an LLM: "This chunk is from a Q4 2024 earnings report. It discusses the company's revenue growth..." This dramatically improves retrieval accuracy for decontextualized chunks. Combine with BM25 for best results. Compute-intensive at indexing time but retrieval quality improvement is significant.

## Hybrid Search (Production Standard)

**Hybrid search = vector similarity + BM25 keyword search.** Always use in production. The combination beats either approach alone.

Why: Vector search captures semantic meaning but can miss exact term matches. BM25 catches exact keywords but misses paraphrases. Combining them with Reciprocal Rank Fusion (RRF) achieves Recall@5 = 0.816, MRR@3 = 0.605 — significantly better than either alone.

**Implementation pattern**:
1. Run vector search → get top-K candidates
2. Run BM25 search → get top-K candidates
3. Merge and deduplicate
4. Apply RRF score fusion: `score = Σ 1/(rank + k)` for each result across retrieval methods
5. Rerank with a cross-encoder model

**Alpha tuning** (balance between dense and sparse):
- If BM25 alone is much worse than dense: users search with paraphrases → weight dense higher (alpha > 0.5)
- If BM25 alone is much better: users search with exact technical terms → weight sparse higher (alpha < 0.5)
- Start at 0.5 (equal weighting) and tune from there

## Reranking

Retrieval returns candidates; reranking determines final order. Add a cross-encoder reranker as the final step of any production RAG pipeline.

**Reranking models**:
- **Cohere Rerank**: Best managed API; `rerank-v3.5` strong across domains
- **Voyage AI Rerank**: Competitive quality, good price
- **BGE-reranker-v2-m3**: Best open-source; self-host with TGI or vLLM
- **FlashRank**: Lightweight, fast, good for latency-sensitive applications

**Two-stage pipeline**: Retrieve 50-100 candidates → rerank to top 5-10. Retrieval for recall, reranking for precision.

## Advanced RAG Patterns

### HyDE (Hypothetical Document Embeddings)
The query "What is quantum entanglement?" embeds differently from a document explaining it. HyDE: generate a hypothetical answer document using the LLM, embed the hypothetical document, use that embedding for retrieval. Bridges the semantic gap between short queries and long documents. +10-20% retrieval improvement on many benchmarks.

### RAPTOR (Recursive Abstractive Processing for Tree-Organized Retrieval)
Recursively cluster and summarize chunks into a hierarchical tree of increasing abstraction. At query time, retrieve from multiple tree levels simultaneously — fine-grained chunks AND high-level summaries. Enables multi-granularity retrieval for complex, multi-section questions. Best for long documents where both specific facts and high-level summaries matter.

RAPTOR + HyDE + reranking achieves ~99% retrieval accuracy on SQuAD.

### GraphRAG (Microsoft Open Source)
Build a knowledge graph from documents. Community detection creates cluster "reports" at multiple granularity levels. Enables multi-hop reasoning across entity relationships. Best for: competitive intelligence, regulatory analysis, multi-entity relationship queries. Implementation: Microsoft's `graphrag` library or LlamaIndex's `KnowledgeGraphIndex`. More expensive to build than vector RAG — worth it only when relationships between entities matter as much as the content itself.

### Self-RAG
The model generates both a retrieval decision (should I retrieve?) and a critique of retrieved documents (is this relevant? is my answer faithful?). Reduces unnecessary retrieval calls and hallucination. Implementation: use special tokens or structured outputs to capture retrieve/no-retrieve and critique signals. Best for applications where retrieval accuracy is critical and you can afford additional LLM calls.

### Corrective RAG (CRAG)
Retrieve → evaluate retrieved docs → if low quality, fall back to web search or broader retrieval. Adds a lightweight evaluator that assesses whether retrieved chunks actually answer the query before generating. Prevents confident-sounding hallucinations from irrelevant context.

### Agentic RAG
Instead of a fixed retrieve-then-generate pipeline, an agent decides when to retrieve, what to retrieve, and can iterate: retrieve → evaluate → refine query → retrieve again → synthesize. Combines a reasoning agent with RAG tools. Best for open-ended research tasks where the right retrieval strategy isn't known upfront.

## RAG Architecture Checklist

For a production RAG system, verify:
- [ ] Chunking preserves semantic units (semantic or structure-aware, not just fixed-size)
- [ ] Using parent-child retrieval (small chunks for retrieval, larger context returned)
- [ ] Hybrid search enabled (vector + BM25)
- [ ] Cross-encoder reranker in the pipeline
- [ ] Metadata filtering to scope retrieval (by document type, date, source, etc.)
- [ ] Evaluating with RAGAS metrics: faithfulness, context precision, context recall, answer relevancy
- [ ] Monitoring retrieval quality in production (not just answer quality)
