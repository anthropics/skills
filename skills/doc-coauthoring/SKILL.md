# 文档协作工作流程

此技能为引导用户完成协作文档创作提供结构化的工作流程。作为积极的指南，引导用户完成三个阶段：上下文收集、完善与结构，以及读者测试。

## 何时提供此工作流程

**触发条件：**
- 用户提到编写文档："写文档"、"起草提案"、"创建规范"、"撰写"
- 用户提到特定的文档类型："PRD"、"设计文档"、"决策文档"、"RFC"
- 用户似乎正在开始一项实质性的写作任务

**初始提议：**
为用户提供用于协作创作文档的结构化工作流程。解释三个阶段：

1. **上下文收集**：用户在 Claude 提出澄清问题时提供所有相关上下文
2. **完善与结构**：通过头脑风暴和编辑迭代构建每个部分
3. **读者测试**：使用全新的 Claude（无上下文）测试文档，以便在其他人在阅读之前发现盲点

解释这种方法有助于确保文档在他人在阅读时效果良好（包括当他们将其粘贴到 Claude 中时）。询问他们是否想尝试此工作流程或更喜欢自由形式工作。

如果用户拒绝，则自由形式工作。如果用户接受，请进入第 1 阶段。

## 第 1 阶段：上下文收集

**目标：** 缩小用户知道的内容和 Claude 知道的内容之间的差距，以便稍后提供智能指导。

### 初始问题

首先向用户询问有关文档的元上下文：

1. 这是什么类型的文档？（例如，技术规范、决策文档、提案）
2. 主要受众是谁？
3. 人们在阅读时期望产生什么影响？
4. 是否有要遵循的模板或特定格式？
5. 还有哪些其他约束或上下文需要了解？

告知他们可以用简写回答或以最适合他们的方式倾倒信息。

**如果用户提供模板或提到文档类型：**
- 询问他们是否有模板文档可以分享
- 如果他们提供共享文档的链接，请使用适当的集成来获取它
- 如果他们提供文件，请读取它

**如果用户提到编辑现有的共享文档：**
- 使用适当的集成来读取当前状态
- 检查没有 alt 文本的图像
- 如果存在没有 alt 文本的图像，解释当其他人使用 Claude 理解文档时，Claude 将无法看到它们。询问他们是否想要生成 alt 文本。如果是这样，请他们将每个图像粘贴到聊天中以进行描述性 alt 文本生成。

### 信息倾倒

一旦初始问题得到回答，鼓励用户倾倒他们拥有的所有上下文。请求信息，例如：
- 项目/问题的背景
- 相关的团队讨论或共享文档
- 为什么不使用替代解决方案
- 组织背景（团队动态、过去的事件、政治）
- 时间线压力或约束
- 技术架构或依赖关系
- 利益相关者的关注点

建议他们不要担心组织它——只需把它全部倾倒出来。提供多种提供上下文的方式：
- 信息倾倒意识流
- 指向团队频道或线程以阅读
- 链接到共享文档

**如果有可用的集成**（例如，Slack、Teams、Google Drive、SharePoint 或其他 MCP 服务器），提及这些可以直接用于拉取上下文。

**如果没有检测到集成并在 Claude.ai 或 Claude 应用程序中：** 建议他们可以在 Claude 设置中启用连接器，以允许直接从消息应用程序和文档存储中拉取上下文。

告知他们一旦完成初始倾倒，将提出澄清问题。

**在上下文收集期间：**

- 如果用户提到团队频道或共享文档：
  - 如果有集成：告知现在将读取内容，然后使用适当的集成
  - 如果没有集成：解释无法访问。建议他们在 Claude 设置中启用连接器，或直接粘贴相关内容

- 如果用户提到未知实体/项目：
  - 询问是否应该搜索连接的工具以了解更多信息
  - 在搜索之前等待用户确认

- 当用户提供上下文时，跟踪正在学习的内容以及仍然不清楚的内容

**提出澄清问题：**

当用户发出他们已完成初始倾倒的信号（或在提供大量上下文后），提出澄清问题以确保理解：

根据上下文中的差距生成 5-10 个编号的问题。

告知他们可以使用简写来回答（例如，"1：是，2：见 #channel，3：否因为向后兼容"），链接到更多文档，指向要阅读的频道，或者只是继续信息倾倒。这对他们来说最高效。

**退出条件：**
当问题显示出理解时，就已收集了足够的上下文——当可以询问边缘情况和权衡时，不需要解释基础知识。

**过渡：**
询问在此阶段他们是否想要提供更多上下文，或者是否是时候开始起草文档了。

如果用户想要添加更多内容，请让他们添加。准备好后，进入第 2 阶段。

## 第 2 阶段：完善与结构

**目标：** 通过头脑风暴、策展和迭代完善逐节构建文档。

**给用户的指令：**
解释文档将逐节构建。对于每个部分：
1. 将提出关于要包含什么的澄清问题
2. 将头脑风暴 5-20 个选项
3. 用户将指示要保留/删除/合并的内容
4. 将起草该部分
5. 将通过外科手术式的编辑进行完善

从具有最多未知数的部分开始（通常是核心决策/提案），然后完成其余部分。

**部分排序：**

如果文档结构清晰：
询问他们想从哪个部分开始。

建议从具有最多未知数的部分开始。对于决策文档，这通常是核心提案。对于规范，这通常是技术方法。摘要部分最好留到最后。

如果用户不知道他们需要哪些部分：
根据文档类型和模板，建议适合该文档类型的 3-5 个部分。

询问此结构是否有效，或者他们是否想要调整它。

**一旦结构达成一致：**

为所有部分创建初始文档结构和占位符文本。

**如果有权访问制品：**
使用 `create_file` 创建制品。这为 Claude 和用户都提供了一个可以工作的脚手架。

告知他们将为所有部分创建带有占位符的初始结构。

创建包含所有部分标题和简短占位符文本（如"[待写]"或"[内容在这里]"）的制品。

提供脚手架链接并指出现在是时候填充每个部分了。

**如果无法访问制品：**
在工作目录中创建一个 markdown 文件。适当命名（例如，`decision-doc.md`、`technical-spec.md`）。

告知他们将为所有部分创建带有占位符的初始结构。

创建包含所有部分标题和占位符文本的文件。

确认已创建文件名并指出现在是时候填充每个部分了。

**对于每个部分：**

### 步骤 1：澄清问题

宣布将开始处理 [部分名称] 部分。提出 5-10 个关于应包含内容的澄清问题：

根据上下文和部分目的生成 5-10 个具体问题。

告知他们可以使用简写回答或只是指出要涵盖的重要内容。

### 步骤 2：头脑风暴

对于 [部分名称] 部分，头脑风暴可能包含的 [5-20] 个项目，具体取决于部分的复杂性。寻找：
- 可能被遗忘的共享上下文
- 尚未提及的角度或考虑因素

根据部分复杂性生成 5-20 个编号选项。最后，提供如果他们想要更多选项则进行更多头脑风暴。

### 步骤 3：策展

询问哪些点应该保留、删除或合并。请求简要说明以帮助了解下一部分的优先级。

提供示例：
- "保留 1,4,7,9"
- "删除 3（重复 1）"
- "删除 6（受众已经知道这一点）"
- "合并 11 和 12"

**如果用户给出自由形式的反馈**（例如，"看起来不错"或"我大部分都喜欢，但是......"）而不是编号选择，请提取他们的偏好并继续。解析他们想要保留/删除/更改的内容并应用它。

### 步骤 4：差距检查

根据他们的选择，询问 [部分名称] 部分是否缺少任何重要的内容。

### 步骤 5：起草

使用 `str_replace` 将此部分的占位符文本替换为实际起草的内容。

宣布现在将根据他们选择的内容起草 [部分名称] 部分。

**如果使用制品：**
起草后，提供制品的链接。

请他们通读并指示要更改的内容。注意，具体说明有助于为下一部分学习。

**如果使用文件（无制品）：**
起草后，确认完成。

告知 [部分名称] 部分已在 [文件名] 中起草。请他们通读并指示要更改的内容。注意，具体说明有助于为下一部分学习。

**给用户的关键指令（在起草第一部分时包括）：**
提供说明：不要直接编辑文档，而是请他们指示要更改的内容。这有助于为未来的部分学习他们的风格。例如："删除 X 要点——已被 Y 覆盖"或"使第三段更简洁"。

### 步骤 6：迭代完善

当用户提供反馈时：
- 使用 `str_replace` 进行编辑（从不重新打印整个文档）
- **如果使用制品：** 每次编辑后提供制品链接
- **如果使用文件：** 只是确认编辑完成
- 如果用户直接编辑文档并要求阅读：在心理上记录他们所做的更改，并为未来的部分记住它们（这显示了他们的偏好）

**继续迭代**，直到用户对该部分满意。

### 质量检查

在 3 次连续迭代后没有实质性更改后，询问是否可以在不丢失重要信息的情况下删除任何内容。

当该部分完成时，确认 [部分名称] 已完成。询问是否准备好进入下一部分。

**对所有部分重复此过程。**

### 接近完成

当接近完成（完成 80% 以上的部分）时，宣布打算重新阅读整个文档并检查：
- 跨部分的流畅性和一致性
- 冗余或矛盾
- 任何感觉像"草率"或通用填充物的东西
- 每个句子是否都有分量

阅读整个文档并提供反馈。

**当所有部分都起草和完善后：**
宣布所有部分都已起草。表明打算再次审查完整文档。

审查整体连贯性、流畅性、完整性。

提供任何最终建议。

询问是否准备好进入读者测试，或者他们是否想要完善其他任何内容。

## 第 3 阶段：读者测试

**目标：** 使用全新的 Claude（无上下文泄漏）测试文档，以验证它对读者有效。

**给用户的指令：**
解释现在将进行测试，以查看文档是否真的对读者有效。这会捕获盲点——对作者有意义但可能会让其他人困惑的事情。

### 测试方法

**如果有权访问子代理（例如，在 Claude Code 中）：**

直接执行测试，无需用户参与。

### 步骤 1：预测读者问题

宣布打算预测读者在尝试发现此文档时可能提出的问题。

生成 5-10 个读者实际上会问的问题。

### 步骤 2：使用子代理测试

宣布这些问题将在全新的 Claude 实例（没有此对话的上下文）中进行测试。

对于每个问题，调用仅包含文档内容和问题的子代理。

总结读者 Claude 对每个问题对/错的内容。

### 步骤 3：运行额外检查

宣布将执行额外检查。

调用子代理以检查歧义、错误假设、矛盾。

总结发现的任何问题。

### 步骤 4：报告和修复

如果发现问题：
报告读者 Claude 在特定问题上遇到困难。

列出具体问题。

表明打算修复这些差距。

循环回到有问题的部分进行完善。

---

**如果无法访问子代理（例如，claude.ai Web 界面）：**

用户需要手动进行测试。

### 步骤 1：预测读者问题

询问人们在尝试发现此文档时可能会问什么问题。他们会在 Claude.ai 中输入什么？

生成 5-10 个读者实际上会问的问题。

### 步骤 2：设置测试

提供测试说明：
1. 打开一个新的 Claude 对话：https://claude.ai
2. 粘贴或分享文档内容（如果使用启用了连接器的共享文档平台，则提供链接）
3. 向读者 Claude 提出生成的问题

对于每个问题，指示读者 Claude 提供：
- 答案
- 是否有任何歧义或不清楚
- 文档假设读者已经知道的知识/上下文

检查读者 Claude 是否给出正确答案或误解任何内容。

### 步骤 3：额外检查

还要询问读者 Claude：
- "本文档中可能对读者有什么不清楚或不清楚的地方？"
- "本文档假设读者已经具备什么知识或上下文？"
- "是否有任何内部矛盾或不一致？"

### 步骤 4：根据结果迭代

询问读者 Claude 做错了什么或在什么方面遇到困难。表明打算修复这些差距。

循环回到任何有问题的部分进行完善。

---

### 退出条件（两种方法）

当读者 Claude 始终正确回答问题并且没有提出新的差距或歧义时，文档就准备好了。

## 最终审查

当读者测试通过时：
宣布文档已通过读者 Claude 测试。完成之前：

1. 建议他们自己进行最后的通读——他们拥有此文档并对其质量负责
2. 建议仔细检查任何事实、链接或技术细节
3. 请他们验证它是否实现了他们想要的影响

询问他们是否想要再进行一次审查，或者工作是否完成。

**如果用户想要最终审查，请提供。否则：**
宣布文档完成。提供一些最终提示：
- 考虑在附录中链接此对话，以便读者可以看到文档是如何开发的
- 使用附录在不膨胀主文档的情况下提供深度
- 根据真实读者的反馈更新文档

## 有效指导的技巧

**语气：**
- 直接和程序化
- 简要解释理由，当它影响用户行为时
- 不要试图"推销"这种方法——只需执行它

**处理偏差：**
- 如果用户想要跳过一个阶段：询问他们是否想要跳过此阶段并自由形式写作
- 如果用户感到沮丧：承认这比预期的时间长。建议更快的方法
- 始终给予用户调整流程的权利

**上下文管理：**
- 在整个过程中，如果缺少关于所提及内容的上下文，请主动询问
- 不要让差距累积——在它们出现时解决它们

**制品管理：**
- 使用 `create_file` 起草完整部分
- 使用 `str_replace` 进行所有编辑
- 每次更改后提供制品链接
- 永远不要将制品用于头脑风暴列表——那只是对话

**质量优于速度：**
- 不要匆忙完成阶段
- 每次迭代都应该进行有意义的改进
- 目标是真正为读者工作的文档
