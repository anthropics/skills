## 知识库的定义

### 什么是知识库

知识库（Knowledge Base, KB）是一个用于存储和管理知识的系统，它能将非结构化、半结构化和结构化数据转化为 AI 系统可检索、可利用的知识。

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        知识库的本质                                           │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   传统视角:                                                                 │
│   ┌─────────────────────────────────────────────────────────────────────┐   │
│   │   知识库 = 结构化的信息存储系统                                      │   │
│   │   例: FAQ 数据库、维基百科、企业文档库                                │   │
│   └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│   AI 时代视角:                                                              │
│   ┌─────────────────────────────────────────────────────────────────────┐   │
│   │   知识库 = LLM 的"外部大脑"                                         │   │
│   │                                                                     │   │
│   │   ┌──────────┐                     ┌──────────────────────────┐     │   │
│   │   │   LLM    │  ← 检索相关知识 ←  │      知识库               │     │   │
│   │   │          │                     │                          │     │   │
│   │   │ 参数知识  │                     │  • 公司内部文档           │     │   │
│   │   │ (训练时   │                     │  • 最新技术资料           │     │   │
│   │   │  学到的)  │                     │  • 实时数据              │     │   │
│   │   │          │                     │  • 领域专业知识           │     │   │
│   │   └──────────┘                     └──────────────────────────┘     │   │
│   │                                                                     │   │
│   │   核心价值:                                                          │   │
│   │   • 补充 LLM 训练截止日期后的信息                                    │   │
│   │   • 提供 LLM 未见过的私有/领域知识                                   │   │
│   │   • 为回答提供可验证的事实依据                                       │   │
│   │   • 降低幻觉，提高准确性                                            │   │
│   └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 知识库的分类

### 按数据结构分类

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                       知识库分类体系                                          │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   1. 向量知识库 (Vector Knowledge Base) ⭐ 最常用                           │
│   ┌─────────────────────────────────────────────────────────────────────┐   │
│   │   原理: 文本 → Embedding 向量 → 向量数据库                          │   │
│   │   检索: 语义相似度搜索 (余弦相似度/内积)                             │   │
│   │   技术: Milvus, Pinecone, Qdrant, Chroma                           │   │
│   │   优势: 语义理解能力强，支持模糊检索                                 │   │
│   │   劣势: 精确匹配较弱，数值/日期等结构化查询不佳                      │   │
│   │                                                                     │   │
│   │   示例: "如何优化显存？" → 找到"GPU 内存管理技巧"相关文档            │   │
│   └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│   2. 图谱知识库 (Knowledge Graph)                                           │
│   ┌─────────────────────────────────────────────────────────────────────┐   │
│   │   原理: 实体 - 关系 - 实体 (三元组)                                  │   │
│   │   检索: 图遍历、SPARQL 查询、子图匹配                                │   │
│   │   技术: Neo4j, ArangoDB, Amazon Neptune                             │   │
│   │   优势: 多跳推理能力强，关系清晰                                     │   │
│   │   劣势: 构建成本高，需要实体抽取和关系标注                           │   │
│   │                                                                     │   │
│   │   示例: (NVIDIA) -[开发]→ (CUDA) -[用于]→ (GPU计算)                 │   │
│   └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│   3. 全文知识库 (Full-text Search)                                          │
│   ┌─────────────────────────────────────────────────────────────────────┐   │
│   │   原理: 倒排索引 + BM25 评分                                        │   │
│   │   检索: 关键词匹配、布尔查询                                        │   │
│   │   技术: Elasticsearch, OpenSearch, MeiliSearch                      │   │
│   │   优势: 精确匹配强，成熟稳定                                        │   │
│   │   劣势: 缺乏语义理解，"显存优化"搜不到"GPU 内存管理"               │   │
│   └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│   4. 混合知识库 (Hybrid) ⭐ 生产推荐                                       │
│   ┌─────────────────────────────────────────────────────────────────────┐   │
│   │   原理: 多种检索方式组合                                             │   │
│   │   检索: 向量搜索 + 关键词搜索 + 图谱查询 → 融合排序                 │   │
│   │   优势: 取长补短，覆盖更全面                                        │   │
│   │   劣势: 架构复杂度高                                                │   │
│   │                                                                     │   │
│   │   典型组合:                                                          │   │
│   │   • Milvus (语义) + Elasticsearch (关键词) → RRF 融合               │   │
│   │   • Qdrant (语义) + Neo4j (关系推理) → 多路召回                     │   │
│   └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 按知识来源分类

| 类别 | 描述 | 示例 |
|------|------|------|
| **企业知识库** | 公司内部文档、制度、流程 | HR 手册、产品文档、运维手册 |
| **领域知识库** | 特定行业的专业知识 | 医学文献、法律条文、金融报告 |
| **通用知识库** | 广泛的通用知识 | 维基百科、百科全书 |
| **实时知识库** | 需要频繁更新的时效性数据 | 股票价格、新闻、天气 |
| **代码知识库** | 代码仓库和技术文档 | GitHub 仓库、API 文档 |

## 知识库 vs 传统数据库

### 核心差异对比

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                     知识库 vs 传统数据库                                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   维度            传统数据库 (RDBMS)          AI 知识库                     │
│   ────            ─────────────────          ──────────                     │
│                                                                             │
│   查询方式        精确查询 (SQL)              语义查询 (自然语言)            │
│                   SELECT * FROM users         "找到关于显存优化的文档"       │
│                   WHERE age > 25              → 语义向量搜索                 │
│                                                                             │
│   数据形式        结构化 (行列表格)           非结构化 + 向量               │
│                   行、列、关系                文本、Embedding、元数据        │
│                                                                             │
│   索引方式        B-Tree / Hash               HNSW / IVF / PQ              │
│                                                                             │
│   核心目标        ACID 事务一致性             高召回率 + 低延迟检索         │
│                                                                             │
│   相似性          精确匹配                    近似最近邻 (ANN)              │
│                   WHERE name = 'Alice'        cosine_similarity > 0.8       │
│                                                                             │
│   典型操作        CRUD                        索引 + 搜索 + 更新            │
│                   INSERT/UPDATE/DELETE/SELECT  Embed → Index → Search        │
│                                                                             │
│   典型延迟        < 1ms                       10-100ms                      │
│                                                                             │
│   一致性          强一致性 (ACID)             最终一致性                     │
│                                                                             │
│   代表产品        MySQL, PostgreSQL           Milvus, Pinecone, Qdrant      │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 互补关系

```python
# 知识库与传统数据库互补使用
# 场景: 企业 AI 助手

# 用传统数据库存储结构化数据
import psycopg2
conn = psycopg2.connect("postgresql://localhost/company")
cursor = conn.cursor()
cursor.execute("SELECT name, department FROM employees WHERE id = %s", (user_id,))
user_info = cursor.fetchone()

# 用知识库存储非结构化文档
from langchain_community.vectorstores import Milvus
knowledge = vectorstore.similarity_search("报销流程是什么？", k=3)

# 组合使用
context = f"用户: {user_info[0]}，部门: {user_info[1]}\n相关文档: {knowledge}"
answer = llm.invoke(f"基于以下信息回答问题：\n{context}\n\n问题: 我的报销流程是什么？")
```

### 企业客服示例

```python
# 企业客服 RAG 示例
from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain_community.vectorstores import Chroma
from langchain.chains import ConversationalRetrievalChain
from langchain.memory import ConversationBufferWindowMemory

# 加载产品知识库
vectorstore = Chroma(
    persist_directory="./product_kb",
    embedding_function=OpenAIEmbeddings()
)

# 构建对话式 RAG
memory = ConversationBufferWindowMemory(
    memory_key="chat_history",
    return_messages=True,
    k=5  # 保留最近 5 轮对话
)

qa_chain = ConversationalRetrievalChain.from_llm(
    llm=ChatOpenAI(model="gpt-4o-mini", temperature=0),
    retriever=vectorstore.as_retriever(search_kwargs={"k": 3}),
    memory=memory,
    return_source_documents=True
)

# 多轮对话
result = qa_chain.invoke({"question": "你们的退款政策是什么？"})
print(result["answer"])
# 基于知识库中的退款政策文档，准确回答
```

## 实战练习

### 练习 1：知识库需求分析

根据以下场景，分析需要什么类型的知识库：

```markdown
场景: 某电商公司想构建 AI 客服，需要：
1. 回答产品相关问题（产品手册、规格参数）
2. 处理退换货查询（公司政策文档）
3. 查询订单状态（数据库实时数据）
4. 回答物流跟踪（第三方 API 实时数据）

分析:
- 向量知识库: 产品手册 + 政策文档（语义检索）
- 关系数据库: 订单状态（精确查询）
- API 调用: 物流跟踪（实时数据）
- 推荐方案: 混合知识库 + Function Calling
```

### 练习 2：评估现有知识库

```python
# kb_assessment.py
def assess_knowledge_base_needs(requirements: dict) -> dict:
    """评估知识库需求"""
    
    recommendations = []
    
    # 是否需要语义搜索
    if requirements.get("semantic_search", False):
        recommendations.append({
            "component": "向量数据库",
            "options": ["Milvus", "Qdrant", "Chroma"],
            "reason": "需要语义理解能力"
        })
    
    # 是否需要精确匹配
    if requirements.get("exact_match", False):
        recommendations.append({
            "component": "全文搜索",
            "options": ["Elasticsearch", "OpenSearch"],
            "reason": "需要精确关键词匹配"
        })
    
    # 是否需要关系推理
    if requirements.get("relationship_reasoning", False):
        recommendations.append({
            "component": "图数据库",
            "options": ["Neo4j"],
            "reason": "需要实体关系推理"
        })
    
    # 是否需要实时数据
    if requirements.get("realtime_data", False):
        recommendations.append({
            "component": "API 集成",
            "options": ["Function Calling", "Tool Use"],
            "reason": "需要实时数据接入"
        })
    
    return {
        "recommendations": recommendations,
        "architecture": "混合知识库" if len(recommendations) > 1 else "单一知识库"
    }

# 使用
needs = assess_knowledge_base_needs({
    "semantic_search": True,
    "exact_match": True,
```
