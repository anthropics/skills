# 训练 vs 推理：大模型的"上学"与"考试"

> 训练和推理是大模型生命周期的两个阶段。训练是"学习知识"，推理是"运用知识"。它们在计算模式、显存占用、性能瓶颈、并行策略等方面有本质区别。

## 一句话区分

```
┌─────────────────────────────────────────────────────────────────────────┐
│                                                                          │
│   训练 (Training)  = 学生上学                                           │
│   ─────────────────────────────                                         │
│   看课本 → 做练习题 → 对答案 → 发现错误 → 调整学习方法 → 重复...       │
│   目标: 学会知识 (调整模型参数)                                          │
│   特点: 耗时长、算力大、但只需要做一次                                   │
│                                                                          │
│   推理 (Inference) = 学生考试                                           │
│   ─────────────────────────────                                         │
│   看到题目 → 用已学知识作答 → 交卷                                      │
│   目标: 快速给出答案 (用固定参数做预测)                                  │
│   特点: 要快、要省、要不断重复 (服务千万用户)                            │
│                                                                          │
│   关键区别:                                                              │
│   训练需要"纠错" (反向传播) → 推理不需要                               │
│   训练做一次就好    → 推理要做千万次 (每个用户请求都是一次推理)         │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘
```

## 全景对比图

```
═══════════════════════════════════════════════════════════════════════════
  训练流程 (Training)
═══════════════════════════════════════════════════════════════════════════

  ┌────────┐    ┌───────────┐    ┌──────────┐    ┌───────────┐    ┌──────────┐
  │ 训练数据 │───→│ 前向传播   │───→│ 算 Loss  │───→│ 反向传播   │───→│ 更新参数  │
  │ (TB级)  │    │ Y = f(X)  │    │ L(Y, Ŷ) │    │ ∂L/∂W    │    │ W -= lr·g│
  └────────┘    └───────────┘    └──────────┘    └───────────┘    └──────────┘
       │                                                                │
       └────────────────────── 循环几万到几十万步 ──────────────────────┘

  输入: 海量训练语料 (几 TB 文本)
  输出: 一组训练好的参数 (如 LLaMA-7B 的 70 亿个数字)
  时长: 几天到几个月
  硬件: 几十到几千张 GPU

═══════════════════════════════════════════════════════════════════════════
  推理流程 (Inference)
═══════════════════════════════════════════════════════════════════════════

  ┌──────────┐    ┌───────────┐    ┌──────────┐
  │ 用户输入  │───→│ 前向传播   │───→│ 输出结果  │
  │ "你好"   │    │ (参数固定) │    │ "你好！"  │
  └──────────┘    └───────────┘    └──────────┘
                       ↑
                  参数不变！
                  没有 Loss、没有反向传播、没有参数更新

  输入: 一条用户请求
  输出: 模型的回答 (逐 token 生成)
  时长: 几百毫秒到几秒
  硬件: 1-8 张 GPU (通常)
```

## 计算模式差异

### 训练：前向 + 反向，完整走一圈

```
训练一个 step 的计算流程:

  ┌─────────────────────────────────────────────────────────────────────┐
  │                        训练 (Training)                              │
  │                                                                     │
  │   前向传播 (Forward)                                                │
  │   ─────────────────                                                 │
  │   输入 X → 经过 32 层 Transformer → 输出 logits → 算 Loss         │
  │                                                                     │
  │   要点:                                                             │
  │   • 一次处理一整个 batch (如 4 条 × 2048 tokens = 8192 tokens)     │
  │   • 所有 token 同时并行计算 (Teacher Forcing)                      │
  │   • 必须保存每层的中间激活值 (后面反向传播要用!)                    │
  │                                                                     │
  │   反向传播 (Backward) ← 推理没有这一步!                            │
  │   ──────────────────                                                │
  │   Loss → 反向经过 32 层 → 算出每个参数的梯度 ∂L/∂W                │
  │                                                                     │
  │   要点:                                                             │
  │   • 计算量约等于前向传播的 2 倍                                     │
  │   • 需要用到前向传播保存的激活值                                     │
  │   • 一个 step 总计算量 ≈ 3 × 前向传播                              │
  │                                                                     │
  │   参数更新 (Update) ← 推理也没有这一步!                            │
  │   ──────────────────                                                │
  │   W = W - lr × optimizer(gradient)                                  │
  │                                                                     │
  └─────────────────────────────────────────────────────────────────────┘
```

### 推理：只有前向，但要一个一个生成

```
推理的计算流程 (以 LLM 为例):

  ┌─────────────────────────────────────────────────────────────────────┐
  │                        推理 (Inference)                             │
  │                                                                     │
  │   用户输入: "中国的首都是"  (4 个 token)                            │
  │                                                                     │
  │   Step 1: "中国的首都是" → 模型 → "北" (生成第1个token)           │
  │   Step 2: "中国的首都是北" → 模型 → "京" (生成第2个token)         │
  │   Step 3: "中国的首都是北京" → 模型 → "。" (生成第3个token)       │
  │   ...逐 token 生成，直到遇到结束符                                  │
  │                                                                     │
  │   要点:                                                             │
  │   • 自回归 (Autoregressive): 每次只生成 1 个 token                 │
  │   • 每生成一个 token 都要跑一次完整的前向传播                       │
  │   • 不需要保存激活值 (没有反向传播!)                                │
  │   • 不需要算梯度 (参数是固定的!)                                    │
  │   • 用 KV Cache 避免重复计算 (详见第10篇)                          │
  │                                                                     │
  └─────────────────────────────────────────────────────────────────────┘
```

### 训练的 Teacher Forcing vs 推理的自回归

```
训练时 (Teacher Forcing — 并行!):
═══════════════════════════════════════

  输入:  "中国 的 首都 是 北京"  (5 个 token 同时喂入)
  标签:  "的 首都 是 北京 。"    (每个位置的"正确答案")

  ┌──────┬──────┬──────┬──────┬──────┐
  │ 中国  │  的   │ 首都  │  是   │ 北京  │  ← 5 个 token 同时输入
  └──┬───┘└──┬───┘└──┬───┘└──┬───┘└──┬───┘
     │       │       │       │       │
     ▼       ▼       ▼       ▼       ▼       ← 5 个位置同时算 (并行!)
  ┌──────┬──────┬──────┬──────┬──────┐
  │预测:的│预测:首│预测:是│预测:北│预测:。│
  └──────┘└──────┘└──────┘└──────┘└──────┘
     vs      vs      vs      vs      vs     ← 和标签比较
  ┌──────┬──────┬──────┬──────┬──────┐
  │标签:的│标签:首│标签:是│标签:北│标签:。│
  └──────┘└──────┘└──────┘└──────┘└──────┘

  → 一次前向传播就处理完所有 token!
  → GPU 利用率极高 (大矩阵运算)

推理时 (自回归 — 串行!):
═══════════════════════════════════════

  Step 1: 输入 "中国的首都是" → 预测下一个 → "北"
  Step 2: 输入 "中国的首都是北" → 预测下一个 → "京"
  Step 3: 输入 "中国的首都是北京" → 预测下一个 → "。"

  → 每步只输出 1 个 token!
  → 必须等上一步完成才能做下一步 (串行依赖)
  → 每步的矩阵运算很"瘦": [1, hidden] × [hidden, vocab]
  → GPU 大量算力闲置 (吃不饱!)

这就是训练和推理在计算模式上最根本的区别:
训练 = 大块数据并行处理 → 充分利用 GPU
推理 = 逐 token 串行生成 → GPU 利用率低
```

## 显存占用差异

这是训练和推理最直观的区别之一。以 LLaMA-7B (FP16) 为例：

### 训练时的显存占用

```
LLaMA-7B 训练显存拆解 (FP16 混合精度训练):

  ┌─────────────────────────────────────────────────────────────────────┐
  │                                                                      │
  │  ① 模型参数 (FP16)                                                  │
  │     7B × 2 bytes = 14 GB                                            │
  │                                                                      │
  │  ② 梯度 (FP16)                                                      │
  │     和参数一一对应，每个参数有一个梯度                                │
  │     7B × 2 bytes = 14 GB                                            │
  │                                                                      │
  │  ③ 优化器状态 (FP32) — 这是大头!                                    │
  │     AdamW 为每个参数维护 2 个状态:                                   │
  │     • 一阶动量 m (梯度的滑动平均):  7B × 4 bytes = 28 GB           │
  │     • 二阶动量 v (梯度²的滑动平均): 7B × 4 bytes = 28 GB           │
  │     • FP32 参数主副本:              7B × 4 bytes = 28 GB            │
  │     小计: 84 GB                                                      │
  │                                                                      │
  │  ④ 激活值 (中间结果，反向传播用)                                     │
  │     取决于 batch_size 和 seq_len                                     │
  │     batch=4, seq=2048 时约 8-20 GB (视是否用 activation checkpointing)│
  │                                                                      │
  │  ⑤ 临时缓冲区                                                       │
  │     梯度通信、AllReduce 缓冲等: 约 2-4 GB                           │
  │                                                                      │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                      │
  │  总计: 14 + 14 + 84 + ~15 + ~3 ≈ 130 GB                           │
  │                                                                      │
  │  一张 A100 80GB 都装不下!                                            │
  │  → 这就是为什么训练需要分布式 (ZeRO、模型并行等)                    │
  │                                                                      │
  └─────────────────────────────────────────────────────────────────────┘
```

### 推理时的显存占用

```
LLaMA-7B 推理显存拆解 (FP16):

  ┌─────────────────────────────────────────────────────────────────────┐
  │                                                                      │
  │  ① 模型参数 (FP16)                                                  │
  │     7B × 2 bytes = 14 GB                                            │
  │                                                                      │
  │  ② KV Cache (随请求动态增长)                                        │
  │     每个用户会话需要缓存所有层的 K 和 V:                             │
  │     大小 = 2(K和V) × 层数 × seq_len × hidden_dim × dtype            │
  │     单个用户 seq=2048:                                               │
  │       2 × 32 × 2048 × 4096 × 2 bytes = 1 GB                        │
  │     同时服务 32 个用户:                                              │
  │       32 × 1 GB = 32 GB                                             │
  │                                                                      │
  │  ③ 不需要的东西:                                                     │
  │     ✗ 梯度          — 0 GB (不算梯度!)                              │
  │     ✗ 优化器状态    — 0 GB (不更新参数!)                             │
  │     ✗ 激活值保存    — 0 GB (不做反向传播!)                           │
  │                                                                      │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                      │
  │  单用户: 14 + 1 = 15 GB (一张 RTX 4090 24GB 就能跑!)               │
  │  32 用户: 14 + 32 = 46 GB (一张 A100 80GB 绰绰有余)                │
  │                                                                      │
  └─────────────────────────────────────────────────────────────────────┘
```

### 显存对比一览

```
  训练 vs 推理显存对比 (LLaMA-7B FP16):

  训练: ████████████████████████████████████████████████  ~130 GB
        [参数14][梯度14][  优化器状态 84  ][激活~15][buf]

  推理: █████████████████                                 ~15 GB (单用户)
        [参数14][KV 1]

        [参数14][   KV Cache 32    ]

  核心原因:
  ┌──────────────┬──────────┬──────────┐
  │     组成      │   训练   │   推理   │
  ├──────────────┼──────────┼──────────┤
  │ 模型参数      │  14 GB   │  14 GB   │ ← 一样
  │ 梯度          │  14 GB   │   0 GB   │ ← 推理不需要

  ├──────────────┼──────────┼──────────┤
  │ 总计          │ ~130 GB  │ 15-46 GB │
  └──────────────┴──────────┴──────────┘
```

## 计算瓶颈差异

训练和推理的性能瓶颈完全不同，这决定了优化方向的根本差异。

### 训练：Compute-Bound（计算瓶颈）

```
训练为什么是 compute-bound？

═══════════════════════════════════════════════════════════════════════════

  训练时的矩阵运算:
  X [batch × seq, hidden] × W [hidden, 4×hidden]
  = [4×2048, 4096] × [4096, 16384]

  这是一个巨大的矩阵乘法!
  • 计算量: 2 × 8192 × 4096 × 16384 ≈ 1.1 TFLOPs (单层单个 Linear)
  • 32 层 × 多个 Linear × (前向 + 反向) → 每个 step 几十 TFLOPs

  GPU 的特性:
  ┌─────────────────────────────────────────────┐
  │  A100 GPU 算力:     312 TFLOPS (FP16)       │
  │  A100 显存带宽:    2,039 GB/s               │
  │                                              │
  │  算力/带宽 = 312T / 2039G ≈ 153             │
  │  → 每读取 1 byte，能做 153 次 FP16 运算     │
  └─────────────────────────────────────────────┘

  训练时的矩阵很"胖" (行列都很大):
  → 每次读一块数据可以做大量计算
  → 计算量 >> 数据读取量
  → GPU 的 Tensor Core 是瓶颈 → compute-bound!

  优化方向: 用更多 GPU、用更高算力的芯片 (H100→B200)
  
═══════════════════════════════════════════════════════════════════════════
```

### 推理（Decode 阶段）：Memory-Bound（带宽瓶颈）

```
推理 (Decode) 为什么是 memory-bound？

═══════════════════════════════════════════════════════════════════════════

  Decode 阶段每步只生成 1 个 token:
  Q [1, hidden] × K^T [hidden, seq_len]
  = [1, 4096] × [4096, 2048]

  这是一个很"瘦"的矩阵乘法!
  • 计算量: 2 × 1 × 4096 × 2048 ≈ 16M FLOPs (很少!)
  • 但需要读取: W [4096 × 4096] = 32MB 权重数据 (每层每个 Linear)

  问题:
  ┌─────────────────────────────────────────────┐
  │  计算量很小:  几十 M FLOPs                   │
  │  但要读的数据: 几百 MB (所有层的权重+KV)     │
  │                                              │
  │  A100 带宽: 2039 GB/s                        │
  │  读 14GB 参数需要: 14/2039 ≈ 6.9 ms          │
  │  但计算这些数据只需要: ~0.5 ms               │
  │                                              │
  │  → GPU 大部分时间在"等数据从显存搬过来"!     │
  │  → 计算单元严重空闲 → memory-bound!          │
  └─────────────────────────────────────────────┘

  生活类比:
  训练 = 工厂流水线: 原材料充足，机器满负荷运转     → 计算是瓶颈
  推理 = 快递小哥: 大部分时间在路上跑，真正送货很快  → "搬运"是瓶颈

  优化方向: 减少数据搬运 (量化减小模型体积)、提高带宽 (HBM3)
  
═══════════════════════════════════════════════════════════════════════════
```

### Arithmetic Intensity 直观对比

```
Arithmetic Intensity (算术强度) = 计算量 / 数据访问量

  ┌─────────────────────────────────────────────────────────────────┐
  │                                                                  │
  │  训练 (大矩阵乘法):                                             │
  │  [8192 × 4096] × [4096 × 16384]                                │
  │  计算量 = 2 × 8192 × 4096 × 16384 = 1.1T FLOPs                │
  │  数据量 = (8192×4096 + 4096×16384) × 2 bytes = 196 MB          │
  │  算术强度 = 1.1T / 196M ≈ 5600                                 │
  │  → 远超 A100 的拐点 (153) → Compute-bound ✓                    │
  │                                                                  │
  │  推理 Decode (瘦矩阵乘法):                                      │
  │  [1 × 4096] × [4096 × 4096]                                    │
  │  计算量 = 2 × 1 × 4096 × 4096 = 33.6M FLOPs                   │
  │  数据量 = (1×4096 + 4096×4096) × 2 bytes = 33.6 MB             │
  │  算术强度 = 33.6M / 33.6M = 1                                  │
  │  → 远低于 A100 拐点 (153) → Memory-bound ✓                     │
  │                                                                  │
  └─────────────────────────────────────────────────────────────────┘

  在 Roofline 模型上:

  性能   _____________________.............  ← 算力上限 (312 TFLOPS)
  (FLOPS) /                    .
         /                     .
        /  memory-bound 区域    . compute-bound 区域
       / ← 带宽决定性能        . ← 算力决定性能
      /                        .
     /                         .
    ──────────────────────────────────── 算术强度
         ↑推理Decode           ↑训练
         (强度≈1)             (强度≈5600)
```

## 并行策略差异

### 训练的并行策略

```
训练时的"三维并行" (3D Parallelism):

═══════════════════════════════════════════════════════════════════════════

  ┌──────────────────────────────────────────────────────────────────┐
  │  数据并行 (Data Parallel, DP)                                    │
  │  ─────────────────────────────                                   │
  │  每张 GPU 拿不同的 batch，算完梯度后 AllReduce 同步              │
  │  目的: 加速训练 (处理更多数据)                                    │
  │                                                                   │
  │  GPU 0: batch 0 ──→ grad 0 ─┐                                   │
  │  GPU 1: batch 1 ──→ grad 1 ─┼── AllReduce ──→ 统一梯度          │
  │  GPU 2: batch 2 ──→ grad 2 ─┤                                   │
  │  GPU 3: batch 3 ──→ grad 3 ─┘                                   │
  └──────────────────────────────────────────────────────────────────┘

  ┌──────────────────────────────────────────────────────────────────┐
  │  张量并行 (Tensor Parallel, TP)                                  │
  │  ──────────────────────────────                                  │
  │  把一个矩阵运算切到多张 GPU 上，每张算一部分                     │
  │  目的: 单层太大放不下一张卡                                       │
  │                                                                   │
  │  W [4096×16384] 切 4 份:                                         │
  │  GPU 0: W₀ [4096×4096]                                          │
  │  GPU 1: W₁ [4096×4096]                                          │
  │  GPU 2: W₂ [4096×4096]                                          │
  │  GPU 3: W₃ [4096×4096]                                          │
  │  → 每层需要 2 次 AllReduce                                      │
  └──────────────────────────────────────────────────────────────────┘

  ┌──────────────────────────────────────────────────────────────────┐
  │  流水线并行 (Pipeline Parallel, PP)                              │
  │  ──────────────────────────────────                              │
  │  不同的层放在不同的 GPU 上，前一组算完传给下一组                  │
  │  目的: 模型太深放不下一张卡                                       │
  │                                                                   │
  │  GPU 0: 层 1-8                                                   │
  │  GPU 1: 层 9-16                                                  │
  │  GPU 2: 层 17-24                                                 │
  │  GPU 3: 层 25-32                                                 │
  │  → 需要微批次 (micro-batch) 减少气泡                             │
  └──────────────────────────────────────────────────────────────────┘

  训练 = DP × TP × PP 三维组合
  如训练 LLaMA-65B: DP=64, TP=8, PP=4 → 共 2048 张 GPU
```

### 推理的并行策略

```
推理时的并行策略 (简单得多):

═══════════════════════════════════════════════════════════════════════════

  ┌──────────────────────────────────────────────────────────────────┐
  │  张量并行 (TP) — 推理最常用                                      │
  │  ──────────────────────────                                      │
  │  目的: 模型太大一张卡放不下、或加速单请求延迟                     │
  │  切法和训练一样，但通信压力更小 (没有反向传播的通信)              │
  │                                                                   │
  │  LLaMA-7B 推理: TP=2 即可 (2 张 GPU，每张放一半参数)            │
  │  LLaMA-70B 推理: TP=8 (8 张 GPU)                                │
  └──────────────────────────────────────────────────────────────────┘

  ┌──────────────────────────────────────────────────────────────────┐
  │  数据并行 — 推理通常不叫 DP，而是"多实例部署"                   │
  │  ────────────────────────────────────────────                    │
  │  开多个独立的推理进程，每个处理不同的用户请求                     │
  │  和训练的 DP 不同: 不需要 AllReduce (因为没有梯度同步!)          │
  │                                                                   │
  │  实例 0: 处理用户 A 的请求                                       │
  │  实例 1: 处理用户 B 的请求                                       │
  │  实例 2: 处理用户 C 的请求                                       │
  └──────────────────────────────────────────────────────────────────┘

  ┌──────────────────────────────────────────────────────────────────┐
  │  流水线并行 — 推理很少用                                         │
  │  ────────────────────────                                        │
  │  原因: 推理是逐 token 生成，每步的计算量很小                     │
  │        PP 的层间通信延迟会占比极高，得不偿失                      │
  │        (训练时一个 micro-batch 很大，通信占比低)                  │
  └──────────────────────────────────────────────────────────────────┘

  推理并行 vs 训练并行:
  ┌────────────────┬────────────┬────────────┐
  │    策略         │    训练    │    推理     │
  ├────────────────┼────────────┼────────────┤
  │ 张量并行 (TP)   │ 常用      │ 最常用     │
  │ 数据并行 (DP)   │ 常用      │ 多实例部署 │
  │ 流水线并行 (PP) │ 常用      │ 很少用     │
  │ 梯度通信        │ AllReduce │ 不需要     │
  └────────────────┴────────────┴────────────┘
```

## Batch 处理差异

### 训练：追求大 Batch，充分喂饱 GPU

```
训练的 Batch 策略:

═══════════════════════════════════════════════════════════════════════════

  训练数据是预先准备好的，batch 大小可以自由控制:

  Global batch size = 4M tokens (常见的大模型训练配置)
  = micro_batch (4) × seq_len (2048) × gradient_accumulation (64) × DP (8)

  ┌──────────────────────────────────────────────────────────────────┐
  │                                                                   │
  │  每个 GPU 一次处理:                                               │
  │  micro_batch=4, seq=2048 → 8192 tokens                          │
  │                                                                   │
  │  矩阵形状: [4×2048, 4096] = [8192, 4096]                        │
  │  → 大矩阵! GPU Tensor Core 满载运转                              │
  │  → 算力利用率可达 50-60% (MFU)                                   │
  │                                                                   │
  └──────────────────────────────────────────────────────────────────┘

  生活类比:
  训练 = 食堂大厨: 一锅炒 100 份，效率极高
  所有"食材"(训练样本) 提前准备好，按节奏投入
```

### 推理：请求零散，需要 Continuous Batching

```
推理的 Batch 策略:

═══════════════════════════════════════════════════════════════════════════

  推理请求来自用户，特点:
  • 时间不可预测 (用户随时发请求)
  • 长度不统一 (有人问一句话，有人让写一篇文章)
  • 实时性要求高 (用户不愿等太久)

  ────── 朴素做法: Static Batching ──────

  等攒够 8 个请求再一起处理:
  ┌──────────────────────────────────────────────┐
  │  请求 1: "你好"          (生成 3 token)      │ ← 很快就完了
  │  请求 2: "写一首诗"      (生成 100 token)    │
  │  请求 3: "写一篇论文"    (生成 500 token)    │ ← 要等很久
  │  ...                                         │
  │  → 请求 1 生成完了还得等请求 3 结束!         │
  │  → 短请求被长请求拖累，GPU 利用率低          │
  └──────────────────────────────────────────────┘

  ────── 现代做法: Continuous Batching ──────

  核心思想: 有请求完成就立即插入新请求，不等整个 batch 结束

  时间轴 →
  ┌────┬────┬────┬────┬────┬────┬────┬────┬────┐
  │ R1 │ R1 │ R1 │ R4 │ R4 │ R4 │ R4 │ R7 │ R7 │  ← 槽位 1
  │ R2 │ R2 │ R2 │ R2 │ R2 │ R5 │ R5 │ R5 │ R5 │  ← 槽位 2
  │ R3 │ R3 │ R3 │ R3 │ R3 │ R3 │ R3 │ R3 │ R6 │  ← 槽位 3
  └────┴────┴────┴────┴────┴────┴────┴────┴────┘
        ↑ R1完成       ↑ R2完成            ↑ R3完成
        立即插入R4     立即插入R5          立即插入R6

  → 每个"槽位"始终有活跃请求
  → GPU 不会因为等待而闲置
  → 这就是 vLLM、TGI 等推理框架的核心技术之一

═══════════════════════════════════════════════════════════════════════════
```

## 精度策略差异

### 训练：混合精度（BF16/FP16），不能太激进

```
训练的精度策略:

═══════════════════════════════════════════════════════════════════════════

  为什么训练不能用低精度 (如 INT8)?
  → 训练需要算梯度，梯度值通常很小 (如 0.00001)
  → 低精度无法表示这么小的数 → 梯度消失 → 训练崩溃!

  现代做法: 混合精度训练 (Mixed Precision)
  ┌──────────────────────────────────────────────────────────────────┐
  │                                                                   │
  │  FP32 (32位)       BF16/FP16 (16位)                              │
  │  ────────────       ──────────────────                            │
  │  优化器状态          前向传播计算                                  │
  │  参数主副本          反向传播计算                                  │
  │  梯度累加            梯度存储                                      │
  │                                                                   │
  │  流程:                                                            │
  │  FP32 参数 → 转成 BF16 → 前向传播 (BF16) → 反向传播 (BF16)      │
  │       ↑                                          │                │
  │       └── 用 FP32 更新 ←── 梯度转成 FP32 ←──────┘                │
  │                                                                   │
  │  核心思想:                                                        │
  │  • 计算用 BF16 → 快! (Tensor Core 加速)                          │
  │  • 存储和更新用 FP32 → 精度不丢!                                  │
  │                                                                   │
  └──────────────────────────────────────────────────────────────────┘

  数据类型对比:
  ┌──────────┬────────┬──────────────┬───────────────────────────┐
  │  类型     │ 字节数 │ 精度          │ 用途                      │
  ├──────────┼────────┼──────────────┼───────────────────────────┤
  │  FP32    │ 4      │ ~7位有效数字  │ 训练: 优化器状态/参数主副本│
  │  BF16    │ 2      │ ~3位有效数字  │ 训练: 计算 (范围大)       │
  │  FP16    │ 2      │ ~4位有效数字  │ 训练: 计算 (精度高一点)   │
  └──────────┴────────┴──────────────┴───────────────────────────┘
```

### 推理：可以激进量化（INT8/INT4 甚至更低）

```
推理的精度策略:

═══════════════════════════════════════════════════════════════════════════

  为什么推理可以用低精度?
  → 推理不算梯度! 不需要精确的小数
  → 参数在训练后就"定型"了，只做矩阵乘法
  → 适当降低精度对输出质量影响很小

  量化 (Quantization) — 把参数压缩到更少的 bit:
  ┌──────────────────────────────────────────────────────────────────┐
  │                                                                   │
  │  FP16 → INT8  (16位 → 8位)                                      │
  │  模型体积减半: LLaMA-7B 14GB → 7GB                               │
  │  速度提升 ~1.5-2x (读取数据量减半!)                               │
  │  质量几乎无损                                                     │
  │                                                                   │
  │  FP16 → INT4  (16位 → 4位)                                      │
  │  模型体积 1/4: LLaMA-7B 14GB → 3.5GB                             │
  │  速度提升 ~2-3x                                                   │
  │  质量略有下降，但大多数场景可接受                                  │
  │                                                                   │
  │  甚至 INT3/INT2... (研究前沿)                                    │
  │                                                                   │
  └──────────────────────────────────────────────────────────────────┘

  量化后的效果 (LLaMA-7B):
  ┌──────────┬──────────┬──────────┬────────────────────────────────┐
  │  精度     │ 参数体积 │ 推理速度 │ 可运行硬件                      │
  ├──────────┼──────────┼──────────┼────────────────────────────────┤
  │  FP16    │ 14 GB    │ 1x (基准)│ A100/A10 等专业 GPU             │
  │  INT8    │  7 GB    │ ~1.5-2x  │ RTX 4090 (24GB) 轻松跑        │
  │  INT4    │  3.5 GB  │ ~2-3x    │ RTX 3060 (12GB) 甚至笔记本!   │
  └──────────┴──────────┴──────────┴────────────────────────────────┘

  为什么推理是 memory-bound 所以量化这么有效？
  → Decode 阶段瓶颈是"从显存读参数到计算单元"
  → INT4 的参数体积只有 FP16 的 1/4
  → 读取时间缩短到 1/4 → 推理速度接近 4x!
  → 这就像快递小哥：包裹变轻了，跑得就快了

═══════════════════════════════════════════════════════════════════════════
```

## Prefill vs Decode：推理的两个阶段

推理不是一个均匀的过程，它分为两个计算特征完全不同的阶段。

```
═══════════════════════════════════════════════════════════════════════════
  推理的两个阶段
═══════════════════════════════════════════════════════════════════════════

  用户输入: "请帮我写一首关于春天的诗"  (10 个 token)
  模型输出: "春风拂面暖阳照，..." (假设 100 个 token)

  ────── 阶段 1: Prefill (预填充) ──────

  处理用户输入的 10 个 token:
  ┌─────────────────────────────────────────────────────┐
  │  输入: [请, 帮, 我, 写, 一, 首, 关, 于, 春, 天]     │
  │                                                      │
  │  10 个 token 同时并行计算! (类似训练的前向传播)      │
  │  矩阵形状: [10, 4096] × [4096, 4096]               │
  │  → 比较"胖"的矩阵 → Compute-bound                  │
  │  → 耗时约几十毫秒                                    │
  │                                                      │
  │  输出: 第一个生成 token + 所有输入的 KV Cache        │
  └─────────────────────────────────────────────────────┘

  ────── 阶段 2: Decode (解码生成) ──────

  逐个生成 100 个 token:
  ┌─────────────────────────────────────────────────────┐
  │  Step 1: 只处理 1 个新 token → 生成 "春"            │
  │  Step 2: 只处理 1 个新 token → 生成 "风"            │
  │  ...                                                 │
  │  Step 100: 只处理 1 个新 token → 生成 "。"          │
  │                                                      │
  │  每步矩阵形状: [1, 4096] × [4096, 4096]            │
  │  → 极"瘦"的矩阵 → Memory-bound                     │
  │  → 每步耗时约几毫秒，但要跑 100 步!                 │
  │  → 总耗时: 100 × 几ms = 几百毫秒到数秒             │
  └─────────────────────────────────────────────────────┘
```

### 两个阶段的对比

```
  ┌─────────────────┬──────────────────┬──────────────────┐
  │                  │     Prefill      │     Decode       │
  ├─────────────────┼──────────────────┼──────────────────┤
  │ 处理内容         │ 全部输入 token   │ 每次 1 个 token  │
  │ 计算模式         │ 并行 (一次算完)  │ 串行 (逐步生成)  │
  │ 矩阵形状         │ [N, hidden]      │ [1, hidden]      │
  │ 瓶颈             │ Compute-bound   │ Memory-bound     │
  │ GPU 利用率       │ 高              │ 低               │
  │ 耗时             │ 短 (一次完成)    │ 长 (步数多)      │
  │ 感知             │ "首字延迟" TTFT │ "每字延迟" TPOT  │
  └─────────────────┴──────────────────┴──────────────────┘

  TTFT = Time To First Token (首 token 延迟) → 取决于 Prefill 速度
  TPOT = Time Per Output Token (每 token 延迟) → 取决于 Decode 速度
```

### 时间线可视化

```
  一次完整的推理请求:

  时间 →
  ├── Prefill ──┤──────────── Decode ────────────────────────┤
  │ 处理输入     │ 生成 token 1, 2, 3, ... N                 │
  │ (并行)       │ (串行, 逐个生成)                           │
  │ ~30ms        │ ~5ms × 100 tokens = ~500ms                │
  ├──────────────┤────────────────────────────────────────────┤

  用户体验:
  用户发送 ──→ [等 30ms] ──→ 第一个字出现 ──→ 逐字流式输出 ──→ 完成
              (Prefill)       (TTFT)            (Decode)

  这就是为什么你和 ChatGPT 对话时:
  1. 点发送后会"卡"一下       → Prefill 阶段
  2. 然后文字一个一个蹦出来    → Decode 阶段
  3. 短问题"卡"的时间短       → 输入少，Prefill 快
  4. 长文档"卡"的时间长       → 输入多，Prefill 慢
```

### 为什么分开理解很重要？

```
  不同阶段需要不同的优化策略:

  ┌──────────────────────────────────────────────────────────────────┐
  │                                                                   │
  │  Prefill 优化 (减少首字延迟):                                    │
  │  • 用 Flash Attention 加速注意力计算                              │
  │  • 用 TP 并行分摊计算                                            │
  │  • 输入太长时做 chunked prefill (分块处理)                       │
  │                                                                   │
  │  Decode 优化 (提高吞吐):                                         │
  │  • 量化 (INT8/INT4) 减少显存读取 → 最有效!                      │
  │  • Continuous Batching 攒多个用户一起 decode                     │
  │  • Speculative Decoding (投机解码): 用小模型猜多个 token          │
  │    大模型一次验证 → 减少 decode 步数                              │
  │  • KV Cache 优化 (PagedAttention): 高效管理显存                  │
  │                                                                   │
  └──────────────────────────────────────────────────────────────────┘
```

```
═══════════════════════════════════════════════════════════════════════════
  训练 vs 推理 全维度对比 (以 LLaMA-7B 为例)
═══════════════════════════════════════════════════════════════════════════

┌────────────────┬────────────────────────┬────────────────────────────┐
│     维度        │       训练              │         推理                │
├────────────────┼────────────────────────┼────────────────────────────┤

│ 频率           │ 做一次 (几天~几个月)    │ 做千万次 (每个用户请求)     │
├────────────────┼────────────────────────┼────────────────────────────┤
│ 计算流程        │ 前向 + 反向 + 参数更新  │ 仅前向 (自回归逐token)     │
│ 计算量/step     │ ≈ 3× 前向传播          │ ≈ 1× 前向传播 (每token)    │
│ 并行度          │ 全序列并行 (Teacher     │ Prefill并行, Decode串行    │
│                │   Forcing)              │                            │
├────────────────┼────────────────────────┼────────────────────────────┤
│ 显存 - 参数     │ 14 GB (FP16)           │ 14 GB (FP16) 或更少       │
│ 显存 - 梯度     │ 14 GB                  │ 0 GB                      │
│ 显存 - 优化器   │ 84 GB (AdamW FP32)     │ 0 GB                      │
│ 显存 - 中间值   │ ~15 GB (激活值)         │ ~1 GB/用户 (KV Cache)     │
│ 显存 - 总计     │ ~130 GB                │ 15-46 GB                  │
├────────────────┼────────────────────────┼────────────────────────────┤
│ 瓶颈           │ Compute-bound          │ Memory-bound (Decode)     │
│                │ (大矩阵, 算力打满)      │ (瘦矩阵, 等数据搬运)      │
│ 算术强度        │ ~5600 (远超拐点)        │ ~1 (远低于拐点)           │
├────────────────┼────────────────────────┼────────────────────────────┤
│ 并行策略        │ DP + TP + PP (三维)     │ TP 为主 + 多实例部署      │
│ GPU 数量        │ 几十~几千张             │ 1-8 张 (单实例)           │
│ 通信            │ AllReduce (梯度同步)    │ AllReduce (TP) 无梯度通信 │
├────────────────┼────────────────────────┼────────────────────────────┤
│ Batch          │ 大batch (数百万token)   │ Continuous Batching       │
│                │ 充分喂饱GPU            │ 动态插入新请求             │
├────────────────┼────────────────────────┼────────────────────────────┤
│ 精度           │ 混合精度 BF16+FP32     │ 量化 INT8/INT4 甚至更低   │
│                │ (梯度需要精度!)         │ (不算梯度，可激进压缩)     │
├────────────────┼────────────────────────┼────────────────────────────┤
│ 优化方向        │ 更多GPU、更大batch     │ 量化、KV Cache优化、       │
│                │ ZeRO节省显存           │ 投机解码、PagedAttention   │
├────────────────┼────────────────────────┼────────────────────────────┤
│ 硬件选择        │ A100/H100 集群         │ A10/L40/消费级 GPU 也行   │
│ 成本结构        │ 一次性高投入            │ 持续运营成本               │
│ 关注指标        │ MFU (算力利用率)        │ TTFT, TPOT (延迟)        │
│                │ 训练吞吐 (tokens/s)    │ 推理吞吐 (requests/s)     │
└────────────────┴────────────────────────┴────────────────────────────┘
```
