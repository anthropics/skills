## 什么是上下文窗口

### 定义

上下文窗口（Context Window）是 LLM 在一次推理中能处理的**最大 Token 数量**，包含输入（Prompt）和输出（Generation）的总和。

```
┌─────────────── 上下文窗口 (Context Window) ───────────────┐
│                                                             │
│  ┌─────────────────── Input ───────────────────┐ ┌Output─┐ │
│  │                                             │ │       │ │
│  │ [System] [History] [RAG Docs] [User Query]  │ │[Reply]│ │
│  │                                             │ │       │ │
│  └─────────────────────────────────────────────┘ └───────┘ │
│                                                             │
│  总 Token 数 = Input Tokens + Output Tokens ≤ Window Size  │
└─────────────────────────────────────────────────────────────┘
```

### 生活类比

把上下文窗口想象成你在图书馆做研究时的**阅览桌**：

| 概念 | 类比 | 说明 |
|------|------|------|
| 上下文窗口 | 桌面面积 | 能同时摊开多少资料 |
| Input Tokens | 参考书和笔记 | 你放在桌上的材料 |
| Output Tokens | 你写的内容 | 你要产出的成果 |
| 位置编码 | 书签/页码 | 记住每本书摊开到哪一页 |
| KV Cache | 每本书占的面积 | 每份材料消耗的桌面空间 |
| 窗口溢出 | 桌面放不下了 | 必须拿走一些材料才能继续 |

### Token 化基础

不同模型的 Tokenizer 会影响"同样的文本占多少上下文"：

```
英文文本 "Hello, how are you?"
  → GPT-4 (cl100k_base): ["Hello", ",", " how", " are", " you", "?"]  = 6 tokens
  → LLaMA 3 (tiktoken):  ["Hello", ",", " how", " are", " you", "?"]  = 6 tokens

中文文本 "你好，最近怎么样？"
  → GPT-4:  ["你好", "，", "最近", "怎么样", "？"]  = 5 tokens
  → LLaMA 3: ["你", "好", "，", "最近", "怎么样", "？"]  = 6 tokens

经验法则：
  英文：1 token ≈ 4 个字符 ≈ 0.75 个单词
  中文：1 token ≈ 1-2 个汉字
  代码：token 效率通常低于自然语言（符号多）
```

  │       Claude 2 ──────────── 100K
  │       LLaMA 2 ──────────── 4K (RoPE)
  │       Code LLaMA ────────── 16K → 100K (RoPE 扩展)
  │       RoPE 扩展研究爆发 ──── PI, NTK, YaRN
  │
2024 ──── GPT-4 Turbo ────────── 128K
  │       Claude 3.5 ─────────── 200K

  │       LLaMA 3.1 ──────────── 128K (增大 RoPE 基数)

  │

  │       Claude 3.5 (extended)── 200K+

### 驱动上下文扩展的关键技术

| 技术突破 | 时间 | 影响 |
|---------|------|------|
| 绝对位置编码 → RoPE | 2022 | 支持相对位置关系，理论上可扩展 |
| FlashAttention | 2022 | 显存从 O(n²) 降为 O(n)，解锁长序列训练 |
| RoPE 扩展 (PI/NTK/YaRN) | 2023 | 不重新训练即可扩展上下文 |
| Ring Attention | 2023 | 跨 GPU 分布式处理超长序列 |
| PagedAttention | 2023 | KV Cache 动态管理，减少碎片 |
| 序列并行 | 2024 | 训练时把长序列切分到多卡 |

## 位置编码：让模型知道"在哪"

Transformer 的 Self-Attention 本身是**位置无关的**——不加位置信息，"猫吃鱼"和"鱼吃猫"对模型来说一样。位置编码告诉模型每个 Token 在序列中的位置。

> 📖 位置编码的基础原理见 [00-training-fundamentals/10-transformer-architecture.md](../00-training-fundamentals/10-transformer-architecture.md)。本节重点讲其对上下文扩展的影响。

### 主流位置编码方案对比

| 方案 | 原理 | 最大长度 | 外推能力 | 代表模型 |
|------|------|---------|---------|---------|
| 绝对位置编码 | 每个位置一个学习向量 | 固定（= 训练长度）| ❌ 完全无法外推 | GPT-2, BERT |
| 正弦位置编码 | sin/cos 函数 | 理论无限 | ⚠️ 有限 | Transformer 原始 |
| RoPE | 旋转矩阵编码相对位置 | 理论无限 | ⚠️ 需要扩展 | LLaMA, Qwen, DeepSeek |
| ALiBi | Attention 加线性偏置 | 理论无限 | ✅ 天然外推 | BLOOM, MPT |

### RoPE 详解

RoPE（Rotary Position Embedding）是当前最主流的位置编码，LLaMA、Qwen、DeepSeek 等都在使用。

```
RoPE 的直觉：

想象一个时钟的指针：
  - 不同"频率"的指针旋转速度不同
  - 秒针转得快（高频）→ 编码近距离位置差异
  - 时针转得慢（低频）→ 编码远距离位置关系

数学形式：
  对 embedding 维度 i，位置 m 的旋转角度：
  θ_i = m × base^(-2i/d)
  
  其中 base = 10000（默认值）
  
  Query 和 Key 在 Attention 前乘上旋转矩阵：
  q'_m = R(m×θ) · q_m
  k'_n = R(n×θ) · k_n
  
  点积只依赖相对距离 (m-n)：
  q'_m · k'_n = f(q, k, m-n)
```

### RoPE 的外推问题

```
训练长度 = 4K tokens

位置 0-4095：训练时见过 → ✅ 模型正常工作
位置 4096+：  从未见过  → ❌ 角度超出训练分布

后果：
  ┌────────────────────────────────────────────────────┐
  │  PPL (Perplexity) vs 序列长度                       │
  │                                                     │
  │  PPL                                                │
  │  1000 ┤                                    ████     │
  │       │                               ████         │
  │   100 ┤                          █████              │
  │       │                     █████                   │
  │    10 ┤════════════════════                         │
  │       │  ← 训练长度内 →  │← 训练长度外 →            │
  │     1 ┼─────────────────┼───────────────────────   │
  │       0    2K    4K    6K    8K    10K   12K       │
  │                   序列长度 (tokens)                 │
  │                                                     │
  │  在训练长度内 PPL 平稳，超过后急剧上升                 │
  └────────────────────────────────────────────────────┘

解决方案 → 见 04-long-context-inference.md (RoPE 扩展)
```

## 标称窗口 vs 有效上下文

### "Lost in the Middle" 现象

2023 年斯坦福的研究（Liu et al.）发现：**即使模型声称支持长上下文，它对中间位置信息的利用能力显著弱于开头和结尾**。

```
实验设置（Needle-in-a-Haystack）：
  1. 构造一个长文本（干草堆）
  2. 在不同位置插入一个关键事实（针）
  3. 提问：这个关键事实是什么？
  4. 测量不同位置的检索准确率

实验结果：

检索准确率 (%)
100 ┤████                                                  ████
    │████                                                  ████
 90 ┤████ ▓▓▓▓                                        ▓▓▓▓████
    │████ ▓▓▓▓                                        ▓▓▓▓████
 80 ┤████ ▓▓▓▓ ▓▓▓▓                            ▓▓▓▓  ▓▓▓▓████
    │████ ▓▓▓▓ ▓▓▓▓                            ▓▓▓▓  ▓▓▓▓████
 70 ┤████ ▓▓▓▓ ▓▓▓▓ ░░░░                 ░░░░  ▓▓▓▓  ▓▓▓▓████
    │████ ▓▓▓▓ ▓▓▓▓ ░░░░  ░░░░     ░░░░  ░░░░  ▓▓▓▓  ▓▓▓▓████
 60 ┤████ ▓▓▓▓ ▓▓▓▓ ░░░░  ░░░░  ░░░░  ░░░░  ░░░░  ▓▓▓▓  ▓▓▓▓████
    │████ ▓▓▓▓ ▓▓▓▓ ░░░░  ░░░░  ░░░░  ░░░░  ░░░░  ▓▓▓▓  ▓▓▓▓████

    开头   1/8   2/8   3/8   4/8   5/8   6/8   7/8  结尾
                           关键信息位置

████ = 高准确率 (>85%)    ▓▓▓▓ = 中等 (65-85%)    ░░░░ = 低准确率 (<65%)
```

### 不同模型的"U 型曲线"差异

| 模型 | 标称窗口 | 开头准确率 | 中间准确率 | 结尾准确率 | 有效利用率 |
|------|---------|----------|----------|----------|----------|
| GPT-3.5 (4K) | 4K | 95% | 52% | 90% | ~70% |
| GPT-4 (128K) | 128K | 98% | 73% | 95% | ~80% |
| Claude 3.5 (200K) | 200K | 99% | 85% | 97% | ~90% |
| LLaMA 3.1 (128K) | 128K | 96% | 68% | 93% | ~75% |
| Gemini 1.5 Pro (1M) | 1M | 98% | 78% | 96% | ~85% |

> **关键洞察**：标称窗口 ≠ 有效窗口。128K 窗口的模型，实际"可靠利用"的上下文可能只有 80-100K。

### 为什么会 Lost in the Middle

```
原因分析：

1. 注意力分布不均匀
   ┌─────────────────────────────────────────────┐
   │  Attention 权重在长序列上的分布                │
   │                                               │
   │  权重 ┤████                          ████████ │
   │       │████░░░░░░░░░░░░░░░░░░░░░░░░████████ │
   │       │████░░░░░░░░░░░░░░░░░░░░░░░░████████ │
   │       ┼────┬────┬────┬────┬────┬────┬─────── │
   │       开头      中间              结尾        │
   │                                               │
   │  Softmax 对位置有隐式偏好：                     │
   │  - 开头：Attention Sink 效应                   │
   │  - 结尾：因果注意力的"短距离偏好"               │
   │  - 中间：被稀释                                │
   └─────────────────────────────────────────────┘

2. 训练数据偏差
   - 训练时大多数答案在 Prompt 的开头或结尾
   - 模型学到了"答案大概在哪"的位置先验

3. Softmax 归一化
   - 上下文越长，Softmax 越"平均"
   - 真正重要的 Token 得到的注意力被稀释
```

### 缓解策略

| 策略 | 方法 | 效果 | 代价 |
|------|------|------|------|
| **信息位置优化** | 关键信息放开头/结尾 | ✅ 免费，立竿见影 | 需要应用层调整 |
| **重复强化** | 重要信息在 Prompt 中重复 2-3 次 | ✅ 明显提升 | 浪费 Token |
| **分块检索** | RAG 分块后只取 Top-K | ✅ 显著 | 可能丢信息 |
| **中间填充训练** | 训练时随机化答案位置 | ✅ 根本解决 | 需要重新训练 |
| **注意力缩放** | 长序列时对 Attention 做温度缩放 | ⚠️ 轻微 | 可能影响质量 |

## 显存与计算复杂度分析

### 上下文长度的三重代价

每增加上下文长度，会同时增加三个维度的开销：

```
┌─────────────────────────────────────────────────────────────────┐
│                上下文长度的三重代价                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 显存代价：KV Cache                                            │
│     KV Cache = 2 × L × H_kv × d_h × dtype × seq_len × batch    │
│     → 线性增长 O(seq_len)                                        │
│     → LLaMA-7B: 每增加 1K tokens → +0.5 GB (FP16, batch=1)      │
│                                                                  │
│  2. 计算代价：Self-Attention                                      │
│     Attention FLOPs = 4 × L × d × seq_len²                      │
│     → 二次方增长 O(seq_len²)                                     │
│     → 128K vs 4K: 计算量增加 1024 倍!                             │
│                                                                  │
│  3. 延迟代价：TTFT (Time to First Token)                          │
│     TTFT ≈ seq_len × Prefill_time_per_token                     │
│     → 线性增长 O(seq_len)                                        │
│     → 128K prompt on A100: TTFT ≈ 10-30 秒                      │
│                                                                  │
│  注：FlashAttention 不减少 FLOPs，只减少 HBM 访问次数              │
│      计算量不变，但实际 wall-clock time 显著降低                    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### KV Cache 显存计算详解

> 📖 公式推导与计算函数详见 [03-inference-serving/05-llm-inference-kv-cache.md](../03-inference-serving/05-llm-inference-kv-cache.md)

```
KV Cache 公式：
  KV_bytes = 2 × num_layers × num_kv_heads × head_dim × dtype_bytes × seq_len × batch_size

其中：
  2           = K 和 V 两个缓存
  num_layers  = Transformer 层数
  num_kv_heads = KV Head 数量（MHA = Q heads, GQA = Q heads / group_size）
  head_dim    = 每个 Head 的维度
  dtype_bytes = 数据类型字节数（FP16=2, FP8=1, INT4=0.5）
  seq_len     = 序列长度
  batch_size  = 批处理大小
```

**主流模型 KV Cache 对比**（Batch=1, FP16）：

| 模型 | Layers | KV Heads | Head Dim | 每 Token KV | 4K seq | 32K seq | 128K seq |
|------|--------|----------|----------|------------|--------|---------|----------|
| LLaMA-7B (MHA) | 32 | 32 | 128 | 512 KB | 2 GB | 16 GB | 64 GB |
| LLaMA-3.1 8B (GQA) | 32 | 8 | 128 | 128 KB | 0.5 GB | 4 GB | 16 GB |
| LLaMA-2 70B (GQA) | 80 | 8 | 128 | 320 KB | 1.25 GB | 10 GB | 40 GB |
| LLaMA-3.1 70B (GQA) | 80 | 8 | 128 | 320 KB | 1.25 GB | 10 GB | 40 GB |
| Qwen2.5-72B (GQA) | 80 | 8 | 128 | 320 KB | 1.25 GB | 10 GB | 40 GB |

**关键发现**：GQA 将 KV Cache 减少了 4-8 倍（取决于 group size）。对于 128K 上下文，这是决定"能否单卡推理"的关键。

### Attention 计算复杂度

```python
def attention_flops(seq_len: int, num_layers: int, hidden_dim: int) -> float:
    """计算 Self-Attention 的 FLOPs (不含 FFN)"""
    # Q×K^T: seq_len × seq_len × hidden_dim
    # Softmax: seq_len × seq_len
    # Attn×V: seq_len × seq_len × hidden_dim
    # Q/K/V 投影: 3 × seq_len × hidden_dim × hidden_dim
    
    # 简化：每层 Attention ≈ 4 × seq_len² × hidden_dim (QK^T + Softmax + AV)
    #       + 4 × seq_len × hidden_dim² (Q/K/V/O 投影)
    per_layer = 4 * seq_len * seq_len * hidden_dim + 4 * seq_len * hidden_dim * hidden_dim
    return per_layer * num_layers

# LLaMA-7B: 32 layers, hidden_dim=4096
for seq in [4096, 32768, 131072]:
    flops = attention_flops(seq, 32, 4096)
    print(f"seq={seq:>7}: Attention FLOPs = {flops:.2e}")
    
# 输出：
# seq=  4096: Attention FLOPs = 2.75e+12  (基准)
# seq= 32768: Attention FLOPs = 1.41e+14  (51× 基准)
# seq=131072: Attention FLOPs = 2.21e+15  (803× 基准)
#
# 结论：上下文从 4K 增加到 128K，Attention 计算量增加 ~800 倍
```

### FlashAttention 的作用

FlashAttention **不减少计算量**（FLOPs 不变），但通过优化内存访问模式显著降低实际延迟：

```
标准 Attention 的内存访问模式：
  1. Q × K^T → 写入 HBM（O(n²) 显存）
  2. Softmax → 读写 HBM
  3. Score × V → 读 HBM + 写 HBM
  
  → 需要 O(n²) 显存存储 Attention 矩阵
  → 大量 HBM 读写（带宽瓶颈）

FlashAttention 的 Tiling 策略：
  1. 把 Q/K/V 分成小块（Block）
  2. 每个 Block 在 SRAM 中计算完整 Attention
  3. 用 Online Softmax 逐块累积结果
  4. 永远不写 O(n²) 的 Attention 矩阵到 HBM
  
  → 显存从 O(n²) 降为 O(n)
  → 大幅减少 HBM 读写
  → 实际速度提升 2-4×（长序列提升更大）

┌────────────────────────────────────────────┐
│         标准 vs FlashAttention              │
│                                             │
│             显存       速度      FLOPs      │
│  标准      O(n²)      1×        基准       │
│  Flash-2   O(n)       2-4×      基准       │
│  Flash-3   O(n)       1.5-2×    基准       │
│            (vs FA2)                         │
│                                             │
│  "FlashAttention = 在不减少计算的前提下，     │
│   通过更好的内存访问模式实现加速"              │
└────────────────────────────────────────────┘
```

> 📖 FlashAttention 的完整实现原理见 [03-inference-serving/07-llm-inference-attention.md](../03-inference-serving/07-llm-inference-attention.md)

## 上下文窗口的实际应用限制

### 场景化建议

不同场景对上下文窗口的实际需求差异很大：

| 场景 | 实际需求 | 推荐窗口 | 限制因素 |
|------|---------|---------|---------|
| 单轮 QA | 问题 + 参考文档 | 4K-16K | 文档太长需 RAG 分块 |
| 多轮对话 | 历史 + 当前 | 8K-32K | 历史越长噪声越多 |
| RAG 应用 | System + Chunks + Query | 8K-32K | 过多 chunk 反而降质 |
| 代码补全 | 当前文件 + 相关文件 | 16K-64K | 远处代码相关性递减 |
| 文档分析 | 完整文档 | 32K-128K | Lost in the Middle |
| Agent 任务 | 工具描述 + 历史 + 当前 | 16K-64K | 工具描述占大量 Token |
| 长文翻译 | 完整原文 | 32K-128K | 一致性要求高 |

### 经验法则

```
上下文窗口使用的黄金法则：

1. 不要用满——预留 20-30% 给输出和安全边际
   128K 窗口 → 实际输入最好不超过 90-100K

2. 质量 > 数量——少但精准的上下文 > 多但噪声的上下文
   RAG: Top-3 高相关 chunk > Top-20 混杂 chunk

3. 位置有讲究——关键信息放开头和结尾
   [System Prompt（开头）] + [...] + [关键指令（结尾）]

4. 长上下文不是万能药——128K ≠ "把所有东西都塞进去"
   超过 32K 后，边际收益递减，成本线性增加

5. 监控实际利用率——跟踪每个请求的 input/output token 分布
   大多数请求可能只用了 2-8K，少数长尾用了 64K+
```

### 成本影响

```
上下文长度 vs 推理成本（以 GPT-4 API 为例）：

  Input Token 价格: $10 / 1M tokens (GPT-4o)

  ┌──────────┬───────────────┬────────────────────┐
  │ 上下文    │ 每次请求成本   │ 10万次请求/天成本   │
  ├──────────┼───────────────┼────────────────────┤
  │  1K      │  $0.01        │  $1,000/天          │
  │  4K      │  $0.04        │  $4,000/天          │
  │  32K     │  $0.32        │  $32,000/天         │
  │  128K    │  $1.28        │  $128,000/天        │
  └──────────┴───────────────┴────────────────────┘

  128K 上下文的成本是 1K 的 128 倍——
  确保你真的需要这么长的上下文！

  → 成本优化策略见 07-context-engineering.md
```

```
┌─────────────────────────────────────────────────────────────┐
│              上下文窗口基础 - 关键要点                         │
├─────────────────────────────────────────────────────────────┤
│                                                              │

│                                                              │

```
