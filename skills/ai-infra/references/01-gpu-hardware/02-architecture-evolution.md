GPU 架构的演进就像工厂的升级换代：

```
┌───────────────────────────────────────────────────────────────────────┐

│  2017 Volta (V100)     "首条自动化流水线"                             │
│  ════════════════       首次引入 Tensor Core，AI 专用硬件的起点       │
│       │                 类比：工厂里第一次用机器人取代手工              │
│       │                                                               │
│  2020 Ampere (A100)    "智能工厂 1.0"                                │
```

### 每代核心突破一句话总结

```
Volta:     给 GPU 装上了"矩阵计算专用芯片" (Tensor Core)
Ampere:    让 GPU 学会了"偷懒" (稀疏加速) 和 "分身术" (MIG)

```

## Volta 架构 (2017)

### 里程碑意义

Volta 是第一个真正为深度学习设计的数据中心 GPU，**首次引入 Tensor Core**。

在 Volta 之前，深度学习用的是通用 CUDA Core 做矩阵乘法——就像用通用螺丝刀拧所有螺丝。Tensor Core 则是"专用电动螺丝刀"，对矩阵运算效率提升了一个数量级。

### 关键规格 (V100)

| 指标 | 数值 |
|------|------|
| SM 数量 | 80 |
| CUDA Cores | 5,120 |
| Tensor Cores | 640 |
| FP32 峰值 | 15.7 TFLOPS |
| FP16 峰值 (Tensor) | 125 TFLOPS |
| 显存 | 32 GB HBM2 |
| 带宽 | 900 GB/s |
| TDP | 300W |

### 第一代 Tensor Core

```
D = A × B + C

┌───────┐   ┌───────┐   ┌───────┐   ┌───────┐
│ A     │ × │ B     │ + │ C     │ = │ D     │
│ 4×4   │   │ 4×4   │   │ 4×4   │   │ 4×4   │
│ FP16  │   │ FP16  │   │ FP32  │   │ FP32  │
└───────┘   └───────┘   └───────┘   └───────┘

每个 Tensor Core 每周期: 64 FMA = 128 FLOPs

为什么是 FP16 输入 + FP32 累加？
  FP16 输入：节省带宽，数据搬运速度翻倍
  FP32 累加：防止大量小数相加时精度损失
  这就是"混合精度"的硬件基础
```

## Ampere 架构 (2020)

### 三大关键创新

#### 1. TF32 格式——"无痛加速"

```
问题：用户不想改代码，但想要加速

FP32:  [1 sign][8 exponent][23 mantissa] = 32 bits  精度高但慢
FP16:  [1 sign][5 exponent][10 mantissa] = 16 bits  快但范围小，易溢出
TF32:  [1 sign][8 exponent][10 mantissa] = 19 bits  取两者优点！

TF32 的巧妙之处：
  保留 FP32 的范围（8-bit 指数） → 不会溢出
  使用 FP16 级别的尾数（10-bit） → 可以用 Tensor Core 加速
  对用户完全透明：代码不变，自动加速

效果：
  FP32 矩阵乘法 → 自动用 TF32 在 Tensor Core 执行
  速度：~8x 提升（相比 CUDA Core FP32）
  精度：对 99% 的 AI 任务没有可见影响
```

#### 2. 结构化稀疏 2:4——"聪明地偷懒"

```
类比：考试中发现一半的题目答案为 0，可以跳过不算

原始矩阵 (Dense):
  [0.5, 0.0, 0.3, 0.0, 0.7, 0.0, 0.1, 0.0]
  
稀疏模式 2:4 (每 4 个元素最多 2 个非零):
  [0.5, _, 0.3, _] → 压缩为 [0.5, 0.3] + 索引 [0,2]
  [0.7, _, 0.1, _] → 压缩为 [0.7, 0.1] + 索引 [0,2]

硬件支持：
  Tensor Core 内置稀疏计算逻辑
  只计算非零元素 → 实际计算量减半 → 2x 加速
  存储也减半 → 带宽需求减半

实际应用：
  训练完成后，对权重做 2:4 剪枝
  在保持精度的前提下，推理速度翻倍
```

#### 3. MIG (多实例 GPU)——"一卡变多卡"

```
类比：一间大教室分隔成多间小会议室

没有 MIG：
  一张 A100 一次只能服务一个用户/模型
  7B 推理只用 14GB 显存，剩下 66GB 闲置

有 MIG：
  一张 A100 80GB 最多分为 7 个独立实例
  ┌─────────────────────────────────────────────┐
  │              A100 80GB                       │
  │  ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ... ×7  │
  │  │10GB │ │10GB │ │10GB │ │10GB │          │
  │  │用户A│ │用户B│ │用户C│ │用户D│          │
  │  └─────┘ └─────┘ └─────┘ └─────┘          │
  └─────────────────────────────────────────────┘
  
  每个实例有独立的：SM、显存、L2 Cache
  互不干扰，硬件级隔离
```

### 关键规格 (A100 80GB)

| 指标 | 数值 |
|------|------|
| SM 数量 | 108 |
| CUDA Cores | 6,912 |
| Tensor Cores | 432 |
| FP32 峰值 | 19.5 TFLOPS |
| TF32 峰值 | 156 TFLOPS |
| FP16 峰值 (Tensor) | 312 TFLOPS |
| FP16 稀疏峰值 | 624 TFLOPS |
| 显存 | 80 GB HBM2e |
| 带宽 | 2.0 TB/s |
| TDP | 400W |

## Hopper 架构 (2022)

### 核心创新：FP8 + Transformer Engine

#### FP8——"8 位也能训大模型"

```
为什么要用 FP8？
  FP16: 每个数占 2 bytes → 带宽需求 X
  FP8:  每个数占 1 byte  → 带宽需求 X/2
  带宽是 LLM 推理的瓶颈 → FP8 直接让吞吐翻倍

但 8 位怎么够精度？NVIDIA 设计了两种互补格式：

E4M3: [1 sign][4 exp][3 mantissa]  范围 ±448，精度较高
  用于：前向传播的权重和激活值
  类比：精确的计算器

E5M2: [1 sign][5 exp][2 mantissa]  范围 ±57344，精度较低
  用于：反向传播的梯度
  类比：粗略的估算器（梯度本身就是近似值，不需要太精确）

数值范围对比：
  FP32:  ±3.4×10³⁸  （太大了，浪费）
  FP16:  ±65504      （训练够用）
  E4M3:  ±448        （前向够用，需要缩放）
  E5M2:  ±57344      （梯度范围足够）
```

#### Transformer Engine——"自动挡精度管理"

```
问题：FP8 范围太小，直接使用会溢出
解决：Transformer Engine 自动管理缩放因子

工作流程：
  1. 监测每层的激活值/梯度范围
  2. 动态计算缩放因子 (scaling factor)
  3. 在 FP8 ↔ FP16 之间自动切换
  
  ┌─── Layer N ────────────────────────────────────────┐
  │  输入 FP16 → 检查范围 → 可以 FP8? → FP8 计算     │
  │                           │                         │
  │                           └→ 不行 → FP16 计算      │
  │  → 输出 FP16                                        │
  └────────────────────────────────────────────────────┘
  
  对用户透明：只需一行 with te.fp8_autocast()
  效果：接近 FP8 的速度，接近 FP16 的精度
```

### 关键规格 (H100 SXM)

| 指标 | 数值 | vs A100 |
|------|------|---------|
| SM 数量 | 132 | 1.2x |
| CUDA Cores | 16,896 | 2.4x |
| Tensor Cores | 528 | 1.2x |
| FP32 峰值 | 67 TFLOPS | 3.4x |
| TF32 峰值 | 989 TFLOPS | 6.3x |
| FP16 峰值 (Tensor) | 1,979 TFLOPS | 6.3x |

| 显存 | 80 GB HBM3 | 同 |

| TDP | 700W | 1.75x |

## Blackwell 架构 (2024)

### 核心创新

#### 1. 双芯片封装——突破物理极限

```
问题：芯片越做越大，但光刻机有尺寸极限（光罩约 800mm²）
解决：把两颗芯片焊在一起，当一颗用

类比：一栋楼盖不了那么高？两栋楼建个空中连廊连起来！

┌─────────────────────────────────────────────────┐
│                 B200 双芯封装                      │
│                                                   │
│  ┌──────────────┐  10 TB/s  ┌──────────────┐    │
│  │   Die 0      │ ←══════→ │   Die 1      │    │
│  │  ~1040亿     │  芯片间   │  ~1040亿     │    │
│  │  晶体管      │  互联     │  晶体管      │    │
│  └──────────────┘           └──────────────┘    │
│                                                   │
│  总计：2080 亿晶体管（H100 是 800 亿）            │
│  对软件完全透明——看起来就是一颗芯片                │
└─────────────────────────────────────────────────┘
```

#### 2. FP4 支持——极致压缩

```
FP8 (8 bits): 每秒处理 X tokens
FP4 (4 bits): 每秒处理 2X tokens（带宽翻倍）

但 4 位只能表示 16 个不同的值！
→ 需要配合"微张量缩放" (Micro-Tensor Scaling)
→ 每小组参数有独立的缩放因子，扩展有效范围
```

#### 3. NVLink 5.0

```
NVLink 4.0 (H100): 900 GB/s，支持 8 GPU 互联
NVLink 5.0 (B200):  1.8 TB/s，支持 72 GPU 互联！

72 GPU 通过 NVLink 直连：
  显存池 = 72 × 192 GB = 13.8 TB 的统一内存空间
  足以存放万亿参数模型的完整状态
```

### 关键规格 (B200)

| 指标 | 数值 | vs H100 |
|------|------|---------|
| 晶体管 | 2080 亿 | 2.6x |
| FP32 峰值 | 90 TFLOPS | 1.3x |
| FP16 峰值 | 2,250 TFLOPS | 1.1x |
| FP8 峰值 | 4,500 TFLOPS | 1.1x |
| FP4 峰值 | 9,000 TFLOPS | 新增 |
| 显存 | 192 GB HBM3e | 2.4x |
| 带宽 | 8 TB/s | 2.4x |
| NVLink 带宽 | 1.8 TB/s | 2x |
| TDP | 1000W | 1.4x |

### 性能一览表

| 指标 | V100 | A100 | H100 | B200 |
|------|------|------|------|------|
| **架构** | Volta | Ampere | Hopper | Blackwell |
| **年份** | 2017 | 2020 | 2022 | 2024 |
| **FP16 TFLOPS** (dense) | 125 | 312 | 989.5 | 2,250 |
| **FP16 TFLOPS** (稀疏) | - | 624 | 1,979 | 4,500 |
| **FP8 TFLOPS** (dense) | - | - | 1,979 | 4,500 |
| **FP8 TFLOPS** (稀疏) | - | - | 3,958 | 9,000 |
| **显存** | 32GB | 80GB | 80GB | 192GB |
| **带宽** | 900 GB/s | 2 TB/s | 3.35 TB/s | 8 TB/s |
| **TDP** | 300W | 400W | 700W | 1000W |

### 性能飞跃的四大驱动力

```
1. Tensor Core 迭代
   V100: 4×4 FP16 → A100: 加 TF32/BF16 → H100: 加 FP8 → B200: 加 FP4
   每代新增更低精度 → 同样硬件面积翻倍算力

2. 内存带宽提升 (HBM 演进)
   HBM2 (900GB/s) → HBM2e (2TB/s) → HBM3 (3.35TB/s) → HBM3e (8TB/s)
   7 年增长 ~9x，是 LLM 推理吞吐提升的关键

3. 互联带宽增长 (NVLink 演进)
   NVLink 2.0 (300GB/s) → 3.0 (600) → 4.0 (900) → 5.0 (1800)
   GPU 间通信加速 → 分布式训练效率提升

4. 低精度数据格式
   训练：FP32 → FP16 → BF16 → FP8 → FP4 (推理)
   每降一个精度 → 带宽/算力理论翻倍
   关键挑战：如何在低精度下保持模型精度
```

### 能效比演进

```
性能/功耗比 (FP16 dense TFLOPS/Watt):
  V100:  125/300   = 0.42 TFLOPS/W
  A100:  312/400   = 0.78 TFLOPS/W   (1.9x vs V100)
  H100:  989.5/700 = 1.41 TFLOPS/W   (1.8x vs A100)
  B200:  2250/1000 = 2.25 TFLOPS/W   (1.6x vs H100)

结论：GPU 性能增长远快于功耗增长（摩尔定律的体现）
但 B200 功耗已达 1000W → 散热成为数据中心重大挑战
  → 液冷正在成为高端 GPU 集群的标配
```

## 架构选型建议

| 场景 | 推荐架构 | 理由 |
|------|----------|------|
| 预算有限/入门 | V100 (二手) | 性价比高，足够学习和小实验 |
| 主流训练推理 | A100 80GB | 成熟稳定，生态完善，二手价格合理 |
| 大模型训练 | H100 | 最强性能，Transformer Engine 真正有用 |
| 超大模型 (>100B) | B200 | 192GB 显存，8TB/s 带宽 |
| 推理服务 | L40S / H100 | 取决于模型大小和延迟要求 |
| 研究实验 | A100 或 H100 | 平衡性能和成本 |
