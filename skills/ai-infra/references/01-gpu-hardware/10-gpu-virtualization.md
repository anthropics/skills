## 为什么需要 GPU 虚拟化

### 生活类比：酒店房间管理

```
一张 H100 就像一个大型会议厅：

  没有虚拟化:
    "整个会议厅给你用一天" → 开个小会也占一整天
    GPU 利用率常常只有 10-30%
    其他人在门外排队等候

  时间分片:
    "上午你用，下午他用" → 轮流使用，排队等候
    不能同时使用，切换有开销

  MIG (空间分片):
    "把会议厅隔成 7 个小会议室" → 同时开 7 个会
    每个小会议室独立使用，互不干扰

  vGPU:
    "每个人拿到一个虚拟会议室" → 看起来像有自己的专属会议厅
    底层共享资源，上层感知不到

  目标: 让一张 GPU 卡像多张小 GPU 卡一样使用
```

## NVIDIA MIG 深入

### MIG 是什么

MIG (Multi-Instance GPU) 是 NVIDIA 在 Ampere 架构引入的硬件级 GPU 分割技术。

```
┌─────────────────────────────────────────────────────────────┐
│                 MIG 工作原理 (以 A100 为例)                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  完整 A100 80GB:                                             │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ GPC0 │ GPC1 │ GPC2 │ GPC3 │ GPC4 │ GPC5 │ GPC6 │   │    │
│  ├──────┴──────┴──────┴──────┴──────┴──────┴──────┤   │    │
│  │              80 GB HBM2e                        │   │    │
│  └─────────────────────────────────────────────────┘   │    │
│                                                             │
│  MIG 模式 (7 个实例):                                        │
│  ┌──────┐┌──────┐┌──────┐┌──────┐┌──────┐┌──────┐┌──────┐  │
│  │1g.10g││1g.10g││1g.10g││1g.10g││1g.10g││1g.10g││1g.10g│  │
│  │GPC0  ││GPC1  ││GPC2  ││GPC3  ││GPC4  ││GPC5  ││GPC6  │  │

│  ┌────────────────┐┌────────────────┐┌──────┐┌──────┐       │
```

### MIG 配置实操

```bash
# 启用 MIG 模式
sudo nvidia-smi -i 0 -mig 1

# 重启 GPU (需要)
sudo nvidia-smi -i 0 -r

# 查看支持的 MIG 配置
nvidia-smi mig -lgip

# 创建 GPU 实例 (GI)
# 方式 1: 7 个 1g.10gb 实例
nvidia-smi mig -cgi 19,19,19,19,19,19,19 -i 0

# 方式 2: 混合配置 (1×3g.40gb + 2×2g.20gb)
nvidia-smi mig -cgi 9,14,14 -i 0

# 创建计算实例 (CI) - 每个 GI 内必须创建 CI 才能使用
nvidia-smi mig -cci -i 0

# 查看 MIG 设备
nvidia-smi mig -lgi  # 列出 GPU 实例
nvidia-smi mig -lci  # 列出计算实例

# 在特定 MIG 实例上运行任务
CUDA_VISIBLE_DEVICES=MIG-GPU-xxxxx-x-x/0/0 python train.py
```

### A100 MIG 配置选项

| Profile | GPC | 显存 | SM 数 | 典型用途 |
|---------|-----|------|-------|---------|
| 1g.5gb / 1g.10gb | 1 | 5/10 GB | 14 | 小模型推理、开发调试 |
| 2g.10gb / 2g.20gb | 2 | 10/20 GB | 28 | 中等推理任务 |
| 3g.20gb / 3g.40gb | 3 | 20/40 GB | 42 | 7B 模型推理、微调 |
| 4g.40gb | 4 | 40 GB | 56 | 较大模型推理 |
| 7g.80gb | 7 | 80 GB | 98 | 全部资源 |

```
  4. Confidential Computing 支持 (安全隔离)
  
  H100 MIG 配置:
  ┌─────────────────────────────────────────────┐
  │  Profile      │ GPC │ 显存   │ 适用         │

```

```
│                 NVIDIA vGPU 架构                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │

│  │   VM 1   │ │   VM 2   │ │   VM 3   │                     │
│  │ ┌──────┐ │ │ ┌──────┐ │ │ ┌──────┐ │                     │
│  │ │vGPU  │ │ │ │vGPU  │ │ │ │vGPU  │ │                     │
│  │ │Driver│ │ │ │Driver│ │ │ │Driver│ │                     │
│  │ └──────┘ │ │ └──────┘ │ │ └──────┘ │                     │
│  └────┬─────┘ └────┬─────┘ └────┬─────┘                     │
│       │            │            │                            │
│  ─────┴────────────┴────────────┴──── vGPU Manager          │
│                     │                                        │
│  ┌──────────────────┴──────────────────┐                     │
│  │          Hypervisor (ESXi/KVM)       │                     │
│  └──────────────────┬──────────────────┘                     │
│                     │                                        │
│  ┌──────────────────┴──────────────────┐                     │
│  │          物理 GPU (A100/H100)        │                     │
│  └─────────────────────────────────────┘                     │
│                                                             │
│  vGPU 类型:                                                  │
│  - C-Series: 计算型（AI/HPC）                                │
│  - Q-Series: 图形型（设计/CAD）                              │
│  - B-Series: 通用型（VDI 虚拟桌面）                          │
│                                                             │
│  需要 vGPU 许可证 (年费)                                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### vGPU vs MIG 对比

| 特性 | MIG | vGPU |
|------|-----|------|
| **硬件支持** | A100/H100 | 大部分数据中心 GPU |
| **分割粒度** | 固定 (1/2/3/4/7) | 灵活 (任意比例) |
| **Hypervisor** | 不需要 | 需要 (ESXi/KVM) |
| **许可证** | 免费 | 需要付费许可 |
| **性能损失** | 几乎无 | 5-10% (虚拟化开销) |
| **迁移支持** | 不支持 | 支持 vMotion |
| **适用场景** | K8s 多租户 | 虚拟化环境/VDI |
| **MIG + vGPU** | 可以结合 | MIG 内再做 vGPU |

## 时间分片 vs 空间分片

### 时间分片 (Time-Slicing)

```
┌─────────────────────────────────────────────────────────────┐
│                  时间分片工作原理                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  GPU 时间轴:                                                 │
│  ┌─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┐         │
│  │ T1  │ T2  │ T3  │ T1  │ T2  │ T3  │ T1  │ T2  │         │
│  └─────┴─────┴─────┴─────┴─────┴─────┴─────┴─────┘         │
│                                                             │
│  3 个任务轮流使用 GPU，每个时间片 ~1ms                        │
│                                                             │
│  优点:                                                       │
│  + 所有 GPU 都支持，不需要特殊硬件                            │
│  + 配置简单，Kubernetes 原生支持                              │
│  + 每个任务都能访问全部显存（理论上）                          │
│                                                             │
│  缺点:                                                       │
│  - 无显存隔离 → 一个任务 OOM 可能影响其他任务                 │
│  - 上下文切换开销 → 频繁切换降低吞吐                          │
│  - 无 QoS 保证 → 任务间性能干扰                              │
│  - 无故障隔离 → 一个任务崩溃可能影响其他                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 空间分片 (MIG)

```
┌─────────────────────────────────────────────────────────────┐
│                  空间分片工作原理 (MIG)                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  GPU 资源:                                                   │
│  ┌──────────┬──────────┬──────────┐                         │
│  │ Slice 1  │ Slice 2  │ Slice 3  │   同时运行              │
│  │ Task 1   │ Task 2   │ Task 3   │   互不干扰              │
│  │ 独立SM   │ 独立SM   │ 独立SM   │                         │
│  │ 独立显存 │ 独立显存 │ 独立显存 │                         │
│  │ 独立缓存 │ 独立缓存 │ 独立缓存 │                         │
│  └──────────┴──────────┴──────────┘                         │
│                                                             │
│  优点:                                                       │
│  + 硬件级隔离 → 一个任务崩溃不影响其他                        │
│  + 显存隔离 → 每个实例有独立的显存空间                        │
│  + QoS 保证 → 算力不会被其他任务抢占                          │
│  + 独立错误报告 → 每个实例独立的 ECC 错误计数                 │
│                                                             │
│  缺点:                                                       │
│  - 只有 A100/H100/B200 支持                                  │
│  - 分割粒度固定（不能任意比例）                                │
│  - 重新配置需要停止所有实例                                   │
│  - 单个实例无法使用全部 NVLink 带宽                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## GPU 共享方案

### NVIDIA MPS (Multi-Process Service)

```
┌─────────────────────────────────────────────────────────────┐
│                   MPS 工作原理                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  没有 MPS:                                                   │
│    进程 1 → CUDA Context 1 ──→ ┌─────┐                     │
│    进程 2 → CUDA Context 2 ──→ │ GPU │ ← 上下文切换         │
│    进程 3 → CUDA Context 3 ──→ └─────┘                     │
│    每个进程独立的 CUDA Context，轮流调度                      │
│                                                             │
│  使用 MPS:                                                   │
│    进程 1 ──→ ┌─────────────┐ ──→ ┌─────┐                  │
│    进程 2 ──→ │  MPS Server  │ ──→ │ GPU │ ← 并行执行      │
│    进程 3 ──→ └─────────────┘ ──→ └─────┘                  │
│    共享单个 CUDA Context，减少开销，提高 SM 利用率            │
│                                                             │
│  适用场景:                                                   │
│  - 多个小模型推理同时服务                                     │
│  - 单个推理任务用不满 GPU 算力                                │
│  - Volta 及以上架构                                          │
│                                                             │
│  限制:                                                       │
│  - 无显存隔离                                                │
│  - 无故障隔离（一个进程崩溃可能影响所有）                      │
│  - 不适合训练任务                                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```bash
# 启动 MPS
export CUDA_VISIBLE_DEVICES=0
nvidia-cuda-mps-control -d  # 后台启动 MPS daemon

# 设置 MPS 线程百分比限制
export CUDA_MPS_ACTIVE_THREAD_PERCENTAGE=33  # 每个客户端最多用 33% SM

# 运行多个推理服务
python inference_service_1.py &  # 自动通过 MPS 共享
python inference_service_2.py &
python inference_service_3.py &

# 停止 MPS
echo quit | nvidia-cuda-mps-control
```

### GPU Operator + 时间分片

```yaml
# Kubernetes GPU 时间分片配置
apiVersion: v1
kind: ConfigMap
metadata:
  name: time-slicing-config
  namespace: gpu-operator
data:
  any: |-
    version: v1
    sharing:
      timeSlicing:
        renameByDefault: false
        failRequestsGreaterThanOne: false
        resources:
          - name: nvidia.com/gpu
            replicas: 4  # 每张 GPU 虚拟成 4 份

---
# Pod 请求 GPU（实际是时间分片的 1/4）
apiVersion: v1
kind: Pod
metadata:
  name: inference-pod
spec:
  containers:
    - name: inference
      image: my-inference:latest
      resources:
        limits:
          nvidia.com/gpu: 1  # 请求 1 个虚拟 GPU
```

## 多租户隔离策略

### 隔离等级

```
┌─────────────────────────────────────────────────────────────┐
│                多租户 GPU 隔离等级                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Level 0: 无隔离（默认）                                     │
│    多个 Pod 申请同一 GPU，时间分片                             │
│    显存不隔离，互相可见                                       │
│    适合：内部开发团队，互相信任                                │
│                                                             │
│  Level 1: 软隔离（MPS + 资源限制）                           │
│    通过 MPS 控制算力百分比                                    │
│    通过 cgroup 限制显存（不完全可靠）                          │
│    适合：同一团队不同项目                                     │
│                                                             │
│  Level 2: 硬隔离（MIG）                                      │
│    硬件级别的算力 + 显存 + 缓存隔离                           │
│    独立的故障域                                              │
│    适合：不同团队共享 GPU 集群                                │
│                                                             │
│  Level 3: 完全隔离（vGPU + 虚拟化）                          │
│    每个租户在独立虚拟机中                                     │
│    Hypervisor 级别隔离                                       │
│    适合：多客户云服务、安全要求高                              │
│                                                             │
│  Level 4: 机密计算（MIG + Confidential Computing）           │
│    H100 支持的安全飞地                                       │
│    数据在使用中也被加密保护                                   │
│    适合：金融、医疗等敏感数据                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 推荐方案

```
场景 → 推荐方案:

  开发环境 (10+ 开发者共享 GPU):
    → 时间分片 + HAMi 显存限制
    理由: 灵活，成本低，开发者不需要完整 GPU

  推理服务 (多模型共享):
    → MIG (如果 A100/H100) 或 MPS (其他 GPU)
    理由: QoS 保证，模型间互不影响

  多租户云 (不同客户):
    → MIG + vGPU 或 MIG + Confidential Computing
    理由: 安全隔离必须保证

  训练集群 (大任务独占):
    → 不分割，整卡分配
    理由: 训练需要全部资源，分割降低效率
```

## 选型建议

### 决策流程

```
你需要 GPU 共享吗？
│
├── 训练任务 (需要全部资源)
│   └── 不分割，整卡分配
│       使用 K8s 调度器做任务排队
│
├── 推理/开发 (可以共享)
│   │
│   ├── GPU 型号支持 MIG？(A100/H100/B200)
│   │   ├── Yes → 用 MIG
│   │   │   强隔离、性能稳定、运维简单
│   │   │
│   │   └── No → 用时间分片 + MPS
│   │       设置 MPS 线程限制
│   │       配合 HAMi 做显存管理
│   │
│   ├── 需要虚拟化环境？
│   │   └── Yes → vGPU (GRID)
│   │       需要购买许可证
│   │
│   └── 需要细粒度显存控制？
│       └── Yes → HAMi
│           支持任意显存大小分配
│
└── 混合场景
    └── MIG 混合配置
        大实例给训练/大推理
        小实例给开发/小推理
```

## 实战练习

### 练习：MIG 配置与测试

```bash
#!/bin/bash
# MIG 快速体验脚本（需要 A100/H100）

# 1. 检查 MIG 支持
nvidia-smi --query-gpu=mig.mode.current --format=csv,noheader

# 2. 启用 MIG
sudo nvidia-smi -i 0 -mig 1

# 3. 创建 3 个实例 (2g.20gb × 3)
sudo nvidia-smi mig -i 0 -cgi 14,14,14

# 4. 创建计算实例
sudo nvidia-smi mig -i 0 -cci

# 5. 查看结果
nvidia-smi

# 6. 在不同实例上运行任务
# 获取 MIG UUID
nvidia-smi -L

# MIG 实例 0
CUDA_VISIBLE_DEVICES=MIG-xxx-0 python -c "
import torch
print(f'Device: {torch.cuda.get_device_name(0)}')
print(f'Memory: {torch.cuda.get_device_properties(0).total_memory / 1024**3:.1f} GB')
"

# 7. 清理
sudo nvidia-smi mig -i 0 -dci
sudo nvidia-smi mig -i 0 -dgi
sudo nvidia-smi -i 0 -mig 0
```

## 小结

```
GPU 虚拟化核心要点:

1. 目标是提高利用率
   推理和开发场景，GPU 利用率常常不到 30%
   虚拟化让多个任务共享，利用率提升到 70%+

2. MIG 是最佳选择（如果硬件支持）
   硬件级隔离、零性能损失、运维简单
   但只有 A100/H100/B200 支持

3. 时间分片是通用方案
   所有 GPU 都支持
   但隔离性差，适合内部低安全要求场景

4. Kubernetes 生态成熟
   GPU Operator + Device Plugin + MIG Manager
   配合 HAMi 实现更细粒度管理

5. 训练不要分割
   训练任务需要全部 GPU 资源
   分割只会降低效率
```
