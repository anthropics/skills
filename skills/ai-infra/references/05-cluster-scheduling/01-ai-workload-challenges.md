## 传统工作负载 vs AI 工作负载

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    AI 工作负载的特殊性                                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   传统 Web 服务                         AI 训练任务                         │
│   ─────────────                         ──────────                         │
│   • 长时间运行                           • 批处理，有明确结束时间            │
│   • 资源需求稳定                         • 资源需求变化大                   │
│   • 可随时弹性伸缩                       • 需要 Gang Scheduling            │
│   • CPU 密集                             • GPU 密集                        │
│   • 无状态/状态可外置                    • 有状态 (checkpoint)             │
│   • 故障重启即可                         • 需要从 checkpoint 恢复          │
│                                                                             │
│   推理服务                                                                  │
│   ─────────────                                                             │
│   • 介于两者之间                                                            │
│   • 可长时间运行                                                            │
│   • GPU 需求稳定                                                            │
│   • 对延迟敏感                                                              │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 资源需求对比

| 特性 | Web 服务 | AI 训练 | AI 推理 |
|------|----------|---------|---------|
| **主要资源** | CPU | GPU | GPU |
| **运行时长** | 持续 | 小时-天 | 持续 |
| **弹性需求** | 高 | 低 | 中 |
| **故障恢复** | 重启 | Checkpoint 恢复 | 重启 |
| **调度方式** | 单 Pod | Gang Scheduling | 单 Pod |

## AI 调度的五大挑战

### 1. Gang Scheduling（组调度）

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                      Gang Scheduling 必要性                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   分布式训练: 需要 4 个 worker 同时启动                                     │
│                                                                             │
│   ❌ 默认调度器行为:                                                        │
│   ─────────────────                                                         │
│   Worker 0 ✅ → Worker 1 ✅ → Worker 2 ❌ (资源不足)                        │
│   → Worker 0, 1 空等 NCCL 初始化                                           │
│   → 最终超时失败                                                            │
│   → 浪费了 Worker 0, 1 的 GPU 资源                                         │
│                                                                             │
│   ✅ Gang Scheduling:                                                       │
│   ─────────────────                                                         │
│   1. 检查是否有足够资源容纳所有 4 个 worker                                 │
│   2. 有: 同时启动所有 worker                                               │
│   3. 无: 等待，不启动任何 worker (避免资源浪费和死锁)                       │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2. GPU 拓扑感知

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         GPU 拓扑对通信的影响                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   8-GPU 节点拓扑示例 (DGX A100):                                           │
│                                                                             │
│   ┌─────────────────────────────────────────────────────────────────────┐   │
│   │        NVSwitch                    NVSwitch                        │   │
│   │           │                            │                           │   │
│   │   ┌───────┼───────┐            ┌───────┼───────┐                   │   │
│   │   │       │       │            │       │       │                   │   │
│   │ GPU0   GPU1   GPU2   GPU3    GPU4   GPU5   GPU6   GPU7            │   │
│   │   └───────┴───────┘            └───────┴───────┘                   │   │
│   │        组 A                         组 B                           │   │
│   │                                                                     │   │
│   └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│   调度策略:                                                                 │
│   • 4-GPU 任务: 优先分配同组 (0,1,2,3) 或 (4,5,6,7)                       │
│   • 8-GPU 任务: 分配整个节点                                               │
│   • 跨节点任务: 考虑网络拓扑，尽量同机架                                    │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 资源碎片化

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         GPU 资源碎片化问题                                    │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   场景: 集群有 3 个节点，每节点 8 GPU                                       │
│                                                                             │
│   Node 1: [█][█][█][█][█][█][ ][ ]   使用 6/8                             │
│   Node 2: [█][█][█][█][ ][ ][ ][ ]   使用 4/8                             │
│   Node 3: [█][█][ ][ ][ ][ ][ ][ ]   使用 2/8                             │
│                                                                             │
│   总计: 使用 12/24 GPU，空闲 12 GPU                                        │
│                                                                             │
│   问题: 新任务需要 8 GPU (单节点)                                          │
│   ─────────────────────────────────────────                                │
│   • 虽然总共有 12 个空闲 GPU                                               │
│   • 但没有任何单节点有 8 个连续空闲                                        │
│   • 任务无法调度！                                                          │
│                                                                             │
│   解决方案:                                                                 │
│   • 碎片整理 (任务迁移)                                                     │
│   • 弹性任务 (可跨节点)                                                     │
│   • MIG 支持更小粒度                                                        │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 4. 抢占与恢复

- **高优先级任务**需要抢占低优先级任务的 GPU
- **低优先级任务**被抢占时需要保存 checkpoint
- 恢复时从 checkpoint 继续训练

### 5. 公平性

- 多团队共享 GPU 集群
- 需要保证各团队的资源配额
- 空闲资源可以借用，但不影响配额

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         调度系统演进                                          │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   传统 HPC                  Kubernetes + 扩展         云原生 AI 平台        │
│   ───────────               ─────────────────         ──────────────        │
│   Slurm/PBS                 K8s + Volcano/Yunikorn    Kubeflow/Ray          │
│                                                                             │
│   特点:                     特点:                     特点:                 │
│   • 专为批处理设计          • 通用容器编排            • AI 原生             │

│                                                                             │
│   适用场景:                 适用场景:                 适用场景:             │
│   • 传统超算中心            • 企业混合负载            • 云原生团队          │

└─────────────────────────────────────────────────────────────────────────────┘
```

### 大多数企业的选择

**Kubernetes + 批调度器扩展** 是目前的主流选择：

- 复用现有 K8s 基础设施
- 灵活性高，可与其他服务混合部署
- Volcano/Yunikorn 补充 AI 调度能力
