## AI 存储需求分析

```
┌─────────────────────────────────────────────────────────────┐
│              AI 工作负载的存储需求                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  工作负载      数据量        IO 模式        性能要求           │
│  ─────────    ─────────    ─────────     ─────────          │
│  训练数据     TB - PB      顺序读取       高吞吐              │
│  Checkpoint   GB - TB      突发写入       高吞吐+低延迟       │
│  模型权重     GB - 百GB    一次性加载      高吞吐              │
│  日志/指标    GB            追加写入       中等吞吐            │
│  数据预处理   TB            随机读写       高 IOPS            │
│                                                             │
│  挑战:                                                       │
│  1. Checkpoint 风暴: 256 GPU 同时写 Checkpoint              │
│     → 70B 模型 = 280GB/次，每 30 分钟一次                   │
│     → 存储需要 280GB/30s ≈ 9.3 GB/s 持续写入               │
│                                                             │
│  2. 数据加载: 256 GPU 同时读训练数据                         │
│     → 每 GPU 需要 ~1 GB/s 读取带宽                          │
│     → 总共需要 256 GB/s 读取带宽                             │
│                                                             │
│  3. 模型加载: 集群重启时所有节点同时加载模型                  │
│     → 32 节点 × 140GB = 4.5 TB 突发读取                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 分布式文件系统

### 主流方案对比

| 文件系统 | 类型 | 吞吐 | 适用规模 | 成本 | 特点 |
|---------|------|------|---------|------|------|
| **Lustre** | 并行 | 极高 | 大型 HPC | 高 | HPC 标配，成熟稳定 |
| **GPFS/Spectrum Scale** | 并行 | 极高 | 企业级 | 高 | IBM，强一致性 |
| **BeeGFS** | 并行 | 高 | 中大型 | 中 | 易部署，开源 |
| **JuiceFS** | FUSE | 高 | 弹性 | 低 | 云原生，对象存储后端 |
| **CephFS** | 分布式 | 中高 | 中大型 | 中 | 统一存储（块+文件+对象） |
| **NFS** | 网络 | 中 | 小型 | 低 | 简单，小规模够用 |

### JuiceFS（云原生推荐）

```
┌─────────────────────────────────────────────────────────────┐
│              JuiceFS 架构                                     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────────────────────────────────┐                │
│  │            JuiceFS Client               │                │
│  │         (FUSE / CSI Driver)             │                │
│  └─────────┬───────────────┬───────────────┘                │
│            │               │                                │
│   ┌────────▼────────┐ ┌───▼───────────────┐                │
│   │ Metadata Engine │ │ Data Storage      │                │
│   │ (Redis/TiKV/    │ │ (S3/MinIO/OSS/    │                │
│   │  MySQL/PostgreSQL│ │  GCS/本地磁盘)    │                │
│   └─────────────────┘ └───────────────────┘                │
│                                                             │
│  优势:                                                       │
│  - POSIX 兼容 → 无需改代码                                  │
│  - 元数据和数据分离 → 弹性扩展                               │

│  - 本地缓存 → 加速热数据访问                                 │
│                                                             │
│  适用: 中小规模 AI 集群，云上/混合云环境                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## Checkpoint 存储加速

```
┌─────────────────────────────────────────────────────────────┐
│              Checkpoint 存储优化策略                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 异步 Checkpoint                                          │
│     训练不暂停，后台线程写 Checkpoint                         │
│     PyTorch: torch.save 在后台进程                           │
│     DeepSpeed: async_checkpoint                              │
│                                                             │
│  2. 分布式 Checkpoint                                        │
│     每个 rank 保存自己的分片，不汇总到一个文件                 │
│     PyTorch DCP (Distributed Checkpoint)                     │
│     写入速度提升 N 倍 (N = GPU 数)                           │
│                                                             │
│  3. 增量 Checkpoint                                          │
│     只保存与上次不同的部分                                    │
│     适合优化器状态（每步变化小）                               │
│                                                             │
│  4. NVMe 缓存                                                │
│     先写本地 NVMe SSD → 后台同步到共享存储                   │
│     本地 NVMe: ~6 GB/s 写入                                  │
│     共享存储: ~1 GB/s 写入                                   │
│     → 训练几乎不停顿                                        │
│                                                             │
│  5. GPUDirect Storage                                        │
│     GPU 显存直接写入 NVMe，绕过 CPU                           │
│     减少 Checkpoint 保存时间 50%+                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 对象存储

```
对象存储在 AI 中的角色:

  训练数据长期存储:
    S3/MinIO/GCS → 成本低，容量无限
    适合 PB 级训练数据

  模型仓库:
    HuggingFace Hub / S3 → 模型文件分发
    → 下载到本地 NVMe 缓存后使用

  Checkpoint 归档:
    热 Checkpoint: 本地 NVMe + 共享存储
    温 Checkpoint: 对象存储 (S3 Standard)
    冷 Checkpoint: 归档存储 (S3 Glacier)

  注意:
    对象存储延迟高 (~10ms) → 不适合训练时直接读取
    需要配合本地缓存使用
```

## 存储架构设计

```
推荐的 AI 集群存储架构:

┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  Layer 1: 本地 NVMe (最快)                                   │
│    容量: 每节点 2-8 TB                                       │
│    用途: Checkpoint 缓存、数据缓存、临时文件                  │
│    性能: 6-14 GB/s                                           │
│                                                             │
│  Layer 2: 共享并行文件系统 (高吞吐)                          │
│    容量: 100 TB - 1 PB                                       │
│    用途: 训练数据、活跃 Checkpoint、模型权重                  │
│    性能: 10-100+ GB/s (并行)                                 │
│    选型: Lustre / GPFS / BeeGFS / JuiceFS                    │
│                                                             │
│  Layer 3: 对象存储 (大容量低成本)                             │
│    容量: PB 级                                               │
│    用途: 数据归档、Checkpoint 归档、模型仓库                  │
│    性能: 1-10 GB/s                                           │
│    选型: S3 / MinIO / GCS                                    │
│                                                             │
│  数据流:                                                     │
│    训练数据: L3 → L2 (预加载) → L1 (缓存热数据)             │
│    Checkpoint: GPU → L1 → L2 → L3 (异步归档)                │
│    模型加载: L3 → L2 → L1 → GPU                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 小结

```
AI 存储核心要点:

1. Checkpoint 是最大的 IO 挑战
   大模型 Checkpoint 几百 GB
   异步 + 分布式 + 本地缓存是标配

2. 分层存储是最佳实践
   NVMe (快) → 并行FS (大) → 对象存储 (廉价)
   数据按热度自动分层

3. 中小规模推荐 JuiceFS
   云原生、POSIX 兼容、弹性扩展
   大规模推荐 Lustre/GPFS

4. 存储带宽需要和 GPU 匹配
   256 GPU 集群至少需要 50-100 GB/s 存储带宽
   存储投资通常占集群总成本 10-15%
```
