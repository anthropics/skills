## 存算分离 vs 存算一体

### 生活类比：餐厅的两种模式

```
存算一体 = 路边摊:
  厨师 + 食材 + 灶台都在一起
  ✓ 食材随手就拿（低延迟）
  ✗ 搬家就要全部带走（扩展困难）
  ✗ 灶台不用时食材也占地方（资源浪费）

存算分离 = 中央厨房 + 门店:
  中央厨房（存储集群）+ 门店（计算节点）
  ✓ 门店可以随意开关（弹性计算）
  ✓ 中央厨房统一管理（统一存储）
  ✗ 需要配送系统（网络带宽）
  ✗ 配送有延迟（存储访问延迟）
```

### 架构对比

```
┌─────────────────────────────────────────────────────────────────┐
│              存算一体 vs 存算分离                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  存算一体 (Converged):                                           │
│  ┌──────────────────────┐                                       │
│  │  计算节点 1           │                                       │
│  │  ┌───────┐ ┌───────┐ │                                       │
│  │  │ GPU×8 │ │ NVMe  │ │  每个节点既计算又存储                  │
│  │  └───────┘ │ 存储  │ │  数据在本地，访问最快                  │
│  │            └───────┘ │                                       │
│  └──────────────────────┘                                       │
│  适用: 固定规模集群、HPC 传统架构                                │
│                                                                 │
│  存算分离 (Disaggregated):                                       │
│  ┌──────────┐     ┌──────────────────┐                          │
│  │ 计算节点  │     │   存储集群        │                          │
│  │ GPU×8    │←───→│ Lustre/GPFS/     │                          │
│  │ (无本地   │ 网络 │ JuiceFS/S3      │                          │
│  │  存储)   │     │                  │                          │
│  └──────────┘     └──────────────────┘                          │
│  适用: 云环境、弹性集群、多租户                                   │
│                                                                 │
│  混合架构 (Hybrid，推荐):                                        │
│  ┌──────────────────────┐     ┌──────────────┐                  │
│  │  计算节点             │     │  共享存储      │                  │
│  │  ┌───────┐ ┌───────┐ │     │  Lustre/     │                  │
│  │  │ GPU×8 │ │ NVMe  │ │←───→│  JuiceFS     │                  │
│  │  └───────┘ │ 缓存  │ │ 网络 │              │                  │
│  │            └───────┘ │     └──────┬───────┘                  │
│  └──────────────────────┘            │                          │
│                                ┌─────▼───────┐                  │
│                                │  对象存储     │                  │
│                                │  S3/MinIO    │                  │
│                                └─────────────┘                  │
│  本地 NVMe 做缓存 + 共享存储做持久化 + 对象存储做归档             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 对比总结

| 维度 | 存算一体 | 存算分离 | 混合架构 |
|------|---------|---------|---------|
| **IO 延迟** | 最低 | 较高 | 低(缓存命中) |
| **扩展性** | 差 | 好 | 好 |
| **成本效率** | 中 | 好 | 最优 |
| **运维** | 简单 | 中等 | 较复杂 |
| **弹性** | 差 | 优秀 | 好 |
| **适用** | 固定集群 | 云/弹性 | 通用推荐 |

## 分层存储架构

### 三层存储模型

```
┌─────────────────────────────────────────────────────────────────┐
│              AI 集群三层存储架构                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─── Layer 1: 本地 NVMe (最快，容量有限) ────────────────┐      │
│  │  位置: 每个计算节点内部                                 │      │
│  │  容量: 2-15 TB/节点                                    │      │
│  │  性能: 6-14 GB/s 顺序，~1M IOPS                       │      │
│  │  用途:                                                  │      │
│  │    • Checkpoint 写入缓存（异步写入后同步到 L2）         │      │
│  │    • 训练数据热缓存（缓存当前 epoch 的数据）            │      │
│  │    • 模型权重本地副本                                   │      │
│  │    • KV Cache 卸载（推理场景）                          │      │
│  └─────────────────────────────────────────────────────────┘      │
│                          │ 异步同步                               │
│                          ▼                                       │
│  ┌─── Layer 2: 共享并行文件系统 (高吞吐，中容量) ────────┐       │
│  │  位置: 独立存储集群（或超融合）                         │       │
│  │  容量: 100 TB - 1 PB                                   │       │
│  │  性能: 10-300+ GB/s 聚合                               │       │
│  │  用途:                                                  │       │
│  │    • 训练数据主存储                                     │       │
│  │    • 活跃 Checkpoint 存储                               │       │
│  │    • 模型权重共享                                       │       │
│  │    • 多任务数据共享                                     │       │
│  └─────────────────────────────────────────────────────────┘      │
│                          │ 定期归档                               │
│                          ▼                                       │
│  ┌─── Layer 3: 对象存储 (大容量低成本) ──────────────────┐       │
│  │  位置: 云上 S3 或自建 MinIO                             │       │
│  │  容量: PB 级，几乎无限                                  │       │
│  │  性能: 1-50 GB/s                                        │       │
│  │  用途:                                                  │       │
│  │    • 原始数据归档                                       │       │
│  │    • Checkpoint 冷存储                                  │       │
│  │    • 模型分发仓库                                       │       │
│  │    • 合规数据保留                                       │       │
│  └─────────────────────────────────────────────────────────┘      │
│                                                                   │
│  数据流:                                                          │
│    训练数据: L3 → L2 (预加载) → L1 (缓存热数据)                  │
│    Checkpoint: GPU → L1 (即时) → L2 (异步) → L3 (归档)           │
│    模型加载: L3 → L1 (预缓存) → GPU (mmap加载)                   │
│                                                                   │
└───────────────────────────────────────────────────────────────────┘
```

## 缓存策略

### Alluxio 分布式缓存

```
Alluxio — AI 存储加速层:

  ┌──────────────────────────────────────────────┐
  │              计算节点 (GPU)                    │
  │                    │                          │
  │              ┌─────▼─────┐                    │
  │              │ Alluxio   │ ← 分布式缓存层      │
  │              │ Worker    │   (NVMe/RAM)        │
  │              └─────┬─────┘                    │
  │                    │                          │
  │       ┌────────────┼────────────┐             │
  │       │            │            │             │
  │  ┌────▼────┐ ┌─────▼─────┐ ┌───▼────┐       │
  │  │ Lustre  │ │   S3      │ │  HDFS  │       │
  │  │         │ │           │ │        │       │
  │  └─────────┘ └───────────┘ └────────┘       │
  └──────────────────────────────────────────────┘

  价值:
    热数据从 Alluxio 缓存读取 → 接近本地 NVMe 性能
    冷数据透传到底层存储 → 不影响访问
    自动缓存管理 → LRU/LFU 淘汰

  AI 场景:
    多个训练任务共享同一数据集 → 第一个任务加载后缓存
    多次 epoch 重复读同一数据 → 全部从缓存读取
    跨集群数据共享 → 避免重复下载
```

### 缓存命中率优化

```python
"""缓存效果分析工具"""

def analyze_cache_efficiency(
    dataset_size_tb: float,
    cache_size_tb: float,
    num_epochs: int,
    data_reuse_ratio: float = 0.8,
):
    """分析缓存对训练性能的影响"""
    
    # 无缓存：每个 epoch 都从存储读取
    total_read_no_cache = dataset_size_tb * num_epochs
    
    # 有缓存：第一个 epoch 读存储，后续从缓存读
    if cache_size_tb >= dataset_size_tb:
        # 缓存能放下全部数据
        total_read_storage = dataset_size_tb  # 首次加载
        total_read_cache = dataset_size_tb * (num_epochs - 1)
        cache_hit_rate = (num_epochs - 1) / num_epochs
    else:
        # 缓存放不下全部数据
        cached_ratio = cache_size_tb / dataset_size_tb
        total_read_storage = dataset_size_tb + dataset_size_tb * (1 - cached_ratio) * (num_epochs - 1)
        total_read_cache = dataset_size_tb * cached_ratio * (num_epochs - 1)
        cache_hit_rate = cached_ratio * (num_epochs - 1) / num_epochs
    
    storage_bandwidth_gbs = 50  # 共享存储 50 GB/s
    cache_bandwidth_gbs = 200   # NVMe 缓存 200 GB/s (多节点聚合)
    
    time_no_cache = total_read_no_cache * 1024 / storage_bandwidth_gbs
    time_with_cache = (total_read_storage * 1024 / storage_bandwidth_gbs +
                       total_read_cache * 1024 / cache_bandwidth_gbs)
    
    speedup = time_no_cache / time_with_cache
    
    print(f"数据集大小: {dataset_size_tb} TB, 缓存大小: {cache_size_tb} TB")
    print(f"训练 {num_epochs} 个 epoch")
    print(f"缓存命中率: {cache_hit_rate:.1%}")
    print(f"无缓存 IO 时间: {time_no_cache:.0f}s")
    print(f"有缓存 IO 时间: {time_with_cache:.0f}s")
    print(f"IO 加速: {speedup:.1f}x")

# 示例
analyze_cache_efficiency(
    dataset_size_tb=5,
    cache_size_tb=8,
    num_epochs=3,
)
```

## 生产级存储方案设计

### 方案 A：中小规模 (32-128 GPU)

```
┌─────────────────────────────────────────────────────┐
│       方案 A: 中小规模 AI 集群 (32-128 GPU)          │
├─────────────────────────────────────────────────────┤
│                                                     │
│  计算: 4-16 节点 × 8 GPU                             │
│                                                     │
│  L1 (本地): 每节点 2× NVMe 3.84TB = 7.68 TB         │
│  L2 (共享): JuiceFS + Redis + MinIO                  │
│             容量: 100 TB (MinIO 4 节点)              │
│  L3 (归档): 云 S3                                    │
│                                                     │
│  成本估算 (月):                                      │
│    本地 NVMe: 已含在 GPU 服务器中                    │
│    MinIO 服务器: 4× $2,000/月 = $8,000               │
│    云 S3 (100TB): $2,300                             │
│    总存储成本: ~$10,000/月                            │
│    占集群总成本: ~8-12%                              │
│                                                     │
│  特点: 部署简单，JuiceFS 运维成本低                   │
│                                                     │
└─────────────────────────────────────────────────────┘
```

## 云上存储架构

```
云上 AI 训练存储架构 (以 AWS 为例):

  ┌──────────────────────────────────────────────┐
  │  GPU 实例 (p5.48xlarge × N)                   │
  │  ┌────────────────────────────────────┐       │
  │  │  本地 NVMe Instance Store          │       │
  │  │  8× 3.84 TB = 30.72 TB            │       │
  │  │  → Checkpoint 缓存 + 数据缓存      │       │
  │  └──────────────┬─────────────────────┘       │
  │                 │                             │
  │        ┌────────▼────────┐                    │
  │        │ FSx for Lustre  │  ← 托管 Lustre     │
  │        │ 100+ GB/s       │    自动扩展         │
  │        │ POSIX 兼容      │    免运维           │
  │        └────────┬────────┘                    │
  │                 │                             │
  │           ┌─────▼──────┐                      │
  │           │    S3      │  ← 无限容量           │
  │           │ 训练数据    │    自动分层           │
  │           │ Checkpoint │    高可用             │
  │           │ 模型仓库    │                      │
  │           └────────────┘                      │
  └──────────────────────────────────────────────┘

  FSx for Lustre:
    自动与 S3 联动（S3 数据自动映射为 Lustre 文件）
    按需扩容，按使用量付费
    免去 Lustre 运维复杂度

  成本对比:
    自建 Lustre (500TB): ~$50,000/月
    FSx for Lustre (500TB): ~$75,000/月 (但免运维)
    决策: 规模大/长期用 → 自建；弹性/短期 → 托管
```

## 成本优化

### 成本优化策略

```
AI 存储成本优化 Top 10:

  1. 分层存储: 热/温/冷/冰 自动分层
     节省: 30-50% 存储成本

  2. Checkpoint 压缩: zstd 压缩 Checkpoint 文件
     节省: 20-40% 存储空间

  3. Checkpoint 清理: 只保留最近 N 个 + 里程碑
     节省: 50-80% Checkpoint 存储

  4. 增量 Checkpoint: 只保存变化部分
     节省: 70-90% 写入量

  5. 数据去重: 训练数据去重后存储
     节省: 10-30% 数据存储

  6. 模型共享: 基础模型共享只读挂载
     节省: 避免多份拷贝（可能节省 TB 级空间）

  7. 本地缓存: NVMe 缓存减少共享存储压力
     节省: 减少 50-70% 的共享存储带宽需求

  8. 右量存储: 定期审计，清理无用数据
     节省: 10-20% (往往积累大量无主数据)

  9. 预留/承诺折扣: 云存储长期承诺折扣
     节省: 20-40% (S3 Savings Plans)

  10. 冷数据归档: 超过 30 天不用的数据自动归档
      节省: Glacier 比 Standard 便宜 85%
```

## 容灾与高可用

```
AI 存储容灾设计:

  RPO (Recovery Point Objective): 能接受丢失多少数据？
    Checkpoint 间隔 30 分钟 → RPO = 30 分钟训练进度
    异步归档延迟 2 小时 → RPO(归档) = 2 小时

  RTO (Recovery Time Objective): 恢复需要多久？
    本地 NVMe 恢复: < 1 分钟
    共享存储恢复: < 5 分钟
    对象存储恢复: < 30 分钟
    Glacier 恢复: 数小时

  高可用策略:
  ┌──────────────────────────────────────────────┐
  │  Level 1: 磁盘级                              │
  │    RAID / 多副本 → 单盘故障不丢数据            │
  │                                              │
  │  Level 2: 节点级                              │
  │    Lustre OST 多副本 / GPFS 副本              │
  │    JuiceFS 依赖对象存储冗余                   │
  │                                              │
  │  Level 3: 站点级                              │
  │    跨 AZ 复制 (S3 Cross-Region Replication)   │
  │    GPFS AFM (Active File Management)          │
  │                                              │
  │  Level 4: 灾备                                │
  │    定期全量备份到异地                          │
  │    Checkpoint 多副本存储                       │
  └──────────────────────────────────────────────┘
```
