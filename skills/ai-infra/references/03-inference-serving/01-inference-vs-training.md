# 推理与训练的差异

> 理解推理与训练的本质区别，找到优化空间。

## 一句话理解

```
  训练 = 学生刷题 (反复练习，调整知识)
  推理 = 学生考试 (用已有知识快速答题)

  训练关注: "学得好不好？" → 优化 Loss
  推理关注: "答得快不快？" → 优化延迟/吞吐
```

## 关键差异深度对比

```
  ┌────────────────────────────────────────────────────────────────────────┐
  │                    训练 vs 推理 — 本质差异                             │
  ├──────────┬───────────────────────┬───────────────────────────────────┤
  │  维度    │  训练                 │  推理                              │
  ├──────────┼───────────────────────┼───────────────────────────────────┤
  │ 计算流程 │ 前向 + 反向传播       │ 仅前向                            │
  │ 优化目标 │ 高吞吐 (样本/秒)     │ 低延迟 + 高吞吐                   │
  │ Batch    │ 大 (数百-数千)        │ 小/动态                           │
  │ 精度     │ FP16/BF16            │ INT8/INT4/FP8                     │
  │ 显存     │ 参数+梯度+优化器+激活 │ 参数+KV Cache                     │
  │ 容错     │ 允许重试/回滚         │ 必须实时响应                      │
  │ 瓶颈     │ Compute-bound (算力) │ Memory-bound (带宽)               │
  │ 硬件策略 │ 尽量堆算力            │ 尽量堆带宽                        │
  └──────────┴───────────────────────┴───────────────────────────────────┘
```

### 为什么训练是 Compute-bound，推理是 Memory-bound？

```
  ┌──────────────────────────────────────────────────────────────────┐
  │                                                                  │
  │  训练: 大 Batch → 矩阵 × 矩阵 → 算力密集                       │
  │                                                                  │
  │    [4096, 4096] × [4096, 4096]  (batch 内多个样本并行)           │
  │    计算量: 2 × 4096³ ≈ 137 GFLOP                                │
  │    读取量: 2 × 4096² × 2 bytes ≈ 67 MB                          │
  │    计算强度: 137 GFLOP / 67 MB ≈ 2048 FLOP/Byte                 │
  │    → 远超 A100 的 208 FLOP/Byte 平衡点 → 算力瓶颈!             │
  │                                                                  │
  │  推理 (Decode): Batch=1 → 向量 × 矩阵 → 带宽密集                │
  │                                                                  │
  │    [1, 4096] × [4096, 4096]  (每次只处理 1 个 token)            │
  │    计算量: 2 × 4096² ≈ 33 MFLOP                                 │
  │    读取量: 4096² × 2 bytes ≈ 33 MB                               │
  │    计算强度: 33 MFLOP / 33 MB ≈ 1 FLOP/Byte                     │
  │    → 远低于平衡点 → 带宽瓶颈!                                    │
  │                                                                  │
  │  类比:                                                           │
  │    训练 = 工厂 24 小时满负荷生产 (机器是瓶颈)                    │
  │    推理 = 快递员送一个包裹 (路上跑的时间是瓶颈，不是装箱时间)    │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘
```

### 显存占用差异

```
  以 LLaMA-7B (FP16) 为例:

  训练显存:
    参数:     7B × 2 bytes = 14 GB
    梯度:     7B × 2 bytes = 14 GB
    优化器:   7B × 4 × 2 = 56 GB (AdamW: m + v 各 FP32)
    激活值:   ~10-20 GB (取决于 batch 和序列长度)
    ─────────────────────────
    总计:     ~100-130 GB → 需要多卡!

  推理显存:
    参数:     7B × 2 bytes = 14 GB
    KV Cache: ~2 GB (seq=4096, batch=1)
    其他:     ~1 GB
    ─────────────────────────
    总计:     ~17 GB → 单张 A100 40G 足够!

  → 推理显存只有训练的 1/6 ~ 1/8
  → 这就是为什么量化推理能在消费级显卡上跑
```

## 推理优化空间

```
  ┌─────────────────────────────────────────────────────────────────┐
  │                    推理优化四大方向                               │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                 │
  │  1. 模型压缩 — 让模型更小                                      │
  │     ├── 量化: FP16→INT4, 模型大小减 4 倍                       │
  │     ├── 剪枝: 移除冗余权重/Head                                │
  │     └── 蒸馏: 训练小模型模仿大模型                             │
  │                                                                 │
  │  2. 计算优化 — 让每步更快                                      │
  │     ├── KV Cache: 避免重复计算历史 K,V                         │
  │     ├── FlashAttention: 减少 IO，不存 N×N 矩阵                │
  │     └── 算子融合: 减少 kernel 启动和显存读写                   │
  │                                                                 │
  │  3. 服务优化 — 让并发更高                                      │
  │     ├── Continuous Batching: 请求完成立即让位                   │
  │     ├── PagedAttention: KV Cache 分页管理                      │
  │     └── Speculative Decoding: 小模型猜+大模型验                │
  │                                                                 │
  │  4. 系统优化 — 让架构更稳                                      │
  │     ├── 推理引擎: TensorRT-LLM / ONNX Runtime                 │
  │     ├── 服务框架: vLLM / SGLang / Triton                       │
  │     └── 部署架构: 负载均衡 / Prefill-Decode 分离               │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
```

## LLM 推理的特殊挑战

### 自回归生成 — 为什么 LLM 推理不同

```
  ┌──────────────────────────────────────────────────────────────────┐
  │  传统模型 (如 ResNet):                                           │
  │    输入图片 → 一次前向 → 输出类别  (一步到位)                    │
  │                                                                  │
  │  LLM (自回归生成):                                               │
  │    输入 prompt → 一个一个生成 token (串行!)                      │
  │                                                                  │
  │    Step 1: "What is" → 模型计算 → "the"                         │
  │    Step 2: "What is the" → 模型计算 → "capital"                  │
  │    Step 3: "What is the capital" → 模型计算 → "of"               │
  │    ...                                                           │
  │    → 生成 100 个 token = 跑 100 次前向传播!                      │
  │                                                                  │
  │  这就像: 你问翻译官一句话                                        │
  │    传统模型: 翻译官想好了一次性说完                               │
  │    LLM: 翻译官一个词一个词往外蹦...                              │
  │                                                                  │
  │  带来的问题:                                                     │
  │  1. 延迟累积: 100 token × 每个 10ms = 1 秒                      │
  │  2. GPU 利用率低: 每步只处理 1 个 token (batch=1 效果)           │
  │  3. 需要 KV Cache: 否则每步都要重算历史                          │
  └──────────────────────────────────────────────────────────────────┘
```

### Memory-bound 的实质

```python
def analyze_arithmetic_intensity():
    """LLM Decode 阶段的计算强度分析"""
    params = 7e9       # 7B 参数
    bytes_per_param = 2  # FP16

    # 生成 1 个 token:
    # 需要读取全部模型权重 (每一层都要读一遍)
    read_bytes = params * bytes_per_param  # = 14 GB

    # 计算量 ≈ 2 × params (矩阵向量乘)
    flops = 2 * params  # = 14 GFLOP

    intensity = flops / read_bytes  # = 1 FLOP/Byte
    # A100 平衡点: 312 TFLOP / 2 TB/s ≈ 156 FLOP/Byte

    print(f"计算强度: {intensity:.2f} FLOP/Byte")
    print(f"A100 平衡点: ~156 FLOP/Byte")
    print(f"差距: {156/intensity:.0f}x → 严重 Memory Bound!")
    print(f"含义: 算力只用了 1/156 → 99.4% 的时间在等数据搬运")
```
