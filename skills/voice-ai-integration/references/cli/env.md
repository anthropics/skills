# Agora CLI Project Environment Export

Verified against Agora CLI `0.1.3`.

`agora project env` is the CLI's primary development-environment command. It exports
project values directly in a format the next tool can consume.

## Core Rule

Use:

```bash
agora project env
```

when the user wants project env vars.

Use:

```bash
agora project env write [path]
```

only when they explicitly want those values materialized into a dotenv file.

## Export Commands

Default behavior:

```bash
agora project env
```

This prints non-sensitive dotenv lines to `stdout`. It does not write a file.

Other supported forms:

```bash
agora project env --shell
agora project env --json
agora project env --format dotenv
agora project env --format shell
agora project env --format json
agora project env --project <project>
agora project env --with-secrets
```

Use cases:

- default dotenv output: paste into `.env`, redirect to a file, or inspect values
- `--shell`: `eval "$(agora project env --shell)"`
- `--json` or `--format json`: agents, CI, or scripts
- `--project <project>`: export a non-current project without changing local context
- `--with-secrets`: include sensitive values such as the app certificate

Option rules:

- `--format` cannot be combined with `--json`
- `--format` cannot be combined with `--shell`
- `--shell` cannot be combined with `--json`

## Exported Variables

The verified `0.1.3` export contract uses these `AGORA_` names:

- `AGORA_PROJECT_ID`
- `AGORA_PROJECT_NAME`
- `AGORA_REGION`
- `AGORA_APP_ID`
- `AGORA_ENABLED_FEATURES`
- `AGORA_FEATURE_RTC`
- `AGORA_FEATURE_RTM`
- `AGORA_FEATURE_CONVOAI`
- `AGORA_APP_CERTIFICATE` only when `--with-secrets` is provided

Do not invent alternate names for the CLI export contract.

## Secrets Rule

Secrets are opt-in.

- By default, `agora project env` exports only non-sensitive development config.
- `AGORA_APP_CERTIFICATE` is emitted only with `--with-secrets`.
- If the target project does not have an app certificate, `--with-secrets` fails instead of silently emitting an empty value.

## File Writing

`agora project env write [path]` writes dotenv output through an Agora-managed block:

```dotenv
# BEGIN AGORA CLI
# Generated by `agora project env`
...
# END AGORA CLI
```

The default behavior is block-oriented, not whole-file replacement.

Supported flags:

```bash
agora project env write
agora project env write apps/web/.env.local
agora project env write --append
agora project env write --overwrite
agora project env write --project <project> --with-secrets
```

Write rules:

- explicit `path`: write exactly that target path
- no `path`: choose the best default `.env*` target based on the current project directory
- if the target already contains an Agora-managed block, update that block in place
- if no managed block exists and the target was auto-selected, append a managed block
- if no managed block exists and the target path was explicit, require `--append` or `--overwrite`
- `--append` and `--overwrite` cannot be used together

## Default Target Selection

When `path` is omitted, `0.1.3` chooses the target in this order:

1. an existing writable `.env*` file that already contains an Agora-managed block, with priority `.env.local` > `.env` > other runtime `.env.*`
2. existing `.env.local`
3. existing `.env`
4. another runtime `.env.*`
5. otherwise create `.env.local`

Template files are excluded from default targeting:

- `.env.example`
- `.env.sample`
- `.env.template`

Do not tell agents to use template files as write targets for real values or secrets.

## Agent Guidance

- For raw env consumption, prefer `agora project env --json` over `agora project show --json`.
- For shell sessions, prefer `agora project env --shell`.
- For repo setup, use `agora project env write` when the user wants the CLI to own an env block in an existing `.env*` file layout.
- `project env` prints the selected format directly; `project env write` is a normal action command and can be consumed like other machine-readable CLI commands.
